# 공격 탐지 및 웹 로그 수집 목적에서 OpenCart 대신 OWASP Juice Shop을 채택해야 하는 이유

## 1. 문서 개요
이 문서는 기존의 **Ubuntu 22.04 + Apache2 + PHP + MySQL + OpenCart** 실습 구성이 왜 일반적인 쇼핑몰 구축 학습에는 적합했는지, 그리고 왜 **공격 탐지와 웹 로그 수집**이 주목적이 된 현재에는 **OWASP Juice Shop**이 더 적합한 대상인지 설명하기 위해 작성하였다.

기존 OpenCart 채택 근거는 다음과 같은 목적에 맞추어져 있었다.

- 쇼핑몰 설치 및 운영 구조 학습
- Apache 기반 웹서버 구축 실습
- 관리자 페이지 및 상품 관리 구조 확인
- 기본 웹 로그 확인
- 취약점 점검의 기초 실습

즉, OpenCart는 **정상적인 전자상거래 웹서비스를 구축하고 운영 구조를 이해하는 실습 대상**으로는 적절하다. 그러나 이번 프로젝트의 핵심 목표가 **공격 요청을 충분히 발생시켜 로그를 수집하고, 그 로그를 바탕으로 공격을 탐지·분류하는 것**이라면, 선택 기준은 달라져야 한다.

---

## 2. 기존 OpenCart 채택 기준과 현재 목적의 차이
기존 문서에서 OpenCart를 선택한 이유는 다음과 같았다.

- 무료 오픈소스 쇼핑몰 플랫폼이라 부담이 적음
- 설치 구조가 비교적 단순함
- Apache 환경과 호환성이 좋음
- 관리자 페이지와 상품/카테고리 관리 기능을 확인하기 좋음
- 로그 확인과 보고서 작성에 유리함

이 기준은 **쇼핑몰 구축 실습**에는 타당하다. 실제로 OpenCart 공식 문서도 Apache 권장, PHP 8.0+, MySQL 계열 데이터베이스 등 전형적인 웹 애플리케이션 운영 환경을 기준으로 설명하고 있다.

하지만 지금 프로젝트의 중심은 단순 설치가 아니라 다음과 같다.

- 다양한 공격 요청을 반복적으로 생성할 수 있어야 함
- 요청별로 의미 있는 웹 로그를 충분히 남길 수 있어야 함
- 공격 유형을 비교적 명확하게 분류할 수 있어야 함
- 합법적이고 통제된 환경에서 실험할 수 있어야 함
- 공격 재현과 데이터셋 축적이 쉬워야 함

따라서 현재는 “실서비스형 쇼핑몰”보다 “보안 실습 및 공격 재현용 웹 애플리케이션”이 더 적절한 선택 기준이 된다.

## 3. OWASP Juice Shop이 더 적합한 핵심 이유

### 3.1 Juice Shop은 처음부터 보안 실습용으로 설계된 애플리케이션이다
OWASP Juice Shop은 OWASP가 제공하는 **의도적으로 취약한 웹 애플리케이션**이다. 공식 소개에서도 보안 교육, 보안 인식 제고 데모, CTF, 보안 도구 시험용 대상으로 설명된다.

즉, Juice Shop은 운영용 전자상거래 서비스가 아니라 **공격 재현과 보안 학습을 위한 표적 시스템**이다. 이 점에서 정상 서비스용 쇼핑몰인 OpenCart와 출발점 자체가 다르다.

이 차이는 매우 중요하다. 공격 탐지 실험에서는 단순히 웹사이트가 동작하는 것보다, **어떤 공격을 어느 정도까지 안전하게 시도할 수 있는가**가 더 중요하기 때문이다.

### 3.2 다양한 공격 시나리오를 한 환경에서 확보하기 쉽다
Juice Shop은 공식적으로 **OWASP Top 10 전반과 다양한 실제 웹 취약점**을 포함한다고 설명된다. 따라서 SQL Injection, XSS, 인증 우회, 민감정보 노출, 비정상 API 호출 등 여러 유형의 공격성 요청을 하나의 애플리케이션에서 비교적 체계적으로 재현할 수 있다.

반면 OpenCart는 본래 **운영 가능한 쇼핑몰 소프트웨어**이므로, 의도적으로 취약점 학습을 위해 설계된 구조가 아니다. OpenCart 환경에서 공격 로그를 얻으려면 특정 버전 취약점, 잘못된 설정, 추가 플러그인 문제, 비정상 입력 처리 등을 별도로 찾아야 할 가능성이 높다. 즉, **공격면이 프로젝트 목적에 맞게 정리되어 있지 않다.**

공격 분류 모델을 만들려면 여러 유형의 악성 요청이 필요하다. 이런 측면에서 Juice Shop은 공격 표면이 더 풍부하고, 실험 재현도 더 쉽다.

### 3.3 공격 로그 수집에 필요한 “반복성”과 “재현성”이 높다
탐지 모델용 로그 데이터는 한 번 수집하고 끝나는 자료가 아니라, 다음과 같은 특성이 필요하다.

- 동일한 공격을 여러 번 재현할 수 있어야 함
- 조건을 바꿔가며 반복 실험할 수 있어야 함
- 공격 전후 로그를 비교할 수 있어야 함
- 정상 요청과 악성 요청을 함께 축적할 수 있어야 함

Juice Shop은 학습·실험용으로 널리 사용되기 때문에 이런 반복 실험 구조에 잘 맞는다. 반면 OpenCart는 기본적으로 **정상적인 쇼핑몰 운영 흐름**을 따라가는 데 초점이 있으므로, 공격 재현 자체를 위해 매번 별도 시나리오를 설계해야 할 수 있다.

즉, OpenCart는 “동작하는 웹서비스”로서의 재현성은 좋지만, **공격 데이터 수집용 재현성**은 Juice Shop보다 불리하다.

### 3.4 합법성과 안전성 측면에서 설명이 쉽다
Juice Shop은 OWASP가 제공하는 취약 실습용 애플리케이션이므로, 보고서에서 **합법적이고 통제된 보안 실험 환경**이라는 점을 설명하기 쉽다. 이는 과제나 프로젝트 보고서에서 매우 중요하다.

반대로 OpenCart에서 공격 재현을 시도할 경우, 그 목적이 “실제 쇼핑몰 소프트웨어를 공격해보는 실험”처럼 보일 수 있다. 물론 로컬 VM과 폐쇄망에서 수행하면 문제는 줄어들지만, 보고서 설득력 측면에서는 Juice Shop 쪽이 훨씬 명확하다.

즉, Juice Shop은 **실습 목적과 윤리적 정당성**을 함께 설명하기 좋다.

### 3.5 공격 탐지 분류 기준을 세우기 쉽다
공격 탐지 프로젝트에서는 단순히 로그를 쌓는 것만으로는 부족하고, 각 요청에 대해 어느 정도의 **라벨링 기준**이 필요하다. 예를 들어 다음과 같은 구분이 가능해야 한다.

- 정상 탐색 요청
- 로그인/인증 관련 비정상 요청
- 스크립트 삽입 시도
- 비정상 파라미터 조작
- 파일 경로 또는 API 악용 시도
- 자동화 도구성 반복 요청

Juice Shop은 다양한 취약 기능과 챌린지 구조를 제공하므로, 어떤 요청이 어떤 공격 의도와 연결되는지 설명하기 쉽다. 반면 OpenCart는 정상 쇼핑몰 기능이 중심이기 때문에, 특정 요청이 “공격 시도”인지 “단순 오동작”인지 구분이 더 어려울 수 있다.

즉, **라벨링 가능한 로그를 만들기에는 Juice Shop이 더 유리하다.**

### 3.6 보안 도구와 함께 쓰기 좋다
Juice Shop은 보안 훈련과 도구 시험용 애플리케이션으로 널리 소개되므로, Apache access/error log뿐 아니라 WAF, reverse proxy, forensic log, audit log 같은 보조 로그 체계와 함께 실험하기 좋다.

예를 들어 다음과 같은 구성이 가능하다.

- 앞단: Apache
- 백엔드: Juice Shop
- 추가: ModSecurity CRS
- 수집 로그: access log, error log, forensic log, audit log

이 구조를 사용하면 **웹 요청 로그 + 보안 이벤트 로그**를 함께 확보할 수 있어, 공격 탐지 성능 비교나 라벨링에도 도움이 된다.

## 4. OpenCart보다 Juice Shop이 유리한 점 요약

| 비교 항목 | OpenCart | OWASP Juice Shop |
|---|---|---|
| 기본 목적 | 실제 쇼핑몰 구축 및 운영 | 보안 학습, 공격 재현, 도구 시험 |
| 공격 시나리오 확보 | 별도 취약점 탐색 필요 가능성 큼 | 다양한 공격 시나리오가 기본 제공됨 |
| 로그 수집 적합성 | 정상 서비스 로그 확인에는 적합 | 공격성 로그 대량 수집에 더 적합 |
| 반복 실험 | 공격 재현 구조가 상대적으로 약함 | 반복·재현 실험에 유리 |
| 라벨링 용이성 | 공격 의도 분류가 상대적으로 모호함 | 공격 의도와 유형 설명이 쉬움 |
| 윤리적 설명 | 실서비스형 앱 대상 실험으로 보일 수 있음 | 교육용 취약 앱이라 목적 설명이 명확함 |
| 보고서 설득력 | 쇼핑몰 구축 보고서에는 강점 | 공격 탐지/보안 실험 보고서에 강점 |

## 5. Apache 로그 수집 관점에서의 실무적 장점
Juice Shop 자체는 일반적으로 Node.js 기반으로 동작하지만, 앞단에 Apache를 두고 **리버스 프록시** 형태로 배치하면 기존에 계획한 Apache 로그 수집 구조를 유지할 수 있다.

즉, 다음과 같은 방식이 가능하다.

1. Ubuntu 22.04 Server를 유지한다.
2. Apache2를 앞단 웹서버로 유지한다.
3. Apache에서 `ProxyPass`, `ProxyPassReverse`로 Juice Shop 백엔드로 전달한다.
4. Apache access/error log, 필요 시 forensic/audit 로그를 함께 수집한다.

이 방식의 장점은 다음과 같다.

- 기존 Apache 로그 분석 계획을 그대로 활용할 수 있음
- 공격성 요청이 모두 Apache를 통과하므로 수집 일관성이 높음
- OpenCart용으로 생각하던 로그 확장 전략을 그대로 이어갈 수 있음
- Apache 기반 보고서 서술과 실제 보안 실험을 동시에 만족할 수 있음

따라서 Juice Shop을 선택한다고 해서 Apache 실습이 무의미해지는 것이 아니라, 오히려 **Apache 로그 수집 프로젝트와 잘 결합될 수 있다.**

## 6. 한계와 보완점
Juice Shop이 항상 모든 면에서 우월한 것은 아니다. 다음 한계는 분명히 존재한다.

### 6.1 실제 상용 쇼핑몰과 완전히 동일한 운영 구조는 아니다
OpenCart는 실제 전자상거래 플랫폼이기 때문에 쇼핑몰 운영 구조 학습에는 더 직접적이다. 반면 Juice Shop은 보안 실습 중심이라, “정상적인 쇼핑몰 구축 학습” 자체가 목적이면 OpenCart가 더 자연스럽다.

### 6.2 정상 비즈니스 트래픽은 별도로 설계해야 할 수 있다
공격 탐지 모델에는 정상 트래픽과 공격 트래픽이 모두 필요하다. Juice Shop에서는 정상 브라우징, 로그인, 장바구니, 상품 조회 같은 일반 사용자 흐름도 수집할 수 있지만, 학습 목적에 맞는 정상 시나리오는 별도로 계획해 두는 편이 좋다.

그러나 이 한계는 치명적이지 않다. 오히려 공격 탐지 프로젝트에서는 정상 시나리오를 직접 설계하는 편이 데이터 품질 관리에 유리할 수 있다.

---

## 7. 최종 결론
현재 프로젝트의 핵심 목표가 **웹 공격 탐지와 웹 로그 수집**이라면, OpenCart보다 OWASP Juice Shop을 채택하는 것이 더 타당하다.

그 이유는 다음과 같다.

1. Juice Shop은 애초에 보안 실습과 공격 재현을 위해 설계된 애플리케이션이다.
2. 다양한 공격 시나리오를 한 환경에서 확보하기 쉽다.
3. 공격 로그를 반복적이고 재현 가능하게 수집하기 좋다.
4. 공격 유형 분류와 라벨링에 유리하다.
5. 합법적이고 통제된 실험 환경이라는 점을 설명하기 쉽다.
6. Apache 앞단 로그 수집 구조와도 결합 가능하다.

따라서 “쇼핑몰 설치 실습”이 아니라 “공격 로그 기반 탐지 모델 구축”이 목적이라면, 실험 대상은 OpenCart보다 **OWASP Juice Shop**이 더 적절하다.

## 8. 보고서용 문단 예시

> 초기에는 Ubuntu 22.04, Apache2, PHP, MySQL, OpenCart 조합을 이용한 쇼핑몰 구축 실습을 고려하였다. 이 구성은 웹서버 설치, 관리자 페이지 확인, 로그 확인 등 일반적인 웹서비스 구축 학습에는 적합하다. 그러나 본 프로젝트의 핵심 목표는 단순 쇼핑몰 구축이 아니라, 다양한 공격 요청을 발생시켜 웹 로그를 수집하고 이를 바탕으로 공격 탐지 및 분류를 수행하는 데 있다. 이러한 목적을 고려할 때, OWASP Juice Shop은 의도적으로 취약하게 설계된 보안 실습용 웹 애플리케이션으로서 다양한 공격 시나리오를 안전하고 반복 가능하게 재현할 수 있으므로, OpenCart보다 본 프로젝트 목적에 더 적합하다.

## 9. 보고서용 짧은 문장 예시

- OpenCart는 쇼핑몰 구축 실습에는 적합하지만, 공격 로그 수집용 표적으로는 한계가 있다.
- OWASP Juice Shop은 보안 학습과 공격 재현을 위해 설계된 의도적 취약 웹 애플리케이션이다.
- 다양한 공격 시나리오를 반복적으로 재현해야 하는 프로젝트 목적상 Juice Shop이 더 적합하다.
- Juice Shop은 공격 탐지용 웹 로그 수집과 라벨링 측면에서 OpenCart보다 유리하다.
- Apache를 앞단 리버스 프록시로 두면 Juice Shop 환경에서도 Apache 로그 수집 전략을 유지할 수 있다.
