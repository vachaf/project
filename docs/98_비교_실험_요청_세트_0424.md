# 98_비교_실험_요청_세트_표준

- 작성 기준일: 2026-04-24
- 목적: Juice Shop 기준 비교 실험용 요청 세트를 표준화한다.
- 적용 범위: 동일 raw log를 기준으로 OpenAI / Anthropic, stage1 / stage2 결과를 비교할 때 사용한다.
- 원칙: 모든 요청은 웹 GUI가 아니라 **커맨드 라인(curl)** 기준으로 수행한다.

---

## 1. 실험 목적

이 문서는 Juice Shop 로컬/내부 실험 환경에서 다음을 비교하기 위한 표준 요청 세트를 정리한다.

- 정상 요청과 공격성 요청이 전처리(`prepare_llm_input.py`)에서 어떻게 분리되는지
- stage1에서 provider별 verdict / severity / confidence 차이가 어떻게 나타나는지
- stage2에서 provider별 사건 우선순위 / 보고서 표현 차이가 어떻게 나타나는지
- 동일 요청 세트에서 low signal / false positive / suspicious incident가 어떻게 달라지는지

비교 실험에서는 다음을 분리해서 본다.

1. **Full pipeline 비교**
   - 같은 raw log
   - provider별 prepare -> stage1 -> stage2 전체 비교
2. **Stage1 비교**
   - 같은 `*_llm_input.json`
   - provider별 stage1 결과 비교
3. **Stage2 비교**
   - 같은 `*_stage1_results.json` 고정
   - provider별 stage2 결과 비교

---

## 2. 기본 전제

- 실험 대상 앱: OWASP Juice Shop
- 기준 URL: `http://127.0.0.1:3000`
- 가능하면 한 환경에서 raw log를 만들고, 그 raw log를 기준으로 provider별 비교를 수행한다.
- `known_asset_ips`는 **raw log를 생성한 환경 기준**으로 맞춘다.
- 요청 라벨을 구분하기 위해 가능한 한 `User-Agent`를 명시한다.
- 요청 간격은 1~3초 수준으로 유지하고, 인증 실패 반복 세트는 1분 내 짧게 몰아서 수행한다.
- 비교 실험 문서상 `stage2 보고서`는 최종 증거가 아니라 사건 요약 보고서다. suspicious incident는 request_id 기준 raw evidence를 함께 본다.

## 3. 실행 전 공통 준비

### 3.1 기본 변수

```bash
BASE_URL="http://127.0.0.1:3000"
DATE_TAG="2026-04-24"
```

### 3.2 curl 공통 옵션

권장:

- `-i`: 응답 헤더 포함
- `-sS`: 필요 시 조용하지만 오류는 표시
- `-A`: 실험 세트 식별용 User-Agent 명시
- `--path-as-is`: path traversal 계열에서 사용

예:

```bash
curl -i -A "lab-normal-home-1" "$BASE_URL/"
```

### 3.3 로그 회수 기준

실험 후에는 최소한 아래를 회수한다.

- export JSON (`security_YYYY-MM-DD_kst.json`)
- prepare 산출물 (`*_llm_input.json`, `*_analysis_candidates.json`, `*_noise_summary.json`)
- stage1 결과 (`*_stage1_results.json`, `*_stage1_errors.json`)
- stage2 결과 (`*_stage2_report.json`, `*_stage2_report.md`)
- 필요 시 stage2 실패 파일 (`*_stage2_report_error.json`, `*_stage2_report_raw_error.json`)

## 6. 인증 / 로그인 세트

인증 실패, 반복 실패, 우회 성공, 정상 성공(가능 시)을 분리해서 본다.

### A-01 단일 로그인 실패

```bash
curl -i \
  -A "lab-auth-fail-single-1" \
  -H "Content-Type: application/json" \
  -d '{"email":"admin@juice-sh.op","password":"1234"}' \
  "$BASE_URL/rest/user/login"
```

- 기대 ground truth: 정상 사용자의 로그인 실패 또는 약한 인증 오용 신호
- 기대 포인트: `suspicious_auth_abuse` 또는 `likely_false_positive` 여부 비교

### A-02 로그인 실패 반복 3회

```bash
curl -i -A "lab-auth-fail-repeat-1" -H "Content-Type: application/json" -d '{"email":"admin@juice-sh.op","password":"1234"}' "$BASE_URL/rest/user/login"
sleep 2
curl -i -A "lab-auth-fail-repeat-2" -H "Content-Type: application/json" -d '{"email":"admin@juice-sh.op","password":"1234"}' "$BASE_URL/rest/user/login"
sleep 2
curl -i -A "lab-auth-fail-repeat-3" -H "Content-Type: application/json" -d '{"email":"admin@juice-sh.op","password":"1234"}' "$BASE_URL/rest/user/login"
```

- 기대 ground truth: 반복 로그인 실패
- 기대 포인트: 반복성 때문에 provider가 더 공격적으로 해석하는지 비교

### A-03 Authentication Bypass 성공 시도

```bash
curl -i \
  -A "lab-auth-bypass-1" \
  -H "Content-Type: application/json" \
  -d '{"email":"admin@juice-sh.op'\'' -- ","password":"x"}' \
  "$BASE_URL/rest/user/login"
```

- 기대 ground truth: 인증 우회 시도(성공 여부 포함)
- 기대 포인트:
  - 성공 시 stage1/stage2가 이를 단순 실패보다 더 강하게 보는지
  - 성공 흔적이 보고서에서 어떻게 서술되는지

### A-04 정상 로그인 성공 (가능 시)

> 정상 계정이 준비된 경우에만 수행한다.

```bash
curl -i \
  -A "lab-auth-success-1" \
  -H "Content-Type: application/json" \
  -d '{"email":"정상계정","password":"정상비밀번호"}' \
  "$BASE_URL/rest/user/login"
```

- 기대 ground truth: 정상 인증 성공
- 기대 포인트: 정상 인증이 오탐되지 않는지 확인

## 7. SQLi 세트

`prepare_llm_input.py`의 SQLi 패턴(`union select`, `or_true`, `sleep`, `information_schema` 등)에 직접 반응하는 요청을 넣는다.

### S-01 Boolean-based SQLi

```bash
curl -i -A "lab-sqli-boolean-1" "$BASE_URL/rest/products/search?q=' OR '1'='1"
```

- 기대 ground truth: SQLi 시도
- 기대 포인트: `or_true`, `quote_termination` 계열 힌트 반영 여부

### S-02 UNION-based SQLi

```bash

```

- 기대 ground truth: SQLi 시도
- 기대 포인트: `union_select`, `sql_comment`, `quote_termination` 계열 반영 여부

### S-03 Time-based SQLi

```bash
curl -i -A "lab-sqli-sleep-1" "$BASE_URL/rest/products/search?q=' OR SLEEP(3)--"
```

- 기대 ground truth: SQLi 시도
- 기대 포인트: `sleep_func`와 응답 지연 해석 여부

### S-04 information_schema 탐색형 SQLi

```bash
curl -i -A "lab-sqli-info-schema-1" "$BASE_URL/rest/products/search?q=' UNION SELECT table_name FROM information_schema.tables--"
```

- 기대 ground truth: SQLi 시도
- 기대 포인트: `information_schema`, `select_from`, `union_select` 반영 여부

## 10. 권장 실행 순서

실험은 아래 순서로 수행한다.

1. 정상 기준선 세트
2. 인증 / 로그인 세트
3. SQLi 세트
4. XSS 세트
5. Traversal / HPP / Dir Probe 세트

이유:

- 정상 트래픽을 먼저 쌓아 기준선을 만든다.
- 인증 계열은 반복성 비교가 중요하므로 별도 묶음으로 본다.
- SQLi / XSS / Traversal은 서로 다른 규칙 힌트를 자극하므로 분리해서 본다.
- HPP / Dir Probe는 low signal 또는 경계형 해석 비교에 유용하다.

## 11. 결과 정리 기준

실험 후에는 최소한 아래를 표로 정리한다.

- 실험 ID
- 요청 목적
- 기대 ground truth
- 실제 HTTP 결과 코드
- prepare 분류(후보 / filtered out / noise category)
- stage1 verdict / severity / confidence
- stage2 핵심 서술
- OpenAI / Anthropic 차이
- 최종 평가(정탐 / 오탐 / 경계형)

권장 비교 포인트:

1. 정상 요청을 얼마나 오탐 없이 처리하는가
2. 로그인 실패 반복을 얼마나 공격적으로 해석하는가
3. 명확한 SQLi / XSS / traversal을 얼마나 일관되게 high-signal로 잡는가
4. HPP 문맥을 별도 공격 맥락으로 잘 설명하는가
5. low signal dir probe / fuzzing을 과도하게 incident로 승격하지 않는가

## 12. 주의사항

- 모든 요청은 **테스트 승인된 Juice Shop 환경에서만** 수행한다.
- 비교 실험에서는 GUI 클릭으로 발생하는 부가 트래픽보다 **CLI 요청 세트**를 기준으로 삼는다.
- stage2 보고서는 최종 증거가 아니라 요약 보고서이므로 suspicious incident는 request_id 기준 raw evidence를 함께 본다.
- provider 비교 시에는 가능하면 같은 raw log, 같은 known_asset_ips, 같은 top-N 조건을 유지한다.
- stage2 차이를 보려면 같은 `*_stage1_results.json`을 고정한 상태의 비교도 별도로 수행한다.

## 13. 회수하는 방법
실험 후에는 최소한 아래를 회수한다. IP 는 LLM의 IP로 변경하기 바란다

- export JSON (`security_YYYY-MM-DD_kst.json`)
- prepare 산출물 (`*_llm_input.json`, `*_analysis_candidates.json`, `*_noise_summary.json`)

- stage2 결과 (`*_stage2_report.json`, `*_stage2_report.md`)
- 필요 시 stage2 실패 파일 (`*_stage2_report_error.json`, `*_stage2_report_raw_error.json`)

# export JSON
scp -r user@192.168.35.145:/opt/web_log_analysis/data/raw/security_2026-04-24_kst.json "$env:USERPROFILE\Downloads\"

# prepare 산출물 llm_input.json
scp -r user@192.168.35.145:/opt/web_log_analysis/data/processed/security_2026-04-24_kst_llm_input.json "$env:USERPROFILE\Downloads\"
# prepare 산출물 analysis_candidates.json
scp -r user@192.168.35.145:/opt/web_log_analysis/data/processed/security_2026-04-24_kst_analysis_candidates.json "$env:USERPROFILE\Downloads\"
# prepare 산출물 noise_summary.json
scp -r user@192.168.35.145:/opt/web_log_analysis/data/processed/security_2026-04-24_kst_noise_summary.json "$env:USERPROFILE\Downloads\"

# stage1 결과 stage1_results.json
