# 00_전체_흐름_요약_가이드

- 문서 상태: 수정본
- 버전: v0.2
- 작성일: 2026-04-02
- 목적: 현재 프로젝트의 **웹서버 → DB 서버 → LLM 서버** 흐름을 한눈에 보이도록 정리하고, 문서 간 역할 충돌을 줄이기 위한 운영 기준을 요약한다.
- 권장 독자: 현재 구조를 처음 다시 보는 사람, 발표/점검 직전에 전체 흐름을 빠르게 확인해야 하는 사람

---

## 1. 가장 짧은 요약

이 프로젝트는 아래 세 문장으로 기억하면 된다.

- **웹서버는 로그를 만든다.**
- **DB 서버는 로그를 저장한다.**
- **LLM 서버는 저장된 로그를 export 하고 분석한다.**

---

## 2. 한눈에 보는 전체 구조

```text
[사용자 / 실험 스크립트 / 공격 도구]
                ↓ HTTP
        [웹서버: juice / Apache Reverse Proxy]
                ↓
         [Juice Shop Docker :127.0.0.1:3000]
                ↓
   Apache가 3종 로그 생성
   - app_access.log
   - app_security.log
   - app_error.log
                ↓
   [웹서버의 apache_log_shipper.py]
                ↓ INSERT
        [DB 서버: maria / MariaDB web_logs]
                ↓ SELECT
        [LLM 서버: llm / 분석 전용 VM]
                ↓
   1) export_db_logs_cli.py
   2) prepare_llm_input.py
   3) llm_stage1_classifier.py
   4) llm_stage2_reporter.py
   5) run_analysis_pipeline.py
                ↓
        [최종 산출물: JSON / Markdown 보고서]
```

---

## 3. 서버별 역할 고정 기준

### 3.1 웹서버 (juice)

역할:

- 외부 요청 수신
- Apache Reverse Proxy 처리
- Juice Shop 연결
- `app_access.log`, `app_security.log`, `app_error.log` 생성
- `apache_log_shipper.py` 로 DB 적재

하지 않는 것:

- DB export 실행
- LLM 호출
- 보고서 생성

### 3.2 DB 서버 (maria)

역할:

- `web_logs` 데이터베이스 저장
- 3개 로그 테이블 저장
- `log_writer` / `log_reader` 권한 분리
- 조회 대상 데이터 제공

하지 않는 것:

- 웹 요청 처리
- LLM 분석 수행
- 보고서 생성

운영 해석 기준:

- DB 서버에 `export_db_logs_cli.py` 사본이 존재할 수 있다.
- 다만 **표준 운영 실행 위치는 LLM 서버** 로 본다.
- DB 서버의 export 도구는 점검용 / 예비용 사본으로 해석한다.

### 3.3 LLM 서버 (llm)

역할:

- DB export 수행
- JSON 원본 저장
- 노이즈 필터링
- 후보 추출
- LLM 1차 분류
- 2차 Markdown 보고서 생성
- 통합 파이프라인 실행

하지 않는 것:

- Apache 로그 생성
- DB 적재
- Reverse Proxy 처리

---

## 4. 로그 3종의 역할

### 4.1 access log

- 운영 확인용
- 상태코드, URI, IP 같은 기본 흐름 확인
- 기준선 비교용

즉, access 는 **주 분석 입력이 아니라 보조 확인용**이다.

### 4.2 security log

- 탐지·분류용 핵심 입력
- suspicious 여부, 후보 추출 대상
- routine 분석의 기본 입력

즉, 현재 파이프라인의 기본 입력은 `security` 다.

### 4.3 error log

- 5xx, 프록시 오류, 예외 상황 조사
- `request_id`, `error_link_id` 연결 확인

즉, 평소에는 보조 입력이지만, 문제 상황에서는 중요하다.

---

## 5. 실제 데이터 이동 순서

1. 사용자가 웹 요청을 보낸다.
2. Apache 가 요청을 받고 Juice Shop 으로 프록시한다.
3. Apache 가 access / security / error 로그를 남긴다.
4. 웹서버의 shipper 가 로그 파일을 읽어 DB 로 적재한다.
5. LLM 서버가 DB 에서 `security` 중심으로 export 한다.
6. LLM 서버가 `prepare_llm_input.py` 로 정제한다.
7. `llm_stage1_classifier.py` 가 1차 분류를 수행한다.
8. `llm_stage2_reporter.py` 가 최종 Markdown 보고서를 만든다.

짧게 줄이면:

```text
요청 발생 → 로그 생성 → DB 적재 → LLM 서버 export → 정제 → stage1 → stage2
```

---

## 6. 시간 기준

- DB 저장 기준: UTC
- 사용자가 입력하는 조회 범위: KST
- export JSON 출력 시간: KST

즉, **저장은 UTC, 사람이 보는 입력/출력은 KST** 로 이해하면 된다.

---

## 7. 현재 가장 헷갈리기 쉬운 지점

### 7.1 export 를 어디서 실행하는가

현재 문서상으로는 두 흔적이 섞여 있다.

- DB 서버에도 `export_db_logs_cli.py` 가 있다고 적혀 있음
- LLM 서버도 export 를 담당한다고 적혀 있음

운영 기준은 아래처럼 고정한다.

- **표준 실행 위치: LLM 서버**
- **DB 서버의 export 도구: 점검용 / 예비용 사본**

### 7.2 왜 security 가 기본 입력인가

- `access` 는 운영 확인용
- `error` 는 예외 조사 보조
- `security` 가 요청 단위 특징량과 후보 추출 기준을 가장 잘 담음

따라서 분석 기본 입력은 `security` 로 고정한다.

### 7.3 파일명은 어떻게 맞추는가

아래처럼 통일한다.

- export 결과: `security_2026-04-02_kst.json`
- prepare 입력: `security_2026-04-02_kst.json`
- stage1 입력: `security_2026-04-02_kst_llm_input.json`
- stage2 입력: `security_2026-04-02_kst_stage1_results.json`

## 8. 실무용 최소 운영 순서

```text
[1] 웹 요청 발생
[2] Apache 로그 생성 확인
[3] shipper 로 DB 적재 확인
[4] LLM 서버에서 security export 실행
[5] prepare_llm_input 실행
[6] stage1 실행
[7] stage2 실행
[8] 보고서 확인
```

---

## 9. 문서 간 표현을 통일할 때 추천하는 고정 문장

> 웹서버는 Apache 로그를 생성하고 shipper 로 DB 에 적재한다.

> DB 서버는 로그를 저장하는 중앙 저장소이며, 적재와 조회만 책임진다.

> LLM 서버는 DB 에 적재된 로그를 export, 정제, 분류, 보고서 생성하는 분석 전용 계층이다.

> 보안 분석 파이프라인의 기본 입력은 `security` 로그이며, `error` 는 보조 근거, `access` 는 운영 확인용 보조 입력이다.

> 시간 처리 원칙은 “DB 저장은 UTC, 조회와 출력은 KST” 로 통일한다.
