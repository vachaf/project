# 진행상황 정리본

- 문서 목적: 중간 발표와 PPT 작성에 바로 사용할 수 있도록 현재 상태를 정리한다.
- 기준 시점: 현재 저장소 문서, 실행 스크립트, 최근 작업 흐름 기준

## 1. 현재 목표

현재 프로젝트의 목표는 **LLM 기반 침입로그 분석 시스템**을 구축하고,  
웹 요청 로그를 수집한 뒤 DB에 저장하고, export → 전처리 → LLM 1차 분류 → LLM 2차 보고서 생성까지 이어지는 분석 파이프라인을 운영 가능한 형태로 만드는 것이다.

추가 목표는 다음과 같다.

- Juice Shop 환경에서 공격성/정상 요청 로그를 수집하고 분석 흐름을 확인한다.
- OpenCart를 비교 실험 대상으로 추가하여, 동일한 요청 의도에 대해 두 서비스의 로그와 보고서가 어떻게 달라지는지 비교한다.
- 최종적으로는 실험 결과를 보고서와 발표 자료 형태로 정리할 수 있는 수준까지 문서와 파이프라인을 맞춘다.

## 2. 현재까지 완료한 작업

### 2.1 인프라 및 역할 분리

- 웹서버 / DB 서버 / LLM 서버의 역할 분리를 정리했다.
- 웹서버는 Apache 로그 생성과 shipper 실행, DB 서버는 로그 저장, LLM 서버는 export 및 분석 수행 구조로 정리했다.
- 전체 파이프라인 기준 문서를 현재 코드 기준으로 다시 정리했다.

### 2.2 웹 애플리케이션 환경

- Juice Shop 실험 환경을 구축했다.
- OpenCart 비교 실험 환경을 추가했다.
- OpenCart는 추가한 지 얼마 되지 않았으며, 현재는 비교 실험과 보고서 비교 단계가 중요하다.

### 2.3 로그 수집 및 저장 구조

- Apache 로그 3종(access / security / error) 구조를 정리했다.
- `apache_log_shipper.py` 기반으로 DB 적재 구조를 정리했다.
- MariaDB `web_logs` 기준 테이블 구조를 문서화했다.
- `log_writer`, `log_reader` 역할 분리를 기준으로 구축 문서를 보강했다.

### 2.4 분석 파이프라인

현재 사용 중인 분석 흐름은 아래와 같다.

```text
Apache 로그 생성
  ↓
apache_log_shipper.py
  ↓
MariaDB(web_logs)
  ↓
export_db_logs_cli.py
  ↓
prepare_llm_input.py
  ↓
llm_stage1_classifier.py
  ↓
llm_stage2_reporter.py
```

또한 `run_analysis_pipeline.py`를 통해 prepare 이후 단계를 자동 실행할 수 있도록 정리했다.

### 2.5 문서 정리

- `00_전체_흐름_요약_가이드.md`를 요약 문서로 정리했다.
- `02_*` 구축문서를 재현 가능한 절차서 기준으로 다시 보강했다.
- `03`, `04`, `05` 문서를 현재 코드 기준으로 정리했다.
- `99_HTML_fallback_fingerprint_구현_검토와_보류_결정.md`를 보류 결정 문서로 정리했다.
- README와 스크립트 설명 문서를 추가 정리했다.

## 3. 현재 확인이 끝난 부분

현재 기준으로 **구조와 문서 정합성**은 많이 정리된 상태다.

### 3.1 정리 완료

- 현재 파이프라인의 단계 구분
- 각 스크립트의 역할
- 기본 입력이 `security`라는 점
- `error`와 `access`의 보조적 역할
- KST 입력 / UTC 조회 / KST 출력 원칙
- stage1 / stage2 기본 모델 사용 정책
- run_analysis_pipeline 중심 실행 흐름

### 3.2 보수 해석 기준 정리 완료

다음 항목은 현재 실제 분석에 유지하는 축으로 정리했다.

- `resp_content_type`
- `response_body_bytes`
- `raw_request_target`
- `path_normalized_from_raw_request`
- `likely_html_fallback_response`

반면 `resp_html_*` fingerprint 계열은 현재 핵심 운영 기능이 아니라는 점을 문서 전체에서 통일했다.

## 4. 현재 진행 중인 작업

### 4.1 OpenCart 비교 실험

- OpenCart를 추가한 지 얼마 되지 않아, 아직 비교 실험 결과를 체계적으로 정리하는 단계가 중요하다.
- 앞으로는 Juice Shop과 OpenCart에 대해 같은 실험 요청 세트를 적용하고, 두 서비스의 보고서를 비교해야 한다.

### 4.2 진행상황과 발표 자료 연결

- 지금은 문서 정리를 거의 마친 상태이므로, 다음 단계는 진행상황을 발표용으로 정리하는 것이다.
- 중간 발표용 PPT에는 “무엇을 만들고 있는가”, “현재 어디까지 되었는가”, “남은 작업은 무엇인가”, “어려운 점은 무엇인가”가 분명히 보이도록 정리해야 한다.

## 5. 남아 있는 작업

### 5.1 실험 및 비교

- Juice Shop과 OpenCart에 동일 의도의 요청 세트를 각각 적용
- 시간 구간을 분리해서 export
- 보고서 결과 비교
- 어떤 요청이 후보로 남고, 어떤 요청이 noise로 빠지는지 비교

### 5.2 발표 준비

- 현재 진행상황을 발표 문장 중심으로 압축
- PPT 문구를 현재 구조와 맞게 수정
- 질의응답 대비용 포인트 정리

### 5.3 운영 측면 보완

- 실험 결과를 반복 가능하게 관리
- 필요 시 known asset IP 정리
- 필요한 경우 pipeline 실행 결과와 산출물 검토 절차를 더 명확히 정리

## 6. 현재 어려운 점 / 리스크

### 6.1 서비스별 포맷 차이

서비스나 장비가 달라지면 로그 포맷이나 요청 구조가 달라질 수 있다.  
이 때문에 파싱 기준과 후보 추출 기준이 쉽게 흔들릴 수 있다.

### 6.2 비교 실험 시 로그 혼재 가능성

Juice Shop과 OpenCart 로그가 같은 DB에 적재되면,  
export 시 시간 구간을 명확히 분리하지 않으면 실험 결과가 섞일 수 있다.

### 6.3 저신호 요청 해석

탐색성 요청, fuzzing, low signal probe 같은 요청은  
무조건 공격으로 올리기 어렵고, noise와 incident 사이 경계가 애매할 수 있다.

### 6.4 문서와 코드 정합성 유지

코드가 변하면 문서가 뒤처지기 쉽다.  
이번에 많이 정리했지만, 앞으로도 문서는 계속 코드 기준으로 유지해야 한다.

## 7. 발표에서 강조할 포인트

발표에서는 아래 내용을 분명히 보여주는 것이 좋다.

1. 현재 시스템은 단순 로그 저장이 아니라  
   **로그 수집 → DB 저장 → export → 전처리 → LLM 분류 → 보고서 생성**까지 이어지는 구조라는 점

2. 기본 입력을 `security` 로그로 잡은 이유  
   - `access`: 운영 확인용  
   - `error`: 보조 근거  
   - `security`: 분석 핵심 입력

3. 현재 보수 해석 기준  
   - `resp_html_*` fingerprint 계열은 보류  
   - 대신 `resp_content_type`, `response_body_bytes`, `raw_request_target`, `likely_html_fallback_response` 중심으로 해석

4. OpenCart를 추가한 이유  
   - Juice Shop 단독 실험만으로는 실제 서비스형 앱과의 차이를 보기 어려움  
   - 비교 실험으로 후보/노이즈/보고서 차이를 더 분명히 볼 수 있음

## 8. 현재 상태 한 줄 요약

다음 핵심 작업은 **OpenCart 비교 실험 결과를 정리하고, 이를 중간 발표 자료로 연결하는 것**이다.
