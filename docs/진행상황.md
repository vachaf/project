# 진행상황 정리본 (Juice Shop 기준 업데이트)

- 문서 목적: 중간 발표와 PPT 작성에 바로 사용할 수 있도록 현재 상태를 정리한다.
- 기준 시점: 현재 저장소 문서, 실행 스크립트, 최근 Juice Shop 실행 결과 기준
- 범위: 본 문서는 **Juice Shop 관련 진행상황만** 반영하며, OpenCart 비교 실험 내용은 제외한다.

## 1. 현재 목표

현재 프로젝트의 목표는 **LLM 기반 침입로그 분석 시스템**을 구축하고,  
Juice Shop 환경에서 발생한 웹 요청 로그를 수집한 뒤 DB에 저장하고,  
`export → 전처리 → LLM 1차 분류 → LLM 2차 보고서 생성`까지 이어지는 분석 파이프라인을 운영 가능한 형태로 만드는 것이다.

현재 Juice Shop 기준 핵심 목표는 다음과 같다.

- Apache 로그 3종(access / security / error)을 안정적으로 수집한다.
- `apache_log_shipper.py`로 DB 서버의 `web_logs`에 적재한다.
- `security` 로그를 기본 입력으로 export 및 전처리한다.
- `prepare_llm_input.py`, `llm_stage1_classifier.py`, `llm_stage2_reporter.py`로 분석 보고서를 생성한다.
- 중간 발표 시 설명 가능한 수준으로 문서와 실행 흐름을 정리한다.

## 2. 현재까지 완료한 작업

### 2.1 인프라 및 역할 분리

- 웹서버 / DB 서버 / LLM 서버의 역할 분리를 정리했다.
- 웹서버는 Apache 로그 생성과 shipper 실행, DB 서버는 로그 저장, LLM 서버는 export 및 분석 수행 구조로 정리했다.
- 전체 파이프라인 기준 문서를 현재 코드 기준으로 다시 정리했다.

### 2.2 Juice Shop 실험 환경

- Juice Shop 실험 환경을 구축했다.
- Apache Reverse Proxy 앞단에서 Juice Shop을 서비스하도록 구성했다.
- Apache 로그 3종이 생성되도록 맞췄다.
- `security` 로그 중심 분석이 가능하도록 운영 구조를 정리했다.

### 2.3 로그 수집 및 저장 구조

- Apache 로그 3종(access / security / error) 구조를 정리했다.
- `apache_log_shipper.py` 기반 DB 적재 구조를 정리했다.
- MariaDB `web_logs` 기준 테이블 구조를 문서화했다.
- `log_writer`, `log_reader` 역할 분리를 기준으로 구축 문서를 보강했다.

### 2.4 분석 파이프라인

현재 사용 중인 분석 흐름은 아래와 같다.

```text
Apache 로그 생성
  ↓
apache_log_shipper.py
  ↓
MariaDB(web_logs)
  ↓
export_db_logs_cli.py
  ↓
prepare_llm_input.py
  ↓
llm_stage1_classifier.py
  ↓
llm_stage2_reporter.py
```

또한 `run_analysis_pipeline.py`를 통해 prepare 이후 단계를 자동 실행할 수 있도록 정리했다.

### 2.5 문서 정리

- `00_전체_흐름_요약_가이드.md`를 요약 문서로 정리했다.
- `02_*` 구축문서를 재현 가능한 절차서 기준으로 다시 보강했다.
- `03`, `04`, `05` 문서를 현재 코드 기준으로 정리했다.
- `99_HTML_fallback_fingerprint_구현_검토와_보류_결정.md`를 보류 결정 문서로 정리했다.
- 실험 요청 세트와 운영 메모를 별도 문서로 정리하기 시작했다.

## 3. 현재 확인이 끝난 부분

현재 기준으로 **구조와 문서 정합성**은 많이 정리된 상태다.

### 3.1 정리 완료

- 현재 파이프라인의 단계 구분
- 각 스크립트의 역할
- 기본 입력이 `security`라는 점
- `error`와 `access`의 보조적 역할
- KST 입력 / UTC 조회 / KST 출력 원칙
- stage1 / stage2 기본 모델 사용 정책
- run_analysis_pipeline 중심 실행 흐름

### 3.2 보수 해석 기준 정리 완료

다음 항목은 현재 실제 분석에 유지하는 축으로 정리했다.

- `resp_content_type`
- `response_body_bytes`
- `raw_request_target`
- `path_normalized_from_raw_request`
- `likely_html_fallback_response`

반면 `resp_html_*` fingerprint 계열은 현재 핵심 운영 기능이 아니라는 점을 문서 전체에서 통일했다.

### 3.3 Juice Shop 기준 파이프라인 실제 검증 완료

Juice Shop 로그를 대상으로 아래 흐름을 실제로 한 번 끝까지 검증했다.

1. `export_db_logs_cli.py`로 `security` 로그 export
2. `prepare_llm_input.py`로 후보 및 filtered out 분리
3. `llm_stage1_classifier.py`로 후보별 1차 분류
4. `llm_stage2_reporter.py`로 최종 Markdown / JSON 보고서 생성

즉, 현재 상태는 “문서만 정리된 단계”가 아니라, **Juice Shop 기준으로 end-to-end 실행이 실제로 검증된 단계**다.

## 4. 최근 Juice Shop 실행 기준 확인 결과

최근 확인한 Juice Shop 실행 기준 결과는 아래와 같다.

- export 대상: `security`
- export row 수: 61건
- prepare 결과:
  - 후보 9건
  - filtered out 52건
  - dedup 제거 0건
- stage1 결과:
  - 성공 9건
  - 오류 0건
- stage2 결과:
  - 최종 Markdown 보고서 생성 완료
  - 최종 JSON 보고서 생성 완료

유형 분포는 아래와 같이 정리되었다.

- `suspicious_sqli`: 2건
- `suspicious_xss`: 3건
- `suspicious_path_traversal`: 4건

filtered out 세부 분포는 아래와 같다.

- `low_signal_fuzzing`: 48건
- `low_signal_dir_probe`: 4건

즉, 현재 파이프라인은 Juice Shop 기준으로
- 공격성 요청 후보 추출,
- 후보 밖 탐색성 요청 분리,
- 최종 보고서 생성
까지 실제로 동작하는 상태다.

## 5. 현재 진행 중인 작업

### 5.1 Juice Shop 기준 실험 정리

- 최근 실행한 Juice Shop 기준본을 발표용으로 설명 가능한 상태로 정리하고 있다.
- 어떤 요청이 후보로 올라오고, 어떤 요청이 `low_signal_fuzzing` 또는 `low_signal_dir_probe`로 빠지는지 해석 기준을 정리 중이다.
- 같은 성격의 실험을 반복 가능하게 관리하기 위해 요청 세트와 세션 기록 기준을 정리하고 있다.

### 5.2 진행상황과 발표 자료 연결

- 지금은 문서 정리를 거의 마친 상태이므로, 다음 단계는 진행상황을 발표용으로 정리하는 것이다.
- 중간 발표용 PPT에는 “무엇을 만들고 있는가”, “현재 어디까지 되었는가”, “남은 작업은 무엇인가”, “어려운 점은 무엇인가”가 분명히 보이도록 정리해야 한다.

## 7. 현재 어려운 점 / 리스크

### 7.1 서비스 특성상 공격성 흔적이 많이 보일 수 있음

Juice Shop은 보안 실습용 애플리케이션이므로,  
일반 서비스형 웹앱보다 공격성 요청이 잘 드러나는 편이다.

즉, 보고서가 공격성 후보를 잘 잡는 것은 장점이지만,  
그 자체를 곧 일반 웹서비스에서도 동일하게 나타나는 분포라고 보기는 어렵다.

### 7.2 저신호 요청 해석

탐색성 요청, fuzzing, low signal probe 같은 요청은  
무조건 공격으로 올리기 어렵고, noise와 incident 사이 경계가 애매할 수 있다.

현재는 다음 원칙을 유지한다.

- `low_signal_fuzzing`, `low_signal_dir_probe`는 기본적으로 incident로 승격하지 않는다.
- stage2에서는 후보 밖 탐색성 요청으로 고정 표기한다.
- 동일 IP, 동일 시간대, 후속 고신호 사건과 결합될 때만 승격 검토한다.

### 7.3 문서와 코드 정합성 유지

코드가 변하면 문서가 뒤처지기 쉽다.  
이번에 많이 정리했지만, 앞으로도 문서는 계속 코드 기준으로 유지해야 한다.

## 8. 발표에서 강조할 포인트

발표에서는 아래 내용을 분명히 보여주는 것이 좋다.

1. 현재 시스템은 단순 로그 저장이 아니라  
   **로그 수집 → DB 저장 → export → 전처리 → LLM 분류 → 보고서 생성**까지 이어지는 구조라는 점

2. 기본 입력을 `security` 로그로 잡은 이유  
   - `access`: 운영 확인용  
   - `error`: 보조 근거  
   - `security`: 분석 핵심 입력

3. 현재 보수 해석 기준  
   - `resp_html_*` fingerprint 계열은 보류  
   - 대신 `resp_content_type`, `response_body_bytes`, `raw_request_target`, `path_normalized_from_raw_request`, `likely_html_fallback_response` 중심으로 해석

4. Juice Shop 기준으로 파이프라인 실제 검증이 끝났다는 점  
   - export 성공
   - 후보 추출 성공
   - stage1 성공
   - stage2 보고서 생성 성공

5. 후보 밖 요청도 버리지 않고 별도 해석하고 있다는 점  
   - `low_signal_fuzzing`
   - `low_signal_dir_probe`

## 9. 현재 상태 한 줄 요약

현재는 **Juice Shop 기준으로 기본 파이프라인과 문서 체계를 정리하고, end-to-end 실행 검증까지 끝낸 단계**이며,  
다음 핵심 작업은 **실험 결과를 반복 가능하게 정리하고 이를 중간 발표 자료로 연결하는 것**이다.
