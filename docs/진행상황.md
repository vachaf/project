# 진행상황 정리본

- 문서 목적: 현재까지의 구축·코드 수정·비교 실험 진행 상황을 한 문서에서 파악할 수 있도록 정리한다.
- 기준 시점: 2026-04-26 E세트 R3 완료 후
- 범위: Juice Shop 및 OpenCart 기반 Apache 로그 분석 파이프라인, A~E 비교 실험, 최근 코드 개선 방향

## 1. 현재 목표

현재 프로젝트의 목표는 **Apache 웹 로그 기반 LLM 침입 로그 분석 파이프라인**을 구축하고, 여러 웹 애플리케이션 환경에서 일반화 가능한 탐지·해석 기준을 검증하는 것이다.

전체 흐름은 다음과 같다.

```text
Apache 로그 생성
  ↓
apache_log_shipper.py
  ↓
MariaDB(web_logs)
  ↓
export_db_logs_cli.py
  ↓
prepare_llm_input.py
  ↓
llm_stage1_classifier.py
  ↓
llm_stage2_reporter.py
```

현재의 핵심 목표는 다음이다.

- Apache `security` 로그를 중심으로 웹 공격 후보를 선별한다.
- `access`, `error` 로그는 운영 확인 및 보조 상관 근거로 둔다.
- LLM이 원본 로그 전체를 무비판적으로 보는 것이 아니라, rule-based 전처리로 선별된 candidate와 context-only 정보를 기반으로 판단하도록 한다.
- Juice Shop에 특화되지 않고 OpenCart 같은 PHP 애플리케이션에서도 동작하는지 확인한다.
- Apache 로그만으로 성공 여부를 단정하지 않는 보수적 해석 기준을 유지한다.

## 2. 현재 시스템 구조

### 2.1 서버 역할

- 웹서버
  - Apache 로그 생성
  - `/opt/apache_log_shipper.py` 실행
  - `/opt/shipper.env`로 DB 접속 정보 관리
- DB 서버
  - MariaDB `web_logs` 저장
  - `apache_access_logs`, `apache_security_logs`, `apache_error_logs` 관리
- LLM 서버
  - DB export
  - prepare
  - Stage1 분류
  - Stage2 보고서 생성

### 2.2 로그 시간 정책

현재 시간 정책은 다음으로 고정했다.

- DB `log_time`: UTC naive datetime 저장
- 사용자의 export 입력: KST 기준
- DB 조회: KST 입력을 UTC로 변환해 조회
- export JSON 출력: KST ISO-8601 문자열

최근 OpenCart 환경에서 서버 timezone이 KST였을 때 `log_time`이 KST naive로 저장되는 문제가 발견되었다. 이를 해결하기 위해 다음 코드를 수정했다.

- `src/apache_log_shipper.py`
  - Apache log_time의 timezone offset을 읽어 UTC naive로 DB 저장
  - `parse_error_time()`에 `%{uc}t` 형식 대응 보강
- `src/export_db_logs_cli.py`
  - timezone suffix 입력 허용
  - KST 입력 / UTC 조회 / KST 출력 흐름을 방어적으로 보강

### 2.3 운영 파일 기준

웹서버 쪽 운영 구조는 다음으로 정리했다.

```text
/opt/shipper.env
/opt/apache_log_shipper.py
```

`/opt/config/llm.env`는 웹서버 shipper 실행과 무관하므로 문서에서 제거했다.

## 3. 현재까지 완료한 코드 개선

### 3.1 URL decode / double decode / HTML entity decode

`prepare_llm_input.py`에서 다음 변형을 분석에 반영했다.

- raw query string
- raw request target
- URL decode depth 1
- URL decode depth 2
- HTML entity decode view

효과:

- SQLi double encoding 탐지 개선
- XSS HTML entity payload 탐지 개선
- C세트와 E세트에서 `&#x3C;script&#x3E;...` 형태를 XSS로 복원 가능

### 3.2 SQLi false positive 완화

교육용 SQL 검색을 공격으로 과승격하지 않도록 완화했다.

예시:

```text
how to use UNION in SQL
SELECT * FROM users tutorial
```

이런 요청은 강한 공격 구조가 없으면 `possible_false_positive_sql_keyword_search` 또는 context-only 후보로 낮춘다.

### 3.3 supporting_events 도입

후보 밖 저신호 요청을 완전히 버리지 않고, 동일 IP/시간대/endpoint family 기준으로 Stage2 context에 보존하는 구조를 넣었다.

원칙:

- 개별 incident로 과승격하지 않는다.
- 시간상 주변 맥락으로만 제공한다.
- 같은 IP와 시간대에서 고신호 사건이 있을 때 보조 근거로 사용한다.

### 3.4 false_positive_review_candidates 도입

교육용 XSS/SQL 검색처럼 오탐 가능성이 큰 요청은 candidate로 과승격하지 않고 review context로 보존한다.

C세트에서 다음 목적에 사용했다.

- `tutorial for onerror event handler`
- 교육용 XSS 검색과 실제 XSS payload 구분

### 3.5 probing_sequence_summaries 도입

D세트 directory probing 실험 이후, 단발 저신호 directory probe를 incident로 올리지 않되 burst context로 요약하는 구조를 추가했다.

특징:

- `context_only` policy
- 반복 경로, 민감 경로, admin 경로, 동일 크기 fallback-like 응답 등을 요약
- candidate 과승격 방지

### 3.6 PHP file disclosure 탐지 보강

E세트 R2에서 `path=php://filter...config.php`가 candidate에서 빠지는 문제가 발견되어 수정했다.

추가한 일반 탐지 신호:

- `php://filter`
- `convert.base64-encode`
- `resource=`
- `config.php`
- `admin/config.php`
- `index.php`

효과:

- `route=php://filter...index.php` 유지 및 score 상승
- `path=php://filter...config.php` 신규 candidate 승격
- `/config.php`, `/admin/config.php` 직접 접근은 과승격하지 않고 context-only 유지

## 4. 비교 실험 진행 상황

### 4.1 A세트

A세트는 초기 비교 실험 기준으로 사용했다.

역할:

- 기본 공격 로그 분석 흐름 검증
- OpenAI / Anthropic 보고서 비교 기준 마련
- 이후 B/C/D/E 세트 문서 구조의 기준 역할

### 4.2 B세트 SQLi / Boolean / Time-based

B세트에서는 SQLi 심화 실험을 진행했다.

주요 결과:

- Boolean Blind에서 `x')) OR 1=1 --`와 `x')) AND 1=2 --`의 `response_body_bytes` 차이가 명확히 관찰됨
- Time-based randomblob delay는 기대한 수준으로 발생하지 않아 실패로 명시
- Round 2B에서는 double encoding, temporal chain, educational SQL FP 완화를 검증

코드 반영:

- double decode SQLi 탐지
- 교육용 SQL 검색 완화
- supporting_events 보존

### 4.3 C세트 XSS

C세트에서는 XSS와 인코딩 복원 능력을 검증했다.

주요 결과:

- 기본 `<script>alert(1)</script>` 탐지 성공
- URL encoded XSS 탐지 성공
- HTML entity encoded XSS 탐지 성공
- `onerror`, `javascript:` 등 XSS context hint 보강
- 교육용 XSS 검색은 candidate 과승격을 줄이고 review context로 보존

코드 반영:

- HTML entity decode view
- `xss:html_entity_decoded_script`
- `xss:event_handler:onerror`
- `xss:javascript_protocol`
- `xss:browser_data_access`
- `xss:external_exfil_intent`

### 4.4 D세트 Path Traversal / HPP / Directory Probing

D세트에서는 path traversal, HPP, directory probing을 검증했다.

주요 결과:

- R1 Path Traversal: `../`, `%2e%2e%2f`, null byte 계열 시도 탐지 성공
- R2 HPP: 중복 parameter와 HPP+SQLi/HPP+XSS context 검증
- R3 Directory Probing: 단발 directory probe를 candidate로 과승격하지 않고 sequence summary로 보존

코드 반영:

- HPP context preservation 유지
- directory probing burst summary
- repeated fallback-like HTML은 성공 근거가 아니라 보조 지표로만 사용

### 4.5 E세트 OpenCart

E세트는 Juice Shop이 아닌 OpenCart에서 탐지 로직이 일반화되는지 검증하기 위한 세트다.

#### R1 — OpenCart route traversal / admin path

결과:

- `route=../../../../etc/passwd` 형태의 route parameter traversal intent 탐지 성공
- `/admin/`, `/admin/index.php`는 개별 incident로 과승격하지 않고 context-only probing sequence로 보존
- OpenCart에서는 `/admin/index.php`가 실제 관리자 로그인 페이지일 수 있으므로 Juice Shop의 SPA fallback과 다르게 해석해야 함

#### R2 — PHP wrapper / config exposure

결과:

- 수정 전에는 `path=php://filter...config.php`가 candidate에서 누락됨
- 코드 수정 후 `route=php://filter...index.php`, `path=php://filter...config.php` 모두 `suspicious_file_disclosure` hint로 candidate 보존
- `/config.php`, `/admin/config.php`는 200/0B 응답이지만 파일 노출 성공으로 단정하지 않고 context-only 유지

남은 개선:

- Stage1 verdict는 아직 `suspicious_path_traversal`로 수렴하는 경향이 있어 `suspicious_file_disclosure` verdict 정식화를 검토할 수 있음

#### R3 — OpenCart product/search SQLi/XSS

결과:

- `search=x')) OR 1=1 --` SQLi candidate 탐지 성공
- `search=<script>alert(1)</script>` XSS candidate 탐지 성공
- `search=&#x3C;script&#x3E;alert(1)...` HTML entity XSS candidate 탐지 성공
- OpenCart의 `/index.php?route=product/search&search=...` 구조에서도 기존 탐지 로직이 동작함을 확인

남은 개선:

- `x'))` 형태에 대해 `sqli:xclose_pattern` 또는 `sqli:quote_termination` 같은 세부 hint 추가 가능
- known asset IP에 `192.168.56.111` 포함 필요

## 5. 현재 판단 기준

### 5.1 성공 단정 금지

현재 Stage2 보고서의 핵심 원칙은 다음이다.

- `status_code=200`만으로 공격 성공을 단정하지 않는다.
- `resp_content_type=text/html`만으로 XSS 실행 성공을 단정하지 않는다.
- `response_body_bytes` 차이는 보조 지표이지 본문 노출 증거가 아니다.
- Apache 로그만으로 파일 내용, DB 결과, 브라우저 실행을 직접 확인할 수 없다.

### 5.2 context-only 정보의 역할

다음 정보는 개별 incident가 아니라 Stage2 문맥으로만 사용한다.

- `supporting_events`
- `false_positive_review_candidates`
- `probing_sequence_summaries`

이를 통해 후보 밖 저신호 요청을 완전히 버리지 않되, 과승격은 막는다.

### 5.3 일반화 우선

실험환경 특화 조건은 피한다.

금지 방향:

- `lab-*` User-Agent 기반 탐지
- 특정 IP 기반 탐지
- 특정 response_body_bytes 값 hard-code
- Juice Shop/OpenCart 이름 기반 분기

허용 방향:

- 일반 URL encoding / double encoding
- 일반 SQLi/XSS/path traversal/PHP wrapper 신호
- Apache 로그 표면 지표 기반 해석

- 단순 로그 저장이 아니라 end-to-end 분석 파이프라인이라는 점
- LLM이 모든 로그를 직접 보는 게 아니라 전처리 candidate를 기반으로 판단한다는 점
- 오탐 억제와 탐지 보존을 동시에 맞추기 위해 context-only 구조를 둔 점
- Juice Shop뿐 아니라 OpenCart에서도 탐지 로직이 일반화되는지 검증했다는 점
- Apache 로그만으로 성공을 단정하지 않는 보수적 분석 원칙

## 8. 현재 리스크

### 8.1 Stage1 verdict taxonomy 한계

현재 일부 기법이 기존 verdict로 흡수된다.

예:

- `php://filter` → `suspicious_path_traversal`로 표현되는 경향
- file disclosure/source disclosure 전용 verdict 부족

### 8.2 LLM provider별 서술 차이

OpenAI는 대체로 보수적으로 성공 단정을 피한다. Anthropic은 위험도와 임팩트를 더 적극적으로 설명한다. 발표/최종 문서에서는 Anthropic의 강한 표현을 Apache 로그 표면 기준으로 제한해 해석해야 한다.

### 8.3 실험환경과 일반환경의 차이

Juice Shop과 OpenCart는 실험용 또는 통제된 환경이다. 따라서 보고서에서는 특정 실험환경의 결과를 일반 서비스에 그대로 확대하지 않고, Apache 로그 표면에서 관찰 가능한 신호로 제한해 설명해야 한다.
