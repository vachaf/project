# 진행상황 요약

작성일: 2026-04-07

## 1. 현재까지 한 일

웹서버에서 생성한 Apache 로그를 DB에 적재하고, LLM 서버에서 `security` 로그를 export하여 전처리, 1차 분류, 2차 보고서 생성까지 이어지는 기본 파이프라인을 확인했다. 현재 기준 운영 순서는 다음과 같다.

`테스트 요청 발생 → 웹서버 로그 생성 → shipper로 DB 적재 → LLM 서버 export → prepare → stage1 → stage2 보고서 생성` 

## 2. 확인된 점

2026-04-02에는 SQL injection 관련 테스트 요청이 raw 로그와 컨테이너 로그에서 확인되었고, 보고서도 이를 의심 인시던트로 비교적 타당하게 해석했다.
2026-04-04에는 보고서가 0건으로 생성됐지만, 원인은 탐지 실패가 아니라 웹서버에서 DB 적재를 누락한 운영 절차 문제였다.
2026-04-07에는 다시 테스트를 수행해 정상 적재 후 분석을 실행했고, `total_exported_rows=26`, `candidate_rows=3`, `distinct_incident_count=3`, `stage1_error_count=0`으로 파이프라인이 정상 동작했다. 

## 3. 2026-04-07 결과 요약

4/7 분석에서는 SQL injection 1건, XSS 1건, path traversal 1건이 각각 후보로 식별되었다. 세 요청은 모두 같은 출발지 IP `192.168.35.113`에서 짧은 시간대에 발생했고, 해당 IP는 known asset IP 목록에 포함되어 있어 내부 테스트 가능성을 함께 고려해야 한다.

* SQLi: 전형적인 `UNION SELECT` 패턴과 500 응답이 함께 관측됨
* XSS: `<script>alert(1)</script>` 형태의 시험 패턴이 관측됨
* Path Traversal: `../` 및 `/etc/passwd` 관련 요청이 관측됨

다만 path traversal은 이전 수동 확인에서 응답이 HTML 페이지 형태였으므로, 현재 단계에서는 **파일 노출 성공이 아니라 시도 탐지**로 보는 것이 더 적절하다.  

## 4. 현재 판단

현재 코드와 판단 로직은 전반적으로 정상 동작하는 편으로 보인다.
입력이 제대로 들어오면 SQLi, XSS, path traversal 3종이 모두 `security` 기반 파이프라인에서 후보화되고, 보고서도 known asset IP를 반영해 과도하게 외부 공격으로 단정하지 않는다.

다만 아직 보완할 점은 있다.

* 웹서버에서 DB 적재를 빠뜨리지 않도록 운영 절차를 더 확실히 해야 함
* path traversal은 “시도 탐지”와 “실제 파일 노출”을 더 명확히 구분해서 써야 함
* 오늘 수행한 테스트 중 후보로 남지 않은 항목은 필요 시 `filtered_out_rows`나 raw export로 추가 확인할 필요가 있음 

## 5. 상위 모델 비교 결과

routine, milestone, presentation으로 비교해 본 결과, 상위 모델 보고서는 전반적으로 더 안정적이었다. 그중에서는 **presentation 보고서가 가장 균형이 좋았다.**
특히 known asset 해석을 유지하면서도 `/etc/passwd` 200 응답을 우선 확인 대상으로 잡고, 동시에 실제 민감정보 노출은 단정하지 않는 점이 좋았다. 다만 여기서도 path traversal은 표현을 조금 더 보수적으로 다듬을 여지가 있다.
