## 1. 문서 목적

이 문서는 Apache에서 생성된 access/security/error 로그를 MariaDB `web_logs`로 적재하고, 이후 JSON export 및 분석 단계로 연결하는 운영 기준을 정리한 문서다.

본 문서의 목적은 다음과 같다.

- 로그 적재 경로와 구성요소 역할을 명확히 정리한다.
- `apache_log_shipper.py`의 동작 방식과 운영 포인트를 설명한다.
- 장애 발생 시 spool 재처리, state 관리, 재시작 절차를 정리한다.
- 운영 보안 관점에서 계정, 권한, 민감정보, 비밀번호 관리 원칙을 정리한다.
- `likely_html_fallback_response` 와 선택적 `resp_html_*` 적재 해석 원칙을 정리한다.
- JSON export와 후속 LLM 분석 연결 흐름까지 포함한 운영 절차를 정리한다.

이 문서는 구조 표준을 정한 `03_로그_표준과_DB_구조.md` 다음 단계 문서이며, 여기서는 "어떻게 운영할 것인가"에 초점을 둔다.

## 2. 운영 대상 범위

본 문서에서 다루는 운영 대상은 아래와 같다.

1. Apache 로그 파일 3종
   - `/var/log/apache2/app_access.log`
   - `/var/log/apache2/app_security.log`
   - `/var/log/apache2/app_error.log`
2. Python shipper
   - `/opt/apache_log_shipper.py`
3. Shipper 내부 상태 관리 경로
   - `state_dir = /var/lib/apache_log_shipper`
   - `spool_dir = /var/spool/apache_log_shipper`
   - `app_log = /var/log/apache2/apache_log_shipper.log`
4. MariaDB 중앙 저장소
   - `192.168.35.223`
   - DB 이름: `web_logs`
5. JSON export 도구
   - `/opt/log_export/export_db_logs_cli.py`
6. 후속 분석 단계
   - KST 기준 JSON export
   - 노이즈 필터링 / 후보 추출
   - LLM 분석 및 보고서 생성

## 3. 전체 운영 흐름

전체 운영 흐름은 아래와 같다.

```text
클라이언트 / 공격 도구 / 실험 스크립트
                ↓
             Apache
                ↓
   ┌──────┼──────┐
   ↓            ↓            ↓
app_access   app_security   app_error
   ↓            ↓            ↓
   └────── apache_log_shipper.py ──────┘
                     ↓
                 MariaDB(web_logs)
                     ↓
              JSON export (KST)
                     ↓
       노이즈 필터링 / 후보 추출 / LLM 분석
```

이 구조에서 핵심 원칙은 다음과 같다.

- Apache는 로그 생성 지점이다.
- shipper는 파일 로그를 읽어 DB로 적재하는 적재기다.
- MariaDB는 중앙 보관소다.
- 원본 보존과 분석용 정제를 분리한다.
- 분석 계층은 DB 적재 이후에 분리해서 운영한다.
- fallback HTML 판정 강화용 metadata 는 응답 본문 전체 대신 요약 형태로 저장한다.

## 4. 구성요소별 역할

### 4.1 Apache

Apache는 모든 외부 요청이 먼저 통과하는 공통 수집 지점이다.

역할:

- access/security/error 로그 생성
- `request_id`, `error_link_id` 기반 상관분석용 식별자 부여
- 사람이 읽기 쉬운 로그와 분석용 확장 로그를 함께 생성
- 필요 시 `text/html` 응답의 구조 특징(title, root marker, asset basename 등)을 security 로그에 부가 기록

### 4.2 `apache_log_shipper.py`

shipper는 로그 파일을 tail 기반으로 읽고, 파싱 후 DB에 배치 적재한다.

역할:

- 로그 파일 변화 감시
- 로그 한 줄 파싱
- DB 컬럼 구조에 맞는 row 생성
- security 로그에 존재하는 `resp_html_*` 키를 DB 컬럼으로 적재
- 배치 INSERT 수행
- 실패 시 spool 저장
- 복구 시 spool 재전송

### 4.3 MariaDB

MariaDB는 중앙 저장소이며, 역할 분리 관점에서 웹서버와 분리된 별도 서버에 위치한다.

역할:

- 원본에 가까운 로그 데이터 보존
- 조건 검색, 집계, 상관분석 수행
- export 및 후속 탐지/분류의 기준 데이터 제공
- fallback HTML 보수 해석을 위한 선택적 `resp_html_*` 컬럼 저장

### 4.4 `export_db_logs_cli.py`

이 도구는 DB에 적재된 로그를 KST 기준으로 JSON export 하는 도구이다.

역할:

- KST 기준 날짜/시간 범위 지정
- access/security/error/all 단위 조회
- JSON 파일 생성
- 분석용 후처리 입력 파일 생성

운영 기준 메모:

- routine 분석용 export 기본값은 `security` 이다.
- `error` 는 5xx 또는 `request_id`/`error_link_id` 연계 확인 시 보조적으로 조회한다.
- `all` 은 원본 보존과 수동 검증용으로 유지한다.

## 5. Shipper 동작 원리

## 5.1 기본 동작

shipper는 access/security/error 로그를 각각 독립적으로 tail 하면서 새 줄만 읽는다.
각 로그 파일은 별도 state를 가지며, 마지막으로 읽은 파일 위치를 inode와 offset으로 저장한다.

즉, 동작 순서는 아래와 같다.

1. 로그 파일 존재 확인
2. inode와 파일 크기 확인
3. 이전 offset 이후의 새 줄만 읽기
4. 줄 단위 파싱
5. 타입별 버퍼에 적재
6. security 로그에 있는 `resp_html_*` 값을 함께 적재
7. 배치 기준 충족 시 DB flush
8. flush 실패 시 spool 저장
9. 주기적으로 spool 재전송 시도

### 5.2 state 파일의 의미

state 파일은 “어디까지 읽었는가”를 기억하기 위한 파일이다.

목적:

- 프로세스 재시작 후 중복 적재 방지
- tail 방식 연속 처리 유지
- 로그 파일 회전 또는 재생성 시 offset 보정

운영상 의미:

- state가 유지되면 마지막 읽은 지점부터 이어서 적재한다.
- state를 초기화하면 파일 처음부터 다시 읽을 수 있다.

### 5.3 spool 디렉터리의 의미

DB 연결 실패나 flush 오류가 발생하면 버퍼 내용을 spool JSON으로 저장한다.

목적:

- DB 장애 시 데이터 유실 최소화
- 네트워크 단절 시 임시 적재 보존
- 복구 후 자동 replay 지원

운영상 의미:

- spool 파일이 누적되면 DB 적재에 실패하고 있다는 뜻이다.
- spool이 정상 replay 되면 파일이 삭제된다.
- spool이 계속 남아 있으면 DB 접속, 권한, 스키마, 네트워크를 점검해야 한다.

### 5.4 배치 flush 기준

shipper는 다음 조건 중 하나를 만족하면 버퍼를 flush 한다.

- 버퍼 건수가 `batch_size` 이상일 때
- 마지막 flush 이후 `flush_interval_sec` 이상 경과했을 때
- `--once` 모드로 실행했을 때

즉, 지나치게 자주 INSERT 하지 않으면서도 지연을 너무 길게 끌지 않는 절충 구조다.

### 5.5 HTML fallback fingerprint 처리 기준

현재 코드 기준에서 shipper 의 역할은 **fingerprint 계산이 아니라, security 로그에 이미 기록된 `resp_html_*` 키를 DB로 적재하는 것**이다.

운영 원칙:

- shipper 자체가 응답 본문을 읽어 fingerprint 값을 생성하지는 않는다.
- `resp_html_*` 값이 security 로그에 있으면 `apache_security_logs` 로 적재한다.
- 상류 생성 로직이 없으면 `resp_html_*` 는 `NULL` 또는 `"-"` 상태로 남을 수 있다.
- 현재 분석 파이프라인에서 path traversal 보수 해석의 주 기준은 `likely_html_fallback_response` 이다.
- `resp_html_*` 가 비어 있다고 해서 실제 파일 노출 성공으로 단정하지 않는다.

## 6. Shipper 실행 옵션 기준

현재 shipper에서 직접 제공하는 주요 실행 옵션은 다음과 같다.

### 6.1 `--test-db`

DB 연결만 확인하고 종료한다.

사용 목적:

- 접속 정보 점검
- 계정 권한 점검
- 운영 전 사전 확인

### 6.2 `--once`

현재 시점까지 누적된 로그 증분만 한 번 읽고 flush 한 뒤 종료한다.

사용 목적:

- 수동 테스트
- 초기 적재 확인
- 운영 전 파이프라인 점검

### 6.3 `--reset-state`

저장된 offset state를 지우고 다시 처음부터 읽게 한다.

사용 목적:

- 테스트 환경 재적재
- state 손상 복구
- 특정 파일 재처리

주의:

- 운영 중 무분별하게 사용하면 기존 로그를 다시 적재해 중복 데이터가 생길 수 있다.
- 실운영에서는 필요성이 명확할 때만 사용해야 한다.

## 7. 운영 디렉터리와 파일 기준

권장 기준 경로는 아래와 같다.

```text
/opt/apache_log_shipper.py
/var/log/apache2/app_access.log
/var/log/apache2/app_security.log
/var/log/apache2/app_error.log
/var/log/apache2/apache_log_shipper.log
/var/lib/apache_log_shipper/
/var/spool/apache_log_shipper/
/opt/web_log_analysis/src/export_db_logs_cli.py
/opt/web_log_analysis/raw/
/opt/web_log_analysis/processed/
/opt/web_log_analysis/reports/
```

운영 원칙은 다음과 같다.

- 로그 파일은 웹 루트 밖의 비공개 경로에 둔다.
- state와 spool은 별도 디렉터리로 분리한다.
- app 로그와 business 로그를 섞지 않는다.
- export 결과물과 후속 산출물은 분석 VM의 별도 작업 디렉터리에 저장한다.

## 8. 서비스 등록과 자동 실행 권장안

현재 코드 자체는 단독 파이썬 스크립트이므로,
실운영에서는 `systemd` 서비스로 등록해 자동 시작과 재시작 정책을 주는 것이 좋다.

아래는 권장 예시이다.

```ini
[Unit]
Description=Apache Log Shipper
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=root
Group=root
EnvironmentFile=-/etc/default/apache-log-shipper
ExecStart=/usr/bin/python3 /opt/apache_log_shipper.py
Restart=always
RestartSec=3

[Install]
WantedBy=multi-user.target
```

설명:

- `EnvironmentFile`로 DB 접속 정보와 경로를 외부화한다.
- `Restart=always`로 비정상 종료 시 자동 복구를 노린다.
- 로그 파일 접근 권한과 `/var/log/apache2/` 접근성 때문에 초기에는 root 실행이 가장 단순하다.

운영 환경이 정리되면 전용 계정으로 낮추는 것을 검토할 수 있다.
다만 그 경우 Apache 로그 읽기 권한, state/spool 디렉터리 쓰기 권한, 환경파일 읽기 권한을 함께 맞춰야 한다.

## 9. 환경변수 기반 설정 원칙

shipper는 주요 설정을 환경변수로 받도록 작성되어 있다.
운영에서는 이 값을 코드 내부 기본값에 의존하지 말고 외부 환경파일로 명시하는 것이 좋다.

권장 예시:

```bash
LOG_DB_HOST=192.168.35.223
LOG_DB_PORT=3306
LOG_DB_USER=log_writer
LOG_DB_PASSWORD=강한비밀번호
LOG_DB_NAME=web_logs

APACHE_ACCESS_LOG=/var/log/apache2/app_access.log
APACHE_SECURITY_LOG=/var/log/apache2/app_security.log
APACHE_ERROR_LOG=/var/log/apache2/app_error.log

SHIPPER_STATE_DIR=/var/lib/apache_log_shipper
SHIPPER_SPOOL_DIR=/var/spool/apache_log_shipper
SHIPPER_APP_LOG=/var/log/apache2/apache_log_shipper.log

SHIPPER_SCAN_INTERVAL_SEC=1.0
SHIPPER_FLUSH_INTERVAL_SEC=2.0
SHIPPER_BATCH_SIZE=100
SHIPPER_SPOOL_RETRY_INTERVAL_SEC=10.0
SHIPPER_CONNECT_TIMEOUT_SEC=5
SHIPPER_READ_TIMEOUT_SEC=10
SHIPPER_WRITE_TIMEOUT_SEC=10
```

운영 원칙:

- DB 비밀번호를 코드에 하드코딩하지 않는다.
- 서버별 값은 환경파일에서만 관리한다.
- 운영/실험/테스트 환경의 값을 분리한다.
- 환경파일 권한은 최소화한다.

### 9.1 현재 코드 기준 보완 포인트

현재 shipper 코드에는 `LOG_DB_PASSWORD` 기본값이 직접 들어가 있다.
이 값은 문서 기준 운영 권장안과 맞지 않으므로,
실운영에서는 반드시 환경변수 또는 별도 환경파일로만 주입하는 방식으로 관리하는 것이 좋다.

즉, 이 항목은 문서상 권고가 아니라 실제 운영 보완 사항으로 봐야 한다.

## 10. 계정 및 권한 운영 기준

현재 문서 기준 계정 구조는 아래와 같다.

- `log_writer@192.168.35.113`
  - 권한: `SELECT, INSERT, UPDATE on web_logs.*`
- `log_reader@192.168.35.%`
  - 권한: `SELECT on web_logs.*`

권장 원칙은 다음과 같다.

### 10.1 적재 계정과 조회 계정을 분리한다

- shipper는 `log_writer`만 사용한다.
- export나 조회 도구는 `log_reader`를 사용한다.
- 분석기나 리포트 도구가 쓰기 권한을 가질 필요는 없다.

### 10.2 최소 권한 원칙을 적용한다

- 적재용 계정은 필요한 범위까지만 권한을 준다.
- root 또는 관리자 계정으로 적재하지 않는다.
- DB 스키마 변경 권한은 운영자에게만 남긴다.

### 10.3 네트워크 범위를 제한한다

- `log_writer`는 웹서버 IP 기준으로 제한한다.
- `log_reader`도 가능한 한 실제 조회 서버 범위로 좁히는 것이 좋다.

## 11. 로그 회전(logrotate) 운영 기준

실험 환경에서는 반복 요청과 점검 도구 실행 때문에 로그가 빠르게 커질 수 있다.
따라서 access/security/error 로그와 shipper app 로그에 회전 정책을 두는 것이 좋다.

권장 원칙:

- 일 단위 또는 크기 기준 회전
- 압축 저장
- 최근 N개만 유지
- 오래된 로그 자동 삭제
- 회전 후에도 shipper가 새 파일을 정상 추적할 수 있는지 확인

권장 예시:

```conf
/var/log/apache2/app_access.log
/var/log/apache2/app_security.log
/var/log/apache2/app_error.log
/var/log/apache2/apache_log_shipper.log {
    daily
    rotate 14
    compress
    missingok
    notifempty
    copytruncate
}
```

설명:

- `copytruncate`는 tail 기반 프로세스와 함께 쓸 때 단순한 편이다.
- 실제 운영에서는 `copytruncate` 대신 서비스 reload/reopen 방식이 더 나을 수 있으나,
  현재 실험 구조에서는 단순성과 재현성이 더 중요하므로 우선 적용 가능하다.

운영 후 확인 항목:

- 회전 후 state가 정상 유지되는가
- 새 로그 파일이 계속 적재되는가
- spool 증가 없이 적재가 이어지는가

---

## 12. 민감정보 및 보안 운영 원칙

기본 상시 저장에서는 아래 항목을 넣지 않는 것을 원칙으로 한다.

- Authorization 헤더 원문
- 전체 Cookie 값
- 요청 본문 원문
- 응답 본문 원문
- 비밀번호, 토큰, 세션 식별자 원문
- 개인식별정보가 포함될 수 있는 원문 본문

권장 방식:

- 길이만 저장
- 존재 여부만 저장
- 마스킹 또는 해시 처리
- 의심 요청만 별도 감사 로그로 분리

운영 원칙:

- DB에는 원본에 가깝게 저장하되 민감정보는 최소화한다.
- LLM 입력용 JSON은 원본보다 더 보수적으로 정제한다.
- 분석 편의 때문에 민감정보 수집 범위를 넓히지 않는다.
- HTML fallback 판정 강화가 필요하면 응답 본문 전체 대신 **선택적 `resp_html_*` metadata**만 보조적으로 저장한다.

## 13. 장애 대응 운영 절차

## 13.1 DB 연결 실패

증상:

- shipper app 로그에 DB 연결 실패 또는 flush 실패 기록
- spool 파일 누적
- DB row 증가 중단

점검 순서:

1. DB 서버 접근 가능 여부 확인
2. DB 포트와 bind-address 확인
3. 계정/비밀번호 확인
4. 대상 DB 및 테이블 존재 여부 확인
5. `--test-db`로 연결만 별도 점검

대응:

- 문제 해결 후 shipper 재시작
- spool replay가 진행되는지 확인
- spool 파일이 감소하는지 확인

## 13.2 로그 파싱 실패

증상:

- shipper app 로그에 parse warning 증가
- 특정 로그 유형만 적재량이 급감

점검 순서:

1. Apache 로그 포맷이 표준안과 일치하는지 확인
2. 로그 파일명이 `app_*` 기준과 일치하는지 확인
3. 직접 로그 한 줄 샘플로 파서 정합성 확인
4. Apache 설정 변경 여부 확인

대응:

- 로그 포맷을 표준안으로 되돌리거나
- shipper 파서를 현재 포맷에 맞게 수정한다.

## 13.3 state 손상 또는 중복 적재 우려

증상:

- 로그를 읽지 못하거나 이미 읽은 구간을 다시 읽는 현상

대응 원칙:

- 먼저 state 파일 내용을 백업한다.
- 테스트 환경에서만 `--reset-state`를 사용한다.
- 운영 환경에서는 중복 적재 위험을 검토한 뒤 제한적으로 수행한다.

## 13.4 spool 누적 지속

증상:

- `/var/spool/apache_log_shipper/` 아래 파일이 계속 쌓임

가능 원인:

- DB 권한 문제
- 스키마 불일치
- 데이터 타입 불일치
- 네트워크 장애
- 비정상 종료 반복

대응:

- app 로그의 첫 에러 원인부터 확인한다.
- DB flush가 성공하는지 `--once` 또는 `--test-db`로 검증한다.
- 원인 해결 전 spool 파일을 임의 삭제하지 않는다.

## 13.5 `resp_html_*` 컬럼이 계속 NULL 인 경우

증상:

- `resp_content_type` 이 `text/html` 인 요청이 있어도 `resp_html_norm_fingerprint`, `resp_html_baseline_match` 등이 계속 `NULL`

가능 원인:

- Apache security log 에 `resp_html_*` key 자체가 기록되지 않음
- `resp_html_*` note 값을 생성하는 상류 로직이 없음
- shipper 이전 단계에서 항상 `"-"`만 기록됨
- fallback fingerprint 기능을 의도적으로 보류함

대응:

- security log 포맷과 실제 `resp_html_*` 출력값부터 먼저 확인한다.
- shipper 이전 단계에 값 생성 로직이 있는지 점검한다.
- mismatch 또는 NULL 만으로 파일 노출 성공으로 단정하지 않는다.

## 14. JSON export 운영 절차

DB 적재가 안정적으로 이뤄진 뒤에는 `export_db_logs_cli.py`로 JSON을 생성한다.

기본 원칙:

- DB 저장 시각은 UTC 기준으로 가정한다.
- 사용자는 KST 기준으로 범위를 입력한다.
- 출력 JSON 시간 필드는 KST 문자열로 변환한다.

대표 사용 방식:

### 14.1 routine 분석용 security export

```bash
python3 export_db_logs_cli.py \
  --host 192.168.35.223 \
  --user log_reader \
  --today \
  --table security \
  --pretty \
  --out /opt/web_log_analysis/raw/security_today_kst.json
```

### 14.2 특정 날짜 security export

```bash
python3 export_db_logs_cli.py \
  --host 192.168.35.223 \
  --user log_reader \
  --date 2026-04-02 \
  --table security \
  --pretty \
  --out /opt/web_log_analysis/raw/security_2026-04-02_kst.json
```

### 14.3 특정 시간대 error 연계 export

```bash
python3 export_db_logs_cli.py \
  --host 192.168.35.223 \
  --user log_reader \
  --start '2026-04-02 20:45:00' \
  --end   '2026-04-02 21:00:00' \
  --table error \
  --pretty \
  --out /opt/web_log_analysis/raw/error_2045_2100_kst.json
```

### 14.4 원본 보존용 all export

```bash
python3 export_db_logs_cli.py \
  --host 192.168.35.223 \
  --user log_reader \
  --today \
  --table all \
  --pretty \
  --out /opt/web_log_analysis/raw/today_all_kst.json
```

### 14.5 interactive 모드

```bash
python3 export_db_logs_cli.py --interactive
```

활용 목적:

- 발표용 샘플 확보
- 특정 시간대 사고 분석
- LLM 입력용 원본 JSON 생성

## 15. 분석 연계 운영 기준

DB 적재 이후 분석 단계에서는 원본 JSON 전체를 바로 LLM에 넘기지 않는 것을 기본 원칙으로 한다.

권장 흐름:

1. routine 분석용 `security` export 확보
2. 필요 시 원본 보존용 `all` export 확보
3. 필요 시 5xx/예외 조사용 `error` export 확보
4. 노이즈 식별
5. `socket.io` polling 등 정상 반복 요청 집계 또는 제외
6. 의심 요청 후보 추출
7. 분석용 정제 JSON 생성
8. LLM 1차 분류
9. LLM 2차 보고서 작성

운영 원칙:

- DB는 원본 보존 계층이다.
- LLM 입력은 분석용 정제 계층이다.
- routine 후보 생성의 기본 입력은 `security` 이다.
- `error` 는 원인 확인용 보조 근거다.
- `access` 는 운영 확인과 기준선 비교용 보조 자료다.
- 잡음은 삭제보다 분리와 집계를 우선한다.
- 보고서 작성 전에는 known asset IP 와 테스트 호스트 목록을 함께 확인한다.
- HTML fingerprint metadata 는 있으면 fallback HTML 가능성 강화 근거로만 사용한다.

예시 출력 파일:

- `security_2026-04-02_kst.json`
- `security_2026-04-02_kst_llm_input.json`
- `security_2026-04-02_kst_analysis_candidates.json`
- `security_2026-04-02_kst_noise_summary.json`
- `security_2026-04-02_kst_stage1_results.json`
- `security_2026-04-02_kst_stage2_report.md`

## 16. 운영 체크리스트

### 16.1 서비스 시작 전

- Apache 설정 문법 확인 완료
- access/security/error 로그 파일 생성 확인
- `web_logs` DB 및 3개 테이블 생성 확인
- `log_writer` / `log_reader` 계정 확인
- shipper 환경파일 작성 완료
- state/spool 디렉터리 권한 확인
- `--test-db` 성공 확인

### 16.2 정상 운영 중

- `apache_log_shipper.log`에 반복 예외가 없는가
- DB row가 계속 증가하는가
- spool 파일이 쌓이지 않는가
- 로그 회전 후에도 적재가 이어지는가
- export가 KST 기준으로 정상 생성되는가
- `resp_html_*` 키가 로그에 있으면 DB에 적재되는가

### 16.3 분석 전

- routine 분석용 `security` export 확보
- 필요 시 원본 보존용 `all` export 확보
- 필요 시 `error` export 확보
- noise summary 생성 여부 확인
- `socket.io` polling 과잉 포함 여부 확인
- 후보 요청 수가 지나치게 적거나 많지 않은지 확인
- known asset IP 와 테스트 호스트 목록 확인
- 보고서 생성 전 표본 검토 수행
- path traversal 200 응답이 있는 경우 `likely_html_fallback_response` 와 `resp_content_type` 우선 확인

## 17. 최종 운영 원칙 요약

이 환경의 운영 원칙은 아래 문장으로 요약할 수 있다.

> Apache는 로그를 생성하고, shipper는 안전하게 적재하며, MariaDB는 원본을 보존하고, 분석은 KST 기준 export 이후 정제 계층에서 수행한다.

또한 보안 운영 측면의 핵심은 아래와 같다.

- 적재 계정과 조회 계정을 분리한다.
- DB 비밀번호는 코드가 아니라 환경파일로 관리한다.
- state와 spool을 이용해 장애 시 데이터 유실을 줄인다.
- 민감정보는 기본 상시 저장에서 제외한다.
- LLM에는 원본 전체가 아니라 정제본을 우선 전달한다.
- 응답 본문 전체 대신 `likely_html_fallback_response` 와 선택적 `resp_html_*` metadata 를 보수적으로 활용한다.
