# 04_로그_적재_및_운영

- 문서 상태: 정리본
- 버전: v1.4
- 작성일: 2026-04-09
- 기준 코드:
  - `src/apache_log_shipper.py`
  - `src/export_db_logs_cli.py`
  - `src/prepare_llm_input.py`

## 1. 목적

Apache 로그 적재와 JSON export 연계를 현재 코드 기준으로 정리한다. 이 문서는 운영 절차만 남기고, 설계 배경 설명은 최소화한다.

## 2. 전체 구조

```text
Apache
  ↓
app_access.log / app_security.log / app_error.log
  ↓
apache_log_shipper.py
  ↓
MariaDB(web_logs)
  ↓
export_db_logs_cli.py
  ↓
prepare_llm_input.py 이후 LLM 분석
```

## 3. 역할 구분

### 3.1 웹서버

- Apache 로그 생성
- shipper 실행

### 3.2 DB 서버

- `web_logs` 저장
- 원본 로그 보존

### 3.3 LLM 서버

- DB export
- 전처리
- stage1 분류
- stage2 보고서 생성

## 4. 로그 역할

- `access`: 운영 확인용
- `security`: 분석 기본 입력
- `error`: 예외/5xx 상관분석용 보조 입력

## 5. shipper 현재 동작

`src/apache_log_shipper.py` 기준 역할:

- access/security/error 로그 파일 tail
- line 파싱
- `apache_access_logs`, `apache_security_logs`, `apache_error_logs` 배치 INSERT
- flush 실패 시 spool 저장
- spool 재전송

현재 shipper는 **HTML fingerprint를 계산하지 않는다.**

- security 로그에 `resp_html_*` 키가 있으면 DB 컬럼으로 적재할 수 있다.
- 하지만 값을 직접 생성하지는 않는다.

## 6. 운영 경로

웹서버 기준 주요 경로:

- `/var/log/apache2/app_access.log`
- `/var/log/apache2/app_security.log`
- `/var/log/apache2/app_error.log`
- `/var/log/apache2/apache_log_shipper.log`
- `/var/lib/apache_log_shipper/`
- `/var/spool/apache_log_shipper/`

분석 VM 기준 권장 경로:

- `/opt/web_log_analysis/src/export_db_logs_cli.py`
- `/opt/web_log_analysis/data/raw/`
- `/opt/web_log_analysis/processed/`
- `/opt/web_log_analysis/reports/`

## 7. shipper 실행 옵션

- `--test-db`: DB 연결만 확인
- `--once`: 현재 증분만 읽고 flush 후 종료
- `--reset-state`: 저장된 offset 초기화

`--reset-state` 는 중복 적재 위험이 있으므로 테스트 상황에서만 제한적으로 사용한다.

## 9. 장애 대응 최소 기준

### 9.1 DB 연결 실패

증상:

- shipper 로그에 DB 오류
- spool 누적
- DB row 증가 중단

확인:

- DB 접속 정보
- 네트워크
- 계정 권한
- 대상 테이블 존재 여부
- `--test-db`

### 9.2 로그 파싱 실패

증상:

- shipper 로그에 parse warning 증가
- 특정 로그만 적재량 급감

확인:

- Apache 로그 포맷
- 로그 파일 경로
- 파서와 로그 샘플 정합성

### 9.3 spool 누적

확인:

- 첫 DB 에러 원인
- 스키마 불일치
- 데이터 타입 오류
- 반복 재시작 여부

운영 원칙:

- 원인 해결 전 spool 파일 임의 삭제 금지

## 10. `resp_html_*` 관련 운영 기준

현재 운영 기준:

- `resp_html_*` 는 현재 핵심 분석 기준이 아니다.
- 값 생성 로직은 보류 상태다.
- 로그에 값이 있으면 DB에 적재될 수 있지만, 기본 운영 기준으로 기대하지 않는다.

현재 실제 분석에서 유지할 필드:

- `resp_content_type`
- `response_body_bytes`
- `raw_request_target`
- `path_normalized_from_raw_request`
- `likely_html_fallback_response`

현재 보류 또는 비핵심 필드:

- `resp_html_norm_fingerprint`
- `resp_html_fingerprint_version`
- `resp_html_baseline_name`
- `resp_html_baseline_match`
- `resp_html_baseline_confidence`
- `resp_html_features_json`

## 11. export 운영 기준

`export_db_logs_cli.py` 기준:

- 기본 테이블: `security`
- 시간 입력: KST
- DB 조회: UTC 변환 후 수행
- 출력 시간: KST ISO-8601

예시:

```bash
python3 /opt/web_log_analysis/src/export_db_logs_cli.py \
  --host 192.168.35.223 \
  --user log_reader \
  --date 2026-04-02 \
  --table security \
  --pretty
```
