# 04_로그_적재_및_운영

- 문서 상태: 운영 정책 문서
- 목적: shipper, spool, 장애 대응, 로그 적재 운영 기준을 정리한다.

실제 export/prepare/stage1/stage2/pipeline 실행 명령은 [docs/01_운영_기준_실행_가이드.md](docs/01_운영_기준_실행_가이드.md)를 본다.

## 1. 전체 구조

```text
Apache
  ↓
app_access.log / app_security.log / app_error.log
  ↓
apache_log_shipper.py
  ↓
MariaDB(web_logs)
  ↓
LLM 서버 export / prepare / stage1 / stage2
```

## 2. 역할 구분

- 웹서버: Apache 로그 생성, shipper 실행
- DB 서버: `web_logs` 저장
- LLM 서버: DB export 이후 분석 수행

## 3. 로그 역할

- `access`: 운영 확인용
- `security`: 분석 기본 입력
- `error`: 예외와 5xx 상관분석 보조

## 4. 웹서버 운영 경로

- `/var/log/apache2/app_access.log`
- `/var/log/apache2/app_security.log`
- `/var/log/apache2/app_error.log`
- `/var/log/apache2/apache_log_shipper.log`
- `/var/lib/apache_log_shipper/`
- `/var/spool/apache_log_shipper/`

## 5. LLM 서버 운영 경로

- `/opt/web_log_analysis/data/raw/`
- `/opt/web_log_analysis/data/processed/`
- `/opt/web_log_analysis/reports/`
- `/opt/web_log_analysis/logs/`

## 6. shipper 현재 동작

- access/security/error 로그 파일 tail
- line 파싱
- `apache_access_logs`, `apache_security_logs`, `apache_error_logs` 배치 INSERT
- flush 실패 시 spool 저장
- spool 재전송

현재 기준:

- shipper는 `resp_html_*` 값을 직접 생성하지 않는다.
- security 로그에 해당 키가 이미 있으면 DB 컬럼에 적재될 수 있다.

## 7. shipper 실행 옵션

- `--test-db`: DB 연결만 확인
- `--once`: 현재 증분만 읽고 flush 후 종료
- `--reset-state`: 저장된 offset 초기화

주의:

- `--reset-state`는 중복 적재 위험이 있으므로 테스트 상황에서만 제한적으로 사용한다.

## 8. 환경변수

- `LOG_DB_HOST`
- `LOG_DB_PORT`
- `LOG_DB_USER`
- `LOG_DB_PASSWORD`
- `LOG_DB_NAME`
- `APACHE_ACCESS_LOG`
- `APACHE_SECURITY_LOG`
- `APACHE_ERROR_LOG`
- `SHIPPER_STATE_DIR`
- `SHIPPER_SPOOL_DIR`
- `SHIPPER_APP_LOG`

주의:

- 현재 코드에는 일부 기본값이 들어 있을 수 있다.
- 운영 시 민감 정보는 환경파일 또는 환경변수로만 주입하는 편이 맞다.

## 9. 장애 대응 최소 기준

### 9.1 DB 연결 실패

확인:

- DB 접속 정보
- 네트워크
- 계정 권한
- 대상 테이블 존재 여부
- `--test-db`

### 9.2 로그 파싱 실패

확인:

- Apache 로그 포맷
- 로그 파일 경로
- 파서와 로그 샘플 정합성

### 9.3 spool 누적

운영 원칙:

- 첫 DB 에러 원인을 먼저 찾는다.
- 원인 해결 전 spool 파일을 임의 삭제하지 않는다.

## 10. `resp_html_*` 운영 기준

현재 운영 기준:

- `resp_html_*`는 핵심 분석 기준이 아니다.
- 값 생성 로직은 보류 상태다.
- 로그에 값이 있으면 DB에 적재될 수 있지만 기본 운영 기준으로 기대하지 않는다.

실제 분석에서 우선 보는 필드:

- `resp_content_type`
- `response_body_bytes`
- `raw_request_target`
- `path_normalized_from_raw_request`
- `likely_html_fallback_response`
