## 1. 문서 목적

이 문서는 현재 진행 중인 프로젝트가 **무엇을 목표로 하는지**, 그리고 그 목표에 맞추어 **어떤 실험 대상을 채택했는지**를 정리하기 위한 문서이다.

기존에는 단순한 웹사이트 또는 쇼핑몰 구축 관점에서 실습 대상을 검토했으나, 프로젝트가 구체화되면서 목적이 달라졌다. 현재 프로젝트의 중심은 단순 서비스 구축이 아니라, 다음과 같은 보안 분석 실험에 있다.

- 정상 요청과 공격성 요청을 함께 발생시키는 것
- 웹서버 로그를 체계적으로 수집하는 것
- 수집한 로그를 DB에 저장하는 것
- 저장된 로그를 바탕으로 공격 징후를 탐지하고 분류하는 것
- 추후 LLM 또는 규칙 기반 분석기로 해석 가능한 형태의 데이터셋을 만드는 것

즉, 이 문서는 “어떤 웹 애플리케이션을 설치할 것인가”보다 먼저, **왜 그 애플리케이션을 선택해야 하는가**를 설명하는 방향 문서이다.

## 2. 프로젝트의 출발점과 방향 변화

초기 단계에서는 Ubuntu 22.04 환경에 Apache2, PHP, MySQL, OpenCart를 조합하여 **간단한 쇼핑몰 구축 실습**을 하는 방향을 검토하였다. 이 구성은 다음과 같은 학습 목표에는 적합하다.

- Ubuntu 서버 설치와 기본 운영 실습
- Apache 기반 웹서버 구축 실습
- 데이터베이스 연동 구조 이해
- 관리자 페이지와 상품 관리 기능 확인
- 기본 웹 로그 확인

하지만 프로젝트의 목적이 점차 구체화되면서, 단순한 쇼핑몰 설치보다 **공격 요청을 발생시키고 그 로그를 분석하는 실험**이 더 중요해졌다. 이 시점부터 실습 대상 선정 기준은 “서비스 구축이 쉬운가”보다 “공격 재현과 로그 수집에 적합한가”로 바뀌었다.

따라서 현재 프로젝트는 일반적인 웹서비스 운영 실습이 아니라, **웹 공격 탐지와 로그 기반 분류 실험 프로젝트**로 보는 것이 맞다.

## 3. 현재 프로젝트의 핵심 목표

현재 프로젝트의 핵심 목표는 아래와 같이 정리할 수 있다.

### 3.1 로그 수집 중심의 실험 환경 구축

모든 외부 요청이 하나의 지점을 통과하도록 구성하고, 그 지점에서 웹 로그를 안정적으로 남기는 것이 중요하다. 이를 통해 정상 요청과 공격성 요청을 같은 기준으로 수집할 수 있어야 한다.

### 3.2 공격 재현이 가능한 실험 대상 선정

공격 탐지 실험을 위해서는 다양한 웹 공격 시나리오를 반복적으로 재현할 수 있어야 한다. 단순한 정상 서비스보다, 보안 실습과 공격 시뮬레이션에 적합한 대상이 필요하다.

### 3.3 DB 저장 및 후속 분석 가능성 확보

수집한 로그는 파일로만 보관하는 데 그치지 않고, MariaDB 같은 중앙 저장소에 정리해 저장해야 한다. 그래야 조건 검색, 집계, 상관분석, 라벨링, 보고서 자동화 같은 후속 작업이 가능해진다.

### 3.4 탐지 및 분류용 데이터셋 축적

프로젝트의 최종 활용 방향은 단순한 로그 확인이 아니라, 공격 징후 탐지와 요청 분류에 활용할 수 있는 데이터셋을 만드는 것이다. 따라서 정상 요청과 공격 요청이 모두 포함된 구조가 필요하다.

### 3.5 합법적이고 통제된 실험 환경 유지

실험은 반드시 본인이 관리하거나 허가받은 환경에서 수행되어야 하며, 대상 애플리케이션 역시 교육용·실습용 맥락에서 설명 가능해야 한다.

## 4. 실험 대상 선정 기준

현재 프로젝트 목적에 맞는 실험 대상을 고르기 위해, 다음 기준을 우선적으로 적용한다.

1. 다양한 공격 시나리오를 반복적으로 재현할 수 있어야 한다.
2. 정상 요청과 공격성 요청을 함께 수집하기 쉬워야 한다.
3. Apache 앞단 로그 수집 구조와 결합하기 쉬워야 한다.
4. 설치와 재배포가 비교적 단순해야 한다.
5. 보고서에서 합법적이고 통제된 실험 환경이라고 설명하기 쉬워야 한다.
6. 추후 DB 저장, 라벨링, 탐지 모델 실험으로 확장하기 쉬워야 한다.

이 기준에 따르면, 단순 쇼핑몰 운영용 애플리케이션보다 **보안 실습용 애플리케이션**이 더 적합하다.

## 5. 왜 OpenCart가 최종 대상이 아닌가

OpenCart는 무료 오픈소스 쇼핑몰 플랫폼으로서, 일반적인 전자상거래 웹서비스 구축을 실습하는 데에는 충분히 의미가 있다. 특히 다음과 같은 장점이 있다.

- 비교적 익숙한 웹서비스 구조를 제공한다.
- Apache, PHP, MySQL 조합과 잘 맞는다.
- 관리자 페이지와 상품 관리 구조를 확인하기 좋다.
- 기본적인 웹 로그 확인과 설치 실습에는 적합하다.

그러나 현재 프로젝트는 “쇼핑몰 구축”이 아니라 “공격 로그 수집과 탐지 실험”이 중심이다. 이 관점에서 OpenCart는 다음과 같은 한계가 있다.

- 다양한 공격 시나리오를 구조적으로 제공하지 않는다.
- 반복 가능한 공격 재현 실험의 출발점으로는 설명력이 약하다.
- 공격 의도와 유형을 비교적 명확히 분리해 로그를 축적하기 어렵다.
- 실험 대상 선택 이유를 보안 실습 관점에서 설명하기가 상대적으로 불리하다.

따라서 OpenCart는 **초기 검토안 또는 비교 대상**으로는 의미가 있지만, 현재 프로젝트의 최종 실험 대상으로 두기에는 우선순위가 낮다.

## 6. 최종 실험 대상: OWASP Juice Shop

현재 프로젝트의 실험 대상은 **OWASP Juice Shop**으로 정리한다.

OWASP Juice Shop은 보안 교육과 공격 재현을 위해 의도적으로 취약하게 설계된 웹 애플리케이션이다. 이 점에서 일반 쇼핑몰 애플리케이션과는 출발 목적이 다르며, 현재 프로젝트와 훨씬 잘 맞는다.

### 6.1 채택 이유

OWASP Juice Shop을 최종 실험 대상으로 선택한 이유는 다음과 같다.

1. **보안 실습용으로 설계된 애플리케이션**이기 때문이다.
2. **다양한 공격 시나리오를 한 환경에서 반복 재현**하기 쉽다.
3. **정상 요청과 공격성 요청을 함께 수집**하기 유리하다.
4. **공격 유형별 로그 축적과 라벨링**에 유리하다.
5. **합법적이고 통제된 실험 환경**이라는 점을 설명하기 쉽다.
6. Apache 리버스 프록시와 결합하면 **기존 로그 수집 전략을 그대로 유지**할 수 있다.

### 6.2 실험 관점에서의 장점

Juice Shop은 단순히 취약점이 있는 서비스가 아니라, 보안 훈련과 재현 실험에 적합하도록 잘 알려진 대상이다. 따라서 다음과 같은 이점이 있다.

- 공격 도구 또는 수동 테스트 시나리오 적용이 쉽다.
- 실험을 다시 수행했을 때 재현성이 높다.
- 공격 탐지용 로그 수집 실험의 목적을 설명하기 쉽다.
- 나중에 WAF, 포렌식 로그, 탐지 규칙, 분류 모델 실험까지 확장하기 쉽다.

## 7. 최종 환경 구성 방향

현재 프로젝트의 상위 구성 방향은 다음과 같이 정리한다.

### 7.1 운영체제

- **Ubuntu 22.04 LTS Server**

선정 이유:

- 장기 지원 기반이라 안정성이 높다.
- 서버 실습 자료와 참고 문서가 풍부하다.
- Apache, Docker 기반 실습과 궁합이 좋다.

### 7.2 앞단 웹서버

- **Apache2**

선정 이유:

- 외부 요청을 먼저 받아 로그 수집 지점을 일원화할 수 있다.
- Reverse Proxy 구성이 쉽다.
- 접근 로그, 보안 로그, 에러 로그를 분리해 설계하기 좋다.
- 실험 환경에서 로그 포맷을 세밀하게 제어할 수 있다.

### 7.3 애플리케이션 실행 방식

- **Docker 컨테이너 기반 실행**

선정 이유:

- Juice Shop 배포와 재실행이 단순하다.
- 실험 환경 재현성이 높다.
- Apache와 애플리케이션 역할을 분리하기 쉽다.
- 실험 중 문제가 생겨도 복구와 재배포가 비교적 간단하다.

### 7.4 실험 대상 애플리케이션

- **OWASP Juice Shop**

선정 이유:

- 공격 재현과 로그 수집에 적합하다.
- 보안 실습 환경으로 설명하기 쉽다.
- 정상 요청과 공격성 요청을 함께 다루기 좋다.

## 8. 실험 환경의 기본 구조

프로젝트의 기본 구조는 다음과 같다.

```text
브라우저 / 점검 도구 / 실험 스크립트
                ↓
              Apache2
                ↓  Reverse Proxy
     OWASP Juice Shop (Docker, 127.0.0.1:3000)
                ↓
     access / security / error 로그 파일
                ↓
        Python log shipper
                ↓
         MariaDB (web_logs)
```

이 구조의 핵심은 **모든 외부 요청이 Apache를 먼저 지나가게 하는 것**이다. 이렇게 하면 요청 수집 지점을 고정할 수 있고, 이후 DB 저장과 분석 단계도 일관되게 가져갈 수 있다.

## 9. 본 프로젝트에서 다루는 실험 범위

이 프로젝트에서 우선적으로 다루는 범위는 다음과 같다.

### 9.1 포함 범위

- Ubuntu 22.04 기반 실습 서버 구성
- Apache2 Reverse Proxy 기반 웹 요청 수집
- Juice Shop 대상 정상 요청 및 공격성 요청 발생
- Apache 로그 수집
- Python shipper를 통한 DB 적재
- MariaDB 기반 조회, 분석, 라벨링 준비
- 추후 LLM 또는 규칙 기반 분석기로 연결 가능한 형태의 데이터 정리

### 9.2 제외 또는 후속 범위

아래 항목은 본 문서에서 직접 다루지 않고, 후속 문서에서 구체화한다.

- Apache VirtualHost 상세 설정 문법
- access / security / error 로그 포맷 상세
- MariaDB 테이블 정의 상세
- Python shipper 구현 상세
- 탐지 정규식과 위험도 점수 계산 상세
- 실시간 경보, 대시보드, 시각화 구현 상세

즉, 본 문서는 프로젝트의 **방향과 실험 대상 선정의 기준서**이며, 구현 상세는 별도 문서에서 다룬다.

## 10. 운영 및 윤리 원칙

실험 환경은 반드시 다음 원칙을 따른다.

1. 실험은 본인이 관리하거나 허가받은 시스템에서만 수행한다.
2. Juice Shop은 학습용 취약 애플리케이션이므로 외부 인터넷에 무분별하게 공개하지 않는다.
3. 정상 요청과 공격성 요청을 구분해 수집하고, 실험 기록을 함께 남긴다.
4. 로그에는 민감정보가 포함될 수 있으므로 저장 범위를 최소화하고 필요 시 마스킹 정책을 둔다.

이 원칙은 프로젝트의 기술적 정당성뿐 아니라 보고서의 설명 가능성을 위해서도 중요하다.

## 11. 결론

현재 프로젝트는 단순한 쇼핑몰 설치 실습이 아니라, **웹 공격 탐지와 웹 로그 기반 분류 실험을 위한 환경 구축 프로젝트**로 정리할 수 있다.

이 목적에 맞추어 최종 방향은 다음과 같이 정리한다.

- 운영체제는 **Ubuntu 22.04 LTS Server**를 사용한다.
- 앞단 웹서버는 **Apache2**를 사용한다.
- 애플리케이션 실행 방식은 **Docker**를 사용한다.
- 실험 대상 애플리케이션은 **OWASP Juice Shop**으로 한다.
- 로그 수집의 중심 지점은 **Apache**로 통일한다.
- 수집된 로그는 후속 분석과 분류를 위해 **DB 저장 및 데이터셋화**를 전제로 설계한다.

즉, 본 프로젝트의 실험 대상은 “구축용 쇼핑몰”이 아니라, **공격 재현과 로그 수집에 적합한 보안 실습용 웹 애플리케이션**이며, 그 기준에 가장 적합한 대상이 OWASP Juice Shop이다.
