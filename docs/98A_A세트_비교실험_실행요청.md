# 98A_A세트_비교실험_실행요청

- 작성 기준일: 2026-04-24
- 문서 역할: `docs/98_비교_실험_요청_세트_표준.md`를 따르는 A세트 실행 문서
- 적용 범위: Juice Shop 기준 baseline/auth 비교 실험

이 문서는 상위 표준안 기준의 서비스별 실행 문서다.

- A세트는 baseline/auth 전용이다.
- SQLi / XSS / Traversal / HPP / Dir Probe 는 별도 세트(B/C/D 등)로 분리한다.
- 향후 같은 형식으로 다른 웹서비스용 A세트 문서도 만들 수 있다.

---

## 1. 실험 목적

A세트는 다음을 확인하기 위한 baseline/auth 중심 요청 세트다.

- 정상 조회/검색 baseline 생성
- 인증 실패와 반복 실패의 해석 차이 확인
- Juice Shop authentication bypass 성공 요청의 탐지/서술 방식 확인
- provider별 stage1 / stage2 결과 차이 비교

이 문서는 요청 세트만 다룬다. 시간 구간 기록, export, pipeline, 산출물 보관, manifest 백업은 상위 표준 문서(`docs/98`)를 따른다.

---

## 2. 기본 변수

```bash
BASE_URL="http://192.168.56.105"
UA_PREFIX="lab-a-set"
```

모든 요청은 `curl` 기준으로 수행한다.

---

## 3. A세트 메타 정의

- `webservice`: `juice-shop`
- `experiment_set`: `A`
- `attack_category`: `baseline`, `auth`
- 권장 실행 시간: 10분 내외의 짧은 시간 구간
- 권장 방식: 시작 시각 기록 -> 요청 세트 실행 -> 종료 시각 기록 -> `--start / --end` export

provider 비교 시 공통 조건:

- 같은 raw log
- 같은 `known_asset_ips`
- 같은 top-N
- 같은 `mode`
- 같은 `config/llm.env`를 한 번만 적용
- provider는 각 실행마다 CLI에서 명시

---

## 4. 실행 순서

1. 실험 시작 시각을 기록한다.
2. 아래 요청 세트를 순서대로 수행한다.
3. 실험 종료 시각을 기록한다.
4. `docs/98`의 시간 구간 export 절차를 따른다.
5. provider별로 서로 다른 `--base-name`으로 pipeline을 실행한다.
6. 결과 확인 시 `meta.provider`가 기대값과 일치하는지 확인한다.

---

## 5. 정상 baseline 세트

### A-01 Home

```bash
curl -i -A "${UA_PREFIX}-home-1" "$BASE_URL/"
```

- 기대 ground truth: 정상 baseline
- 기대 관찰 포인트: 후보 밖 정상 처리 또는 일반 정상 트래픽 유지

### A-02 Product List

```bash
curl -i -A "${UA_PREFIX}-products-1" "$BASE_URL/rest/products"
```

- 기대 ground truth: 정상 baseline
- 기대 관찰 포인트: 조회형 요청을 공격 후보로 과도하게 올리지 않는지 확인

### A-03 Search apple

```bash
curl -i -A "${UA_PREFIX}-search-apple-1" "$BASE_URL/rest/products/search?q=apple"
```

- 기대 ground truth: 정상 baseline
- 기대 관찰 포인트: `benign_normal_search` 또는 후보 밖 정상 처리 여부

### A-04 Search phone

```bash
curl -i -A "${UA_PREFIX}-search-phone-1" "$BASE_URL/rest/products/search?q=phone"
```

- 기대 ground truth: 정상 baseline
- 기대 관찰 포인트: 일반 검색과 공격성 검색의 경계 해석

### A-05 Search bag

```bash
curl -i -A "${UA_PREFIX}-search-bag-1" "$BASE_URL/rest/products/search?q=bag"
```

- 기대 ground truth: 정상 baseline
- 기대 관찰 포인트: provider별 오탐 여부 비교

### A-06 Product Detail

```bash
curl -i -A "${UA_PREFIX}-product-1-1" "$BASE_URL/rest/products/1"
```

- 기대 ground truth: 정상 baseline
- 기대 관찰 포인트: 단순 상세 조회를 정상 트래픽으로 유지하는지 확인

### A-07 WhoAmI

```bash
curl -i -A "${UA_PREFIX}-whoami-1" "$BASE_URL/rest/user/whoami"
```

- 기대 ground truth: 정상 baseline
- 기대 관찰 포인트: 비인증 상태 API 확인 요청을 오탐하지 않는지 확인

### A-08 Mixed browsing patterns

```bash
curl -i -A "${UA_PREFIX}-browse-apple-1" "$BASE_URL/rest/products/search?q=apple"
curl -i -A "${UA_PREFIX}-browse-phone-1" "$BASE_URL/rest/products/search?q=phone"
curl -i -A "${UA_PREFIX}-browse-bag-1" "$BASE_URL/rest/products/search?q=bag"
```

- 기대 ground truth: 정상 baseline
- 기대 관찰 포인트: 짧은 연속 탐색을 benign browsing으로 유지하는지 확인

## 6. 인증 / 로그인 세트

### A-09 Login Fail (single)

```bash
curl -i \
  -A "${UA_PREFIX}-login-fail-1" \
  -H "Content-Type: application/json" \
  -d '{"email":"admin@juice-sh.op","password":"1234"}' \
  "$BASE_URL/rest/user/login"
```

- 기대 ground truth: 단일 로그인 실패
- 기대 관찰 포인트: `suspicious_auth_abuse`와 `likely_false_positive` 사이 경계 해석

### A-10 Login Fail (repeat x3)

```bash
curl -i -A "${UA_PREFIX}-login-repeat-1" -H "Content-Type: application/json" -d '{"email":"admin@juice-sh.op","password":"1234"}' "$BASE_URL/rest/user/login"
sleep 2
curl -i -A "${UA_PREFIX}-login-repeat-2" -H "Content-Type: application/json" -d '{"email":"admin@juice-sh.op","password":"1234"}' "$BASE_URL/rest/user/login"
sleep 2
curl -i -A "${UA_PREFIX}-login-repeat-3" -H "Content-Type: application/json" -d '{"email":"admin@juice-sh.op","password":"1234"}' "$BASE_URL/rest/user/login"
```

- 기대 ground truth: 반복 로그인 실패
- 기대 관찰 포인트: 반복성 때문에 단일 실패보다 더 공격적으로 해석하는지 확인

### A-11 Authentication Bypass Success (confirmed Juice Shop request)

아래 요청은 실제 확인된 Juice Shop authentication bypass 성공 요청을 반영한 것이다.

```bash
curl -i \
  -A "${UA_PREFIX}-login-bypass-1" \
  -H "Content-Type: application/json" \
  -d '{"email":"admin@juice-sh.op'\'' -- ","password":"x"}' \
  "$BASE_URL/rest/user/login"
```

- 기대 ground truth: 성공한 인증 우회 시도
- 기대 관찰 포인트: 200 응답, token 발급, admin 계정 성공 흔적을 stage1/stage2가 단순 실패보다 강하게 서술하는지 확인

### A-12 Login Success (optional)

정상 계정이 준비된 경우에만 수행한다.

```bash
curl -i \
  -A "${UA_PREFIX}-login-success-1" \
  -H "Content-Type: application/json" \
  -d '{"email":"admin@juice-sh.op","password":"admin123"}' \
  "$BASE_URL/rest/user/login"
```

- 기대 ground truth: 정상 인증 성공
- 기대 관찰 포인트: 정상 로그인 성공을 오탐하지 않는지 확인

### A-13 Cart / Basket access (optional)

```bash
curl -i -A "${UA_PREFIX}-cart-1" "$BASE_URL/rest/basket/1"
```

- 기대 ground truth: 인증 상태 확인 또는 보호 리소스 접근 패턴
- 기대 관찰 포인트: 보호 리소스 접근을 어떤 수준의 신호로 해석하는지 확인

---

## 7. 실행 후 기록 항목

A세트 실행 후에는 최소한 아래를 기록한다.

- 시간 구간
- raw export 파일명
- provider / model
- 실제 결과 `meta.provider`
- 기대 ground truth
- prepare 결과
- stage1 verdict / severity / confidence
- stage2 핵심 요약
- OpenAI / Anthropic 차이
- 최종 평가

결과 검증 메모:

- OpenAI 실행 결과는 `stage1_results.json` 또는 `stage2_report.json`에서 `meta.provider=openai`인지 확인한다.
- Anthropic 실행 결과는 같은 위치에서 `meta.provider=anthropic`인지 확인한다.
- provider가 기대값과 다르면 env 오염 또는 CLI provider 누락을 먼저 의심한다.

---

## 8. 확장 기준

이 문서는 A세트 전용이지만, 구조는 B세트/C세트로 그대로 확장 가능하게 유지한다.

- A세트: baseline / auth
- B세트: SQLi
- C세트: XSS
- D세트: Traversal / HPP / Dir Probe

즉, 공통 절차는 `docs/98`에 두고, 각 세트 문서는 요청 목록과 ground truth만 분리해 관리한다.

---

## 9. 주의사항

- A세트는 baseline/auth 전용이다.
- SQLi / XSS / Traversal / HPP / Dir Probe 는 이 문서에 섞지 않는다.
- 모든 요청은 테스트 승인된 Juice Shop 환경에서만 수행한다.
- stage2 보고서는 사건 요약 보고서이므로 `request_id` 기준 raw evidence와 함께 확인한다.
- 상위 표준 문서의 단일 `llm.env` + CLI provider 명시 정책을 그대로 따른다.
