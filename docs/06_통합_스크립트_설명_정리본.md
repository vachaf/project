# 06_통합_스크립트_설명

- 문서 상태: 설명 문서
- 목적: 현재 파이프라인에 포함된 스크립트의 역할, 입력, 출력, 연결 관계를 빠르게 확인한다.
- 기준 코드:
  - `src/apache_log_shipper.py`
  - `src/export_db_logs_cli.py`
  - `src/prepare_llm_input.py`
  - `src/llm_stage1_classifier.py`
  - `src/llm_stage2_reporter.py`
  - `src/run_analysis_pipeline.py`

## 1. 전체 흐름

```text
Apache 로그 생성
  ↓
apache_log_shipper.py
  ↓
MariaDB(web_logs)
  ↓
export_db_logs_cli.py
  ↓
prepare_llm_input.py
  ↓
llm_stage1_classifier.py
  ↓
llm_stage2_reporter.py
```

`run_analysis_pipeline.py`는 export 이후 단계를 한 번에 실행하거나, 중간 산출물에서 재개할 수 있습니다.

## 2. 스크립트별 역할

### 2.1 `apache_log_shipper.py`

역할:
- Apache의 access, security, error 로그를 읽어서 MariaDB에 적재합니다.
- DB flush 실패 시 spool에 임시 저장하고, 이후 재전송을 시도합니다.

입력:
- `/var/log/apache2/app_access.log`
- `/var/log/apache2/app_security.log`
- `/var/log/apache2/app_error.log`
- 환경변수: `LOG_DB_*`, `APACHE_*`, `SHIPPER_*`

출력:
- `apache_access_logs`
- `apache_security_logs`
- `apache_error_logs`
- spool 파일
- `apache_log_shipper.log`

현재 기준:
- shipper는 `resp_html_*` 값을 직접 생성하지 않습니다.
- security 로그에 해당 키가 이미 있으면 DB 컬럼으로 적재할 수 있습니다.

### 2.2 `export_db_logs_cli.py`

역할:
- DB에 적재된 로그를 KST 기준 시간 범위로 조회해서 JSON으로 export 합니다.

입력:
- DB 접속 정보
- 시간 옵션: `--today`, `--date`, `--start/--end`
- 테이블 옵션: `--table access|security|error|all`

출력:
- export JSON 파일
- 상위 키: `meta`, `counts`, `data`

현재 기준:
- 기본 테이블은 `security`입니다.
- 사용자는 KST 기준으로 입력합니다.
- DB 조회는 UTC로 변환해서 수행합니다.
- 출력 시간은 KST ISO-8601 문자열입니다.

### 2.3 `prepare_llm_input.py`

역할:
- export JSON에서 분석 후보를 추리고, noise를 집계하고, stage1 입력용 JSON을 만듭니다.

입력:
- `export_db_logs_cli.py` 결과 JSON

출력:
- `<base>_llm_input.json`
- `<base>_analysis_candidates.json`
- `<base>_noise_summary.json`
- `<base>_filtered_out_rows.json` (`--write-filtered-out` 사용 시)

현재 기준:
- 기본 입력 소스는 `security`입니다.
- noise는 완전 삭제보다 분리와 집계를 우선합니다.
- 후보는 `incident_group_key` 기준으로 dedup 됩니다.
- 보수 해석에 직접 쓰는 필드는 `resp_content_type`, `response_body_bytes`, `raw_request_target`, `path_normalized_from_raw_request`, `likely_html_fallback_response`입니다.

### 2.4 `llm_stage1_classifier.py`

역할:
- `analysis_candidates`를 순회하며 후보별 1차 분류를 수행합니다.
- Structured Outputs(JSON Schema) 기반으로 결과를 고정 형식으로 저장합니다.

입력:
- `<base>_llm_input.json`
- `OPENAI_API_KEY`
- mode 또는 `--model`

출력:
- `<base>_stage1_results.json`
- `<base>_stage1_errors.json`

기본 모델:
- `routine`: `gpt-5.4-mini`
- `milestone`: `gpt-5.4`
- `presentation`: `gpt-5.4`

현재 기준:
- dry-run은 실제 live 분류가 아니라 요청 계획 또는 placeholder 결과 확인용입니다.
- 주요 결과 필드는 `verdict`, `severity`, `confidence`, `false_positive_possible`, `reasoning_summary`, `evidence_fields`, `recommended_actions`입니다.

### 2.5 `llm_stage2_reporter.py`

역할:
- stage1 결과를 바탕으로 incident를 요약하고 최종 Markdown 보고서를 생성합니다.

입력:
- `<base>_stage1_results.json`
- 필요 시 `<base>_llm_input.json`
- 필요 시 `<base>_stage1_errors.json`
- `KNOWN_ASSET_IPS`

출력:
- `<base>_stage2_report_input.json`
- `<base>_stage2_report.json`
- `<base>_stage2_report.md`
- `<base>_stage2_report_error.json`

기본 모델:
- `routine`: `gpt-5.4-mini`
- `milestone`: `gpt-5.4`
- `presentation`: `gpt-5.4`

현재 기준:
- `request_id` 우선으로 incident를 묶고, 없으면 fallback dedup 기준을 사용합니다.
- `known_asset_ips`는 prepare 후보 필드가 아니라 stage2 문맥 정보입니다.
- `low_signal_fuzzing`, `low_signal_dir_probe`는 기본적으로 incident로 승격하지 않고 후보 밖 탐색성 요청으로 유지합니다.

### 2.6 `run_analysis_pipeline.py`

역할:
- prepare → stage1 → stage2 흐름을 자동으로 실행합니다.
- 중간 산출물부터 재시작할 수 있습니다.

시작점:
- `--export-input`
- `--llm-input`
- `--stage1-results`

출력:
- `processed/` 아래 중간 산출물
- `reports/` 아래 최종 보고서
- `<work-dir>/pipeline_manifest.json`

현재 기준:
- 기본 `processed-dir`: `<work-dir>/processed`
- 기본 `reports-dir`: `<work-dir>/reports`
- 현재 파이프라인 관리용 기본 엔트리포인트는 이 스크립트입니다.

## 3. 현재 분석 기준

### 3.1 입력 우선순위

- `security`: 기본 분석 입력
- `error`: 예외, 5xx, 상관분석 보조 입력
- `access`: 운영 확인 및 기준선 비교용

### 3.2 보수 해석 기준

path traversal 계열에서는 아래 순서로 확인합니다.

1. `resp_content_type`
2. `response_body_bytes`
3. `raw_request_target`
4. `path_normalized_from_raw_request`
5. `likely_html_fallback_response`

`resp_html_*` fingerprint 계열은 현재 보류 또는 선택 항목으로 봅니다.

## 4. 자주 확인하는 파일

- 원본 export: `data/raw/*.json`
- prepare 결과: `processed/*_llm_input.json`
- stage1 결과: `processed/*_stage1_results.json`
- stage2 결과: `reports/*_stage2_report.md`
- 실행 기록: `pipeline_manifest.json`
