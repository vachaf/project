# 06_통합_스크립트_설명_정리본

- 문서 상태: 스크립트 참조 문서
- 목적: 각 스크립트의 역할, 입력, 출력, 연결 관계를 빠르게 확인한다.

실제 실행 명령은 [docs/01_운영_기준_실행_가이드.md](docs/01_운영_기준_실행_가이드.md)를 본다.

## 1. 전체 흐름

```text
Apache 로그 생성
  ↓
apache_log_shipper.py
  ↓
MariaDB(web_logs)
  ↓
export_db_logs_cli.py
  ↓
prepare_llm_input.py
  ↓
llm_stage1_classifier.py
  ↓
llm_stage2_reporter.py
```

`run_analysis_pipeline.py`는 export 이후 단계를 한 번에 실행하거나 중간 산출물에서 재개할 수 있다.

## 2. 스크립트별 역할

### 2.1 `apache_log_shipper.py`

- Apache access/security/error 로그를 읽어 MariaDB에 적재한다.
- DB flush 실패 시 spool에 임시 저장하고 재전송을 시도한다.
- `resp_html_*` 값을 직접 생성하지는 않는다.

입력:

- `/var/log/apache2/app_access.log`
- `/var/log/apache2/app_security.log`
- `/var/log/apache2/app_error.log`
- `LOG_DB_*`, `APACHE_*`, `SHIPPER_*`

출력:

- `apache_access_logs`
- `apache_security_logs`
- `apache_error_logs`
- spool 파일
- `apache_log_shipper.log`

### 2.2 `export_db_logs_cli.py`

- DB 로그를 KST 기준 시간 범위로 조회해 JSON으로 export 한다.
- 기본 테이블은 `security`다.
- 출력 JSON 상위 키는 `meta`, `counts`, `data`다.

### 2.3 `prepare_llm_input.py`

- export JSON에서 분석 후보를 추리고 noise를 집계한다.
- stage1 입력용 `*_llm_input.json`을 만든다.
- 기본 입력 소스는 `security`다.

출력:

- `<base>_llm_input.json`
- `<base>_analysis_candidates.json`
- `<base>_noise_summary.json`
- `<base>_filtered_out_rows.json` 선택

### 2.4 `llm_stage1_classifier.py`

- `analysis_candidates`를 순회하며 1차 분류를 수행한다.
- 코드상 provider 미지정 시 기본 fallback은 OpenAI다.
- 운영 기준에서는 provider를 생략하지 않고 `--provider openai` 또는 `--provider anthropic`를 명시한다.
- Claude 실행 시 `--provider anthropic --model "$ANTHROPIC_MODEL"` 패턴을 쓴다.

출력:

- `<base>_stage1_results.json`
- `<base>_stage1_errors.json`

### 2.5 `llm_stage2_reporter.py`

- stage1 결과를 incident 단위로 묶어 최종 보고서를 생성한다.
- 코드상 provider 미지정 시 기본 fallback은 OpenAI다.
- 운영 기준에서는 provider를 생략하지 않고 `--provider openai` 또는 `--provider anthropic`를 명시한다.
- Claude 실행 시 `--provider anthropic --model "$ANTHROPIC_MODEL"` 패턴을 쓴다.
- Anthropic provider도 사용 가능하며 정상적인 stage2 결과 생성 사례가 있다.
- Anthropic stage2에서 `stop_reason=max_tokens`가 나오면 출력 절단 가능성을 먼저 본다.
- stage2 실패 시 raw/error dump 파일이 생성될 수 있다.
- 정상 종료 여부는 `<base>_stage2_report.json`의 `meta.selected_model`, `meta.response_id`, `meta.stop_reason`에서 확인한다.
- 실패 진단은 `<base>_stage2_report_error.json`과 `<base>_stage2_report_raw_error.json`의 `stop_reason`, `parse_error`, `raw_response.usage.output_tokens`를 함께 본다.
- stage2는 최종 판정기가 아니라 incident 요약 보고서 생성기다. 최종 판단 전 raw evidence 대조가 권장된다.
- `KNOWN_ASSET_IPS`는 선택 문맥 정보다. 현재 운영 기준의 필수 입력은 아니다.

출력:

- `<base>_stage2_report_input.json`
- `<base>_stage2_report.json`
- `<base>_stage2_report.md`
- `<base>_stage2_report_error.json`
- `<base>_stage2_report_raw_error.json`

### 2.6 `run_analysis_pipeline.py`

- prepare -> stage1 -> stage2를 자동 실행한다.
- `--export-input`, `--llm-input`, `--stage1-results`에서 시작할 수 있다.
- 코드 기본 `processed-dir`은 `<work-dir>/data/processed`이며, 현재 운영 기준과 맞는다.
- 운영 기준에서는 `config/llm.env`를 한 번만 적용하고 provider는 `--llm-provider openai` 또는 `--llm-provider anthropic`로 명시한다.
- Claude 통합 실행은 `--llm-provider anthropic --stage1-model "$ANTHROPIC_MODEL" --stage2-model "$ANTHROPIC_MODEL"` 패턴을 쓴다.

## 3. 현재 분석 기준

- 기본 입력: `security`
- 보조 입력: `error`
- 운영 확인: `access`

보수 해석 기준:

1. `resp_content_type`
2. `response_body_bytes`
3. `raw_request_target`
4. `path_normalized_from_raw_request`
5. `likely_html_fallback_response`

`resp_html_*` fingerprint 계열은 현재 보류 또는 선택 항목이다.
