## 1. 문서 목적

이 문서는 Apache 기반 웹 로그 수집 환경에서 **어떤 로그를 어떤 형식으로 남기고, 그것을 어떤 DB 구조에 저장할 것인지**를 표준으로 정의하는 문서다.

이번 프로젝트의 목적은 단순 웹서비스 운영이 아니라, 정상 요청과 공격성 요청을 함께 수집하여 **탐지·분류·상관분석이 가능한 로그 데이터셋**을 만드는 데 있다.

따라서 본 문서는 다음 기준을 확정한다.

- 로그 수집 기준 지점
- 로그 종류와 역할 분리 기준
- 요청 식별자 정책
- 로그 필드 이름과 DB 컬럼명 기준
- MariaDB 데이터베이스 및 테이블 구조
- 주요 인덱스 기준
- 원문 보존 및 민감정보 최소화 원칙
- 시간 처리 기준

  - access log
  - security log
  - error log

즉, 본 문서는 **설치 문서가 아니라 표준 정의 문서**다.

## 3. 전체 구조 표준

현재 프로젝트의 로그 흐름 표준은 다음과 같다.

```text
클라이언트 / 공격 도구 / 실험 스크립트
                ↓
             Apache
                ↓
   ┌──────┼──────┐
   ↓            ↓            ↓
access log   security log   error log
   ↓            ↓            ↓
   └────── log shipper ──────┘
                ↓
             MariaDB
                ↓
  apache_access_logs
  apache_security_logs
  apache_error_logs
```

이 구조의 핵심 원칙은 다음과 같다.

1. **로그 수집의 중심 지점은 Apache로 통일한다.**
2. **운영 확인용 로그와 분석용 로그를 분리한다.**
3. **DB는 단일 테이블이 아니라 3개 테이블 구조로 통일한다.**
4. **원문 로그와 정규화 필드를 함께 보존한다.**
5. **요청 식별자 기반 상관분석이 가능하도록 설계한다.**

## 4. 왜 3종 로그 구조를 사용하는가

기본 access 로그 하나만 저장하는 방식은 단순하고 시연이 쉬운 장점이 있다.
하지만 공격 탐지와 후속 분류를 생각하면 정보가 부족하다.

부족해지기 쉬운 항목은 다음과 같다.

- 요청 식별자
- 에러 로그 연결 키
- 처리 시간
- 입출력 바이트 수
- TTFB
- 요청/응답 콘텐츠 타입
- 후속 공격 라벨 필드

따라서 이번 프로젝트는 로그를 다음처럼 역할별로 나눠 수집한다.

### 4.1 access log

기본 운영 확인용 로그다.

주요 목적:

- 서비스가 정상적으로 요청을 받고 있는지 확인
- URI, 상태코드, IP 단위 기본 조회
- 사람이 읽기 쉬운 수준의 요청 흐름 확인
- 보안 로그와 비교할 기준선 제공

운영 기준 메모:

- 현재 표준 access 로그는 **사람이 읽기 쉬운 기본 로그**를 우선한다.
- 현재 표준 access 로그에는 `request_id`를 기본 포함하지 않는다.
- 따라서 access 로그는 **LLM 탐지·분류의 주 입력**이 아니라 운영 확인과 기준선 확인용으로 사용한다.

### 4.2 security log

탐지·분류용 핵심 로그다.

주요 목적:

- 요청 단위 특징량 저장
- SQL Injection, XSS, 스캐닝, 경로 탐색 등 분석에 필요한 메타데이터 확보
- request_id 기반 상관분석
- 향후 attack_label, risk_score 저장

운영 기준 메모:

- 규칙 기반 후보 추출과 LLM 분석의 **기본 입력은 security log**로 둔다.
- access 로그와 중복 해석이 필요할 때도, 대표본은 security 로그를 우선한다.

### 4.3 error log

장애·경고·에러 분석용 보조 로그다.

주요 목적:

- 프록시 오류 추적
- 서버 오류와 특정 요청의 연결
- access/security 로그와의 상관분석
- 요청 실패 원인 확인

운영 기준 메모:

- error 로그는 평소에 비어 있을 수 있다.
- 다만 5xx, `request_id`, `error_link_id`가 연계되는 경우에는 **원인 확인용 보조 근거**로 매우 중요하다.

### 4.4 분석 계층에서의 우선순위

현재 표준에서 로그 3종의 분석 우선순위는 다음과 같이 정리한다.

1. **security log**: 탐지·분류의 기본 입력
2. **error log**: 5xx 또는 예외 상황의 원인 확인용 보조 입력
3. **access log**: 운영 확인과 기준선 비교용 보조 입력

즉, 3종 로그를 모두 저장하되, **보안 분석 파이프라인의 기본 입력은 security log**로 두는 것이 표준이다.

## 5. 로그 파일명 표준

Apache 로그 파일명은 다음으로 통일한다.

- `/var/log/apache2/app_access.log`
- `/var/log/apache2/app_security.log`
- `/var/log/apache2/app_error.log`

이 파일명 표준을 사용하는 이유는 다음과 같다.

1. 특정 애플리케이션 이름에 종속되지 않는다.
2. Juice Shop 외 다른 실험 대상에도 그대로 재사용할 수 있다.
3. shipper 기본 설정과 일치시키기 쉽다.
4. 문서와 코드의 기준을 단일화할 수 있다.

즉, 로그 파일명은 `juice_shop_*` 계열보다 **`app_*` 계열을 표준으로 사용**한다.

## 6. 요청 식별자 정책

요청 간 상관분석을 위해 다음 식별자 정책을 사용한다.

### 6.1 주 요청 식별자

- `request_id = %{UNIQUE_ID}e`

이 값은 모든 요청에 안정적으로 부여되므로,
분석 파이프라인에서 **주 식별자**로 사용한다.

### 6.2 에러 연계 보조 키

- `error_link_id = %L`

이 값은 Apache error log와 연결할 때 유용하다.
다만 모든 요청에 항상 존재하지 않을 수 있으므로,
주 식별자가 아니라 **보조 키**로 사용한다.

### 6.3 로그 간 연결 기준

- security ↔ error: 기본적으로 `request_id`, `error_link_id` 기준
- access ↔ security: 현재 표준에서는 **직접 연결을 기본 가정하지 않는다**
- access ↔ error: 필요 시 시각, method, uri, status_code, `error_link_id` 등을 보조적으로 사용

중요 메모:

- 현재 표준 access 로그에는 `request_id`를 기본 포함하지 않는다.
- 따라서 access ↔ security 직접 매칭은 보조적·제한적으로만 수행한다.
- 향후 access 로그 포맷을 확장해 `request_id`를 넣는 경우에는 직접 상관분석을 강화할 수 있다.

즉, 상관분석의 중심은 `request_id`이며, 이는 기본적으로 **security/error 축에서 가장 안정적으로 활용**한다.

## 7. 데이터베이스 표준

### 7.1 DB 이름

데이터베이스 이름은 다음으로 통일한다.

```sql
web_logs
```

이 이름을 쓰는 이유는 다음과 같다.

1. Juice Shop, OpenCart 등 특정 앱 이름에 종속되지 않는다.
2. 웹 로그 중앙 저장소라는 의미가 명확하다.
3. 향후 실험 대상이 바뀌어도 재사용 가능하다.

### 7.2 문자셋

문자셋은 `utf8mb4` 계열을 사용한다.

이유는 다음과 같다.

- 다양한 User-Agent와 헤더 문자열 저장
- 다국어 URI 및 파라미터 저장
- 확장성 확보

## 10. 로그 키 이름과 DB 컬럼명 기준

이번 표준에서는 security 로그의 key 이름도
가능한 한 DB 컬럼명과 동일한 풀네임을 사용한다.

예를 들면 다음과 같다.

- `request_id`
- `error_link_id`
- `src_ip`
- `peer_ip`
- `raw_request`
- `query_string`
- `status_code`
- `duration_us`
- `ttfb_us`
- `req_content_type`
- `resp_content_type`

이 방식을 택하는 이유는 다음과 같다.

1. 파서와 DB 매핑이 단순해진다.
2. 문서와 코드가 같은 이름을 사용하게 된다.
3. 축약 키(`reqid`, `errid`, `qs`, `ua`)보다 가독성이 높다.
4. export 및 후속 분석 코드 작성이 쉬워진다.

즉, **로그 키 이름과 DB 컬럼명을 최대한 일치시키는 것**을 표준으로 삼는다.

## 11. 정규화 저장과 원문 보존 원칙

모든 로그 테이블에는 `raw_log` 컬럼을 둔다.

그 이유는 다음과 같다.

1. 파싱 실패 시 원문을 다시 확인할 수 있다.
2. 정규식 수정 후 재처리가 가능하다.
3. 사람이 실제 로그 원문을 바로 검토할 수 있다.
4. 정규화 필드가 잘못 저장된 경우 비교 검증이 가능하다.

즉, 이번 구조는 **정규화 필드 + 원문 보존**을 함께 택한다.

## 12. 시간 처리 표준

시간 처리 기준은 다음과 같이 정리한다.

### 12.1 DB 저장 기준

- `log_time`, `created_at` 등 DB 저장 시각은 **UTC 기준 저장**을 원칙으로 한다.

### 12.2 사용자 조회 기준

- 사용자가 조회 범위를 지정할 때는 **KST(Asia/Seoul)** 기준 입력을 허용한다.
- 조회 시에는 KST 범위를 UTC로 변환하여 DB를 조회한다.

### 12.3 JSON export 기준

- 출력 JSON의 시간 필드는 **KST ISO-8601(+09:00)** 형식으로 변환하여 제공한다.

이 원칙을 두는 이유는 다음과 같다.

1. 저장 기준을 단일화하면 혼동이 줄어든다.
2. 사용자는 현지 시간대 기준으로 조회하는 편이 편하다.
3. 보고서와 분석 결과를 KST로 맞추기 쉽다.

## 13. 민감정보 최소 저장 원칙

보안 분석용 로그라고 하더라도 모든 값을 상시 저장하는 것은 바람직하지 않다.

기본 원칙은 다음과 같다.

### 13.1 기본 저장 권장 대상

- 요청 라인
- URI
- query string
- 상태코드
- 처리 시간
- 바이트 수
- host
- referer
- user_agent
- content-type 계열

### 13.2 상시 저장 비권장 대상

- Authorization 헤더 전체
- 전체 Cookie 값
- 요청 본문 원문
- 응답 본문 전체
- 민감정보가 포함될 수 있는 인증 토큰 값

필요 시 이들 값은 별도 정책, 마스킹, 제한적 활성화 방식으로 다루어야 한다.

---

## 14. 운영상 주의점

### 14.1 로그는 빠르게 증가한다

특히 Juice Shop 실험 환경에서는 `/socket.io/` 같은 반복 요청이 많아질 수 있다.
따라서 DB 구조만 정의할 것이 아니라,
로그 로테이션과 보관 정책을 함께 운영해야 한다.

### 14.2 error 로그는 비어 있을 수 있다

정상 운영 중에는 `apache_error_logs`가 비어 있을 수 있다.
이는 장애가 없다는 뜻일 수 있으므로,
에러 로그가 없다고 해서 비정상으로 해석할 필요는 없다.

### 14.3 security 로그는 분석용 핵심 테이블이다

기본 access 로그만으로는 공격 징후 분류가 부족할 수 있으므로,
후속 탐지와 분석은 `apache_security_logs`를 중심으로 설계하는 것이 적절하다.

실무 기준으로는 다음처럼 해석한다.

- routine 분석: `apache_security_logs` 중심
- 5xx 또는 예외 조사: `apache_security_logs` + `apache_error_logs`
- 운영 확인 또는 발표용 흐름 확인: 필요 시 `apache_access_logs` 보조 활용

## 15. 결론

이번 프로젝트의 로그 저장 표준은 다음과 같이 정리할 수 있다.

1. Apache를 로그 수집의 중심 지점으로 사용한다.
2. access / security / error 로그의 3종 분리 구조를 사용한다.
3. 로그 파일명은 `app_access.log`, `app_security.log`, `app_error.log`로 통일한다.
4. 요청 상관분석의 주 키는 `request_id`, 보조 키는 `error_link_id`로 사용한다.
5. DB 이름은 `web_logs`로 통일한다.
6. 테이블은 `apache_access_logs`, `apache_security_logs`, `apache_error_logs` 3개로 고정한다.
7. security 로그 key 이름은 DB 컬럼명과 최대한 동일하게 맞춘다.
8. 탐지·분류 파이프라인의 기본 입력은 security 로그로 둔다.
9. 정규화 컬럼과 `raw_log`를 함께 저장한다.
10. DB 저장 시각은 UTC, 사용자 조회와 출력은 KST 기준으로 처리한다.
11. 민감정보는 기본값으로 최소 저장한다.

즉, 이 구조는 단순 로그 보관을 위한 구조가 아니라,
**웹 공격 탐지와 로그 기반 분류 실험을 위한 분석 지향 표준 구조**라고 볼 수 있다.
