## 1. 목적

Apache 로그와 MariaDB 저장 구조를 현재 코드 기준으로 정리한다.

## 2. 전체 구조

```text
Apache
  ↓
access / security / error log
  ↓
apache_log_shipper.py
  ↓
MariaDB(web_logs)
  ↓
apache_access_logs
apache_security_logs
apache_error_logs
```

## 3. 로그 역할

### 3.1 access log

- 운영 확인용
- 기본 요청 흐름 확인

현재 분석 파이프라인의 주 입력은 아니다.

### 3.2 security log

- 분석 기본 입력
- 요청 단위 메타데이터 저장
- `request_id` 기반 상관분석 중심

### 3.3 error log

- 예외/장애 확인용
- `request_id`, `error_link_id` 연계 보조

## 4. 로그 파일명 표준

- `/var/log/apache2/app_access.log`
- `/var/log/apache2/app_security.log`
- `/var/log/apache2/app_error.log`

## 5. 식별자 기준

- 주 식별자: `request_id`
- 보조 식별자: `error_link_id`

현재 문서 기준으로 access 로그는 `request_id` 직접 매칭의 중심이 아니다. security/error 축이 상관분석의 기준이다.

## 6. DB 기본

- DB 이름: `web_logs`
- 문자셋: `utf8mb4`

최종 테이블:

- `apache_access_logs`
- `apache_security_logs`
- `apache_error_logs`

## 8. `resp_html_*` 관련 정리

`apache_security_logs` 에는 아래 선택 컬럼이 있을 수 있다.

- `resp_html_norm_fingerprint`
- `resp_html_fingerprint_version`
- `resp_html_baseline_name`
- `resp_html_baseline_match`
- `resp_html_baseline_confidence`
- `resp_html_features_json`

현재 기준:

- 컬럼/적재 경로는 있을 수 있다.
- 하지만 값 생성 로직은 보류 상태다.
- 현재 분석 파이프라인의 핵심 기준으로 사용하지 않는다.

현재 유지해야 하는 실제 사용 필드:

- `resp_content_type`
- `response_body_bytes`
- `raw_request_target` 관련 값은 downstream prepare 결과에서 사용
- `path_normalized_from_raw_request` 관련 값은 downstream prepare 결과에서 사용
- `likely_html_fallback_response` 관련 값은 downstream prepare 결과에서 사용

## 9. 시간 처리

- DB 저장 기준: UTC
- 사용자 조회 기준: KST
- export 출력 기준: KST ISO-8601

이 기준은 `export_db_logs_cli.py` 와 문서 전체에서 동일하게 유지한다.

## 10. 운영 메모

- 분석 기본 입력은 `apache_security_logs`
- 예외 조사 시 `apache_error_logs` 보조 사용
- 운영 확인 시 `apache_access_logs` 보조 사용
- 모든 테이블은 `raw_log` 를 유지해 원문 확인이 가능해야 한다
