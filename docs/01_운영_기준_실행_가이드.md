# 01_운영_기준_실행_가이드

- 문서 상태: 운영 기준 문서
- 목적: 현재 실제 운영/실행 기준의 single source of truth

이 문서가 실제 운영 명령, 경로, OpenAI/Anthropic 실행 차이, 수동 절차의 기준 문서다.  
다른 문서와 표현이 다르면 실제 운영은 이 문서를 우선한다.

## 1. 목적

- 실제 운영자가 그대로 따라 할 수 있는 절차를 한 문서에 모은다.
- 흩어진 명령과 경로 기준을 현재 운영 기준으로 통일한다.
- OpenAI와 Anthropic 실행 차이를 한곳에서 확인하게 한다.
- provider 오염을 막기 위한 단일 env 운영 원칙을 명확히 한다.

## 2. 현재 실제 디렉터리 구조

기준 작업 루트는 `/opt/web_log_analysis`다.

```text
/opt/web_log_analysis/
├── config/
│   └── llm.env
├── data/
│   ├── raw/
│   └── processed/
├── reports/
├── logs/
├── src/
└── .venv/
```

- raw 입력: `/opt/web_log_analysis/data/raw`
- 전처리 결과: `/opt/web_log_analysis/data/processed`
- 보고서 결과: `/opt/web_log_analysis/reports`
- 실행 로그: `/opt/web_log_analysis/logs`

주의:

- `export_db_logs_cli.py`는 기본적으로 `data/raw/`에 출력하므로 현재 운영 기준과 맞는다.
- `prepare_llm_input.py`, `llm_stage1_classifier.py`, `llm_stage2_reporter.py`는 운영 기준 경로를 맞추기 위해 `--out-dir`를 명시한다.
- `run_analysis_pipeline.py` 기본값은 `<work-dir>/data/processed`와 `<work-dir>/reports`다. 현재 운영 기준과 맞으므로 표준 pipeline 실행에서는 별도 디렉터리 옵션을 생략한다.

## 3. 서버 역할 구분

- 웹서버: Apache 로그 생성, shipper 실행
- DB 서버: `web_logs` 저장
- LLM 서버: export, prepare, stage1, stage2, pipeline 실행

운영 기준:

- export는 DB 서버가 아니라 LLM 서버에서 수행한다.
- routine 분석의 기본 입력은 `security` export다.

## 4. env 파일 사용 정책

기준 파일은 `/opt/web_log_analysis/config/llm.env` 하나다.

예시:

```dotenv
OPENAI_API_KEY=YOUR_OPENAI_API_KEY
OPENAI_BASE_URL=https://api.openai.com/v1

ANTHROPIC_API_KEY=YOUR_ANTHROPIC_API_KEY
ANTHROPIC_BASE_URL=https://api.anthropic.com/v1
ANTHROPIC_MODEL=claude-sonnet-4-6
ANTHROPIC_MAX_TOKENS=8192

LOG_DB_HOST=DB_SERVER_IP
LOG_DB_PORT=3306
LOG_DB_NAME=web_logs
LOG_DB_USER=log_writer
LOG_DB_PASSWORD=YOUR_DB_PASSWORD
KNOWN_ASSET_IPS=OPTIONAL_ASSET_IP_LIST
```

운영 원칙:

- `config/llm.env` 하나만 사용한다.
- OpenAI / Anthropic 관련 값을 같은 파일 안에 함께 둔다.
- `LLM_PROVIDER`는 이 파일의 기본값으로 두지 않는 것을 권장한다.
- provider는 실행 인자에서 명시한다.
- 같은 셸에서 여러 env 파일을 번갈아 `source`하는 방식은 사용하지 않는다.

provider 지정 원칙:

- OpenAI pipeline: `--llm-provider openai`
- Anthropic pipeline: `--llm-provider anthropic`
- stage1/stage2 단독 실행: `--provider openai` 또는 `--provider anthropic`

적용:

```bash
cd /opt/web_log_analysis
source .venv/bin/activate
set -a
source ./config/llm.env
set +a
```

## 5. provider 선택 기준

- OpenAI 실행은 반드시 `--llm-provider openai` 또는 `--provider openai`를 준다.
- Anthropic 실행은 반드시 `--llm-provider anthropic` 또는 `--provider anthropic`를 준다.
- 코드상 provider 기본 fallback은 여전히 `openai`지만, 운영 기준에서는 기본값에 의존하지 않는다.
- 같은 셸에서 이전 provider 환경변수가 남아도 실행 인자를 명시하면 그 값이 우선한다.

### 6.4 stage1

OpenAI:

```bash
python ./src/llm_stage1_classifier.py \
  --provider openai \
  --input ./data/processed/security_2026-04-23_kst_llm_input.json \
  --out-dir ./data/processed \
  --mode routine \
  --pretty
```

Anthropic:

```bash
python ./src/llm_stage1_classifier.py \
  --provider anthropic \
  --model "$ANTHROPIC_MODEL" \
  --input ./data/processed/security_2026-04-23_kst_llm_input.json \
  --out-dir ./data/processed \
  --mode routine \
  --pretty
```

### 6.5 stage2

OpenAI:

```bash
python ./src/llm_stage2_reporter.py \
  --provider openai \
  --stage1-results ./data/processed/security_2026-04-23_kst_stage1_results.json \
  --llm-input ./data/processed/security_2026-04-23_kst_llm_input.json \
  --out-dir ./reports \
  --mode routine \
  --pretty
```

Anthropic:

```bash
python ./src/llm_stage2_reporter.py \
  --provider anthropic \
  --model "$ANTHROPIC_MODEL" \
  --stage1-results ./data/processed/security_2026-04-23_kst_stage1_results.json \
  --llm-input ./data/processed/security_2026-04-23_kst_llm_input.json \
  --out-dir ./reports \
  --mode routine \
  --pretty
```

기준:

- 현재 운영 기준 명령 세트에서는 `--known-asset-ips`를 기본 명령에서 넣지 않는다.
- 필요하면 선택적으로만 추가한다.
- Anthropic provider도 정상적인 stage2 결과 생성이 가능하다.
- 정상 종료 확인은 `<base>_stage2_report.json`의 `meta.provider`, `meta.selected_model`, `meta.response_id`, `meta.stop_reason`에서 본다.
- 실패 진단은 `<base>_stage2_report_error.json`의 `stop_reason`, `parse_error`, `raw_dump_path`를 먼저 확인하고, 이어서 `<base>_stage2_report_raw_error.json`의 `raw_response.usage.input_tokens`, `raw_response.usage.output_tokens`를 본다.
- `stop_reason=end_turn`이면 정상 종료로 본다.
- Anthropic stage2에서 `stop_reason=max_tokens`가 확인되면 출력이 잘린 것이므로 `ANTHROPIC_MAX_TOKENS` 조정 또는 더 큰 출력 한도를 지원하는 모델 검토가 필요하다.
- `stop_reason=max_tokens` 또는 `output_tokens`가 상한치 근처면 출력 절단 가능성을 먼저 의심한다.
- stage2 보고서는 raw evidence 원문 뷰어가 아니라 incident 요약 보고서다.
- raw evidence는 prepare와 stage1까지는 더 직접적으로 활용되지만, stage2에서는 사건형 요약 입력으로 축약된다.
- suspicious incident는 stage2만 보고 끝내지 않고 `request_id` 기준 raw evidence 대조를 표준 절차로 삼는다.

### 6.5.1 suspicious incident 수동 확인 절차

현재 구조:

- prepare는 규칙 기반으로 후보를 추린다.
- stage1은 `raw_request`, `raw_request_target`, `user_agent`, `raw_log_excerpt`에 준하는 단서를 포함한 raw evidence를 더 직접적으로 보고 1차 분류한다.
- stage2는 이를 incident 단위 사건형 요약 보고서로 정리한다.

운영 절차:

1. `stage2_report.md` 또는 `stage2_report.json`에서 suspicious incident를 확인한다.
2. 해당 incident의 `request_id`를 확인한다.
3. DB의 `apache_security_logs.raw_log` 원문을 `request_id` 기준으로 조회한다.
4. 같은 시간대의 `apache_error_logs`와 앱 로그를 함께 대조한다.
5. payload 정황, 도구 사용 정황, 서버 반응, 성공 정황을 분리해 최종 판단한다.

판단 원칙:

- payload 존재만으로 exploit 성공을 단정하지 않는다.
- `200` 응답만으로 파일 노출, 인증 우회, 실제 침해 성공을 단정하지 않는다.
- stage2 설명은 운영자용 요약이고, 최종 판단은 raw evidence 대조 후 내린다.

### 6.6 pipeline

공통 환경 적용:

```bash
cd /opt/web_log_analysis
source .venv/bin/activate
set -a
source ./config/llm.env
set +a
```

OpenAI:

```bash
python ./src/run_analysis_pipeline.py \
  --llm-provider openai \
  --export-input ./data/raw/security_2026-04-24_13-20-00_to_2026-04-24_13-37-00_kst.json \
  --work-dir /opt/web_log_analysis \
  --base-name op-security_2026-04-24_13-20-00_to_2026-04-24_13-37-00_kst \
  --mode routine \
  --pretty
```

Anthropic:

```bash
python ./src/run_analysis_pipeline.py \
  --llm-provider anthropic \
  --stage1-model "$ANTHROPIC_MODEL" \
  --stage2-model "$ANTHROPIC_MODEL" \
  --export-input ./data/raw/security_2026-04-24_13-20-00_to_2026-04-24_13-37-00_kst.json \
  --work-dir /opt/web_log_analysis \
  --base-name cl-security_2026-04-24_13-20-00_to_2026-04-24_13-37-00_kst \
  --mode routine \
  --pretty
```

기본 경로와 다르게 저장해야 할 때만 `--processed-dir` 또는 `--reports-dir`를 선택적으로 지정한다.

비교 실험 원칙:

- 비교 실험에서는 env를 여러 개 `source`하지 않는다.
- OpenAI와 Anthropic 비교는 같은 `config/llm.env`를 한 번만 적용한 뒤, 각 실행마다 provider를 CLI에서 다르게 명시한다.
- `op-` prefix 산출물은 `meta.provider=openai`여야 한다.
- `cl-` prefix 산출물은 `meta.provider=anthropic`여야 한다.

## 10. Troubleshooting

### 10.1 op- 산출물인데 provider=anthropic 으로 찍히는 경우

증상:

- 파일명은 `op-...`인데 `stage1_results.json` 또는 `stage2_report.json`의 `meta.provider=anthropic`
- `meta.selected_model`도 `claude-*` 또는 Anthropic 모델명으로 기록됨

주요 원인:

- 같은 셸에서 이전에 Anthropic용 env를 먼저 `source`했고, 이후 OpenAI용 env를 다시 읽어도 `LLM_PROVIDER=anthropic` 같은 값이 남아 있음
- 실행 명령에서 provider를 생략해 환경변수 fallback이 적용됨

대응:

- `config/llm.env` 하나만 사용한다.
- 여러 env 파일을 번갈아 `source`하지 않는다.
- OpenAI 실행에는 반드시 `--llm-provider openai` 또는 `--provider openai`를 준다.
- 실행 후 `stage1_results.json` 또는 `stage2_report.json`의 `meta.provider`, `meta.selected_model`을 확인한다.

## 11. 자주 헷갈리는 부분

- export는 LLM 서버에서 실행한다.
- prepare와 stage1 결과는 `./data/processed` 기준이다. `./processed`가 아니다.
- stage2 결과는 `./reports` 기준이다.
- pipeline 기본값은 `./data/processed`와 `./reports` 기준이므로 운영 경로와 맞는다.
- 코드상 provider 기본값은 OpenAI지만, 운영 기준에서는 provider 생략을 허용하지 않는다.
- 비교 실험에서는 `config/llm.env`를 한 번만 적용하고 provider는 실행 인자 기준으로 고정한다.
- `KNOWN_ASSET_IPS`는 현재 운영 기준의 필수 항목이 아니다.
- stage2 정상 종료 여부는 `<base>_stage2_report.json`의 `meta.stop_reason`에서 확인한다.
- stage2 실패 시 `<base>_stage2_report_error.json`과 `<base>_stage2_report_raw_error.json`을 함께 보고 `stop_reason`, `parse_error`, `raw_response.usage.output_tokens`를 확인한다.
- stage2 보고서는 최종 증거가 아니라 incident 요약 보고서다.
- raw evidence는 prepare와 stage1까지는 더 직접적으로 활용되지만, stage2에서는 사건형 요약 입력으로 축약된다.
- suspicious/high incident는 `request_id` 기준으로 `apache_security_logs.raw_log`, 같은 시간대 error log, app log를 함께 대조한 뒤 최종 판단한다.
- Anthropic stage2에서 `stop_reason=max_tokens`가 보이면 출력 절단 가능성을 먼저 확인한다.

## 12. 짧은 순서 요약

요청 발생  
-> Apache 로그 생성  
-> shipper DB 적재  
-> LLM 서버 export  
-> prepare  
-> stage1  
-> stage2 또는 pipeline  
-> 보고서 회수
