# 06_비교_실험_요청_세트_표준

- 문서 상태: 초안
- 버전: v1.0
- 작성일: 2026-04-09
- 목적: Juice Shop 과 OpenCart 를 같은 기준으로 비교하기 위한 표준 실험 요청 세트를 정의한다.

---

## 1. 문서 목적

이 문서는 앞으로 Juice Shop 과 OpenCart 에 대해 반복적으로 사용할 **표준 실험 요청 세트**를 정의한 문서다.

핵심 목적은 다음과 같다.

1. 같은 성격의 정상 요청을 두 앱에 각각 보내고 비교한다.
2. 같은 성격의 탐색성 요청을 두 앱에 각각 보내고 비교한다.
3. 같은 성격의 공격성 요청을 두 앱에 각각 보내고 비교한다.
4. 보고서 비교 시 “어떤 요청을 보냈는지 기억이 안 나는 문제”를 줄인다.
5. 앞으로 같은 기준으로 재실험할 수 있게 한다.

즉, 이 문서는 단발성 메모가 아니라 **반복 가능한 실험 기준 문서**다.

---

## 2. 기본 원칙

### 2.1 요청 세트는 공통 기준으로 유지한다

Juice Shop 과 OpenCart 는 웹 구조와 경로가 다르다.  
따라서 완전히 동일한 URL 을 그대로 재사용하는 것이 아니라, **같은 실험 의도**를 유지한 채 앱별 경로만 매핑한다.

예:
- 정상 검색 요청
- 관리자 경로 조회
- 저신호 fuzzing
- path traversal 시도
- 입력 파라미터 기반 XSS/SQLi 시도

즉, **의도는 같게, 경로는 앱별로 다르게** 가져간다.

### 2.2 같은 실험 세트는 앱마다 각각 실행한다

비교 실험의 목적은 “같은 요청을 두 앱이 어떻게 다르게 보이게 만드는가”를 보는 것이다.  
따라서 한 번만 실행해서는 안 되고, **Juice Shop 1회 + OpenCart 1회**가 필요하다.

### 2.3 동일 DB 적재 시 export 는 시간 구간으로 분리한다

Juice Shop 과 OpenCart 로그가 같은 DB 의 같은 테이블에 적재되면,  
비교 실험 시 export 는 반드시 **앱별 실험 시간 구간**으로 분리한다.

권장 예시:
- Juice Shop 세션: 14:00 ~ 14:10
- OpenCart 세션: 14:20 ~ 14:30

즉, 하루 전체 export 를 그대로 비교 기준으로 쓰지 않는다.

### 2.4 내부 테스트 IP 는 known asset 문맥으로 본다

테스트를 내부 장비에서 수행하는 경우 source IP 는 내부 자산으로 보일 수 있다.  
따라서 보고서 해석 시:
- 내부 테스트
- 자체 호출
- 운영 점검
가능성을 함께 고려한다.

### 2.5 범위 밖의 과도한 공격 자동화는 하지 않는다

이 문서의 목적은 비교 실험 기준을 고정하는 것이지, 대규모 스캔이나 무차별 자동화 공격을 확대하는 것이 아니다.  
요청 수는 최소화하고, 실험 의도가 분명한 범위에서 반복 가능하게 운영한다.

## 3. 실험 세트 구성 개요

표준 실험 요청 세트는 아래 4개 묶음으로 구성한다.

1. **A세트: 정상 요청 기준선**
2. **B세트: 후보 밖 탐색성 요청**
3. **C세트: 통제된 공격성 요청**
4. **D세트: 관리/운영 보조 요청**

운영 권장 순서:
- A → B → C → D

이 순서를 유지하면, stage2 보고서에서 정상 요청 / 저신호 탐색 / 사건 후보가 어떻게 나뉘는지 보기 쉽다.

## 4. 앱별 대표 경로 매핑

아래 표는 “실험 의도” 기준의 대표 경로 매핑 예시다.  
실제 앱 버전이나 설치 상태에 따라 경로는 조정할 수 있다.

| 실험 의도 | Juice Shop 예시 | OpenCart 예시 |
|---|---|---|
| 메인 페이지 | `/` | `/` |
| 검색 | `/rest/products/search?q=test` | `/index.php?route=product/search&search=test` |
| 관리자/관리자 유사 경로 | `/administration` 또는 `/admin` 유사 경로 | `/admin/` |
| 존재하지 않는 관리자형 경로 | `/admin_99` | `/admin_99` |
| robots/sitemap | `/robots.txt`, `/sitemap.xml` | `/robots.txt`, `/sitemap.xml` |
| 상품/카탈로그 조회 | 제품 API 또는 상품 페이지 | 상품 상세/카테고리 페이지 |
| path traversal 시도 대상 | 정적 파일 경로 하위 | 이미지/카탈로그 경로 하위 |
| 파라미터 기반 XSS/SQLi 시험 | 검색/조회 파라미터 | 검색/조회 파라미터 |

주의:
- 경로는 앱 구조에 따라 달라질 수 있으므로, 실험 전 1회 수동 확인 후 고정한다.
- 문서의 핵심은 **정확한 URL 문자열 하나**가 아니라 **동일 실험 의도 유지**다.

## 5. A세트: 정상 요청 기준선

목적:
- 정상 탐색/조회가 얼마나 benign 으로 잘 분리되는지 보기
- 앱별 정상 검색 요청 분포 차이 확인
- 후보 밖 normal_search 기준선 확보

### A-1. 메인 페이지 접속
- 1회 요청
- 목적: 기본 landing page 응답 확인

예시:
- GET `/`

### A-2. 검색 요청
- 키워드 3종
- 목적: benign_normal_search 분포 확인

권장 키워드:
- `apple`
- `phone`
- `bag`

예시:
- Juice Shop: `/rest/products/search?q=apple`
- OpenCart: `/index.php?route=product/search&search=apple`

### A-3. 카테고리/상품 조회
- 카테고리 1건, 상품 상세 1건
- 목적: 정상 콘텐츠 조회 패턴 확인

### A-4. 로그인/계정 화면 접근
- 실제 로그인 시도는 선택
- 목적: 인증 관련 정상 페이지 요청 확인

### A-5. 장바구니/관리 화면 접근
- 실제 구매 흐름 필요 없음
- 목적: 쇼핑몰형 일반 동작 분포 확인

권장 요청 수:
- 총 8 ~ 12건

---

## 6. B세트: 후보 밖 탐색성 요청

목적:
- low_signal_fuzzing / low_signal_dir_probe 류가 어떻게 보이는지 확인
- 정상 요청과 저신호 탐색이 어떻게 분리되는지 확인
- stage2 의 “후보 밖 탐색성 요청” 섹션 비교

### B-1. robots / sitemap
- GET `/robots.txt`
- GET `/sitemap.xml`

### B-2. 관리자 유사 경로
- GET `/admin`
- GET `/admin_99`

### B-3. 흔한 탐색성 경로
- GET `/.git`
- GET `/.env`
- GET `/phpinfo.php`
- GET `/login`
- GET `/backup.zip`

주의:
- 앱 구조와 무관하게 흔히 보이는 **저신호 탐색성 요청**만 사용한다.
- 너무 많은 사전/wordlist 확장은 하지 않는다.

권장 요청 수:
- 총 6 ~ 10건

---

## 7. C세트: 통제된 공격성 요청

목적:
- 같은 의도의 공격성 요청이 앱마다 어떻게 다르게 보이는지 비교
- stage1 / stage2 의 보수 해석 차이 비교
- OpenCart 와 Juice Shop 의 응답 특성 차이 확인

### C-1. path traversal 시도

의도:
- 정적 파일 또는 공개 경로 하위에서 `../` 계열 상승 시도
- 404 / 200 / text/html fallback / 정규화 여부 비교

표준 예시 1:
- `/public/images/../../etc/passwd`

표준 예시 2:
- `/public/images/%2e%2e/%2e%2e/etc/passwd`

표준 예시 3:
- `/public/images/..%2f..%2fetc/passwd`

운영 원칙:
- 앱별로 “정적 자산 경로처럼 보이는 위치” 하위에 붙여서 시도
- 결과가 200 이어도 즉시 성공으로 해석하지 않음
- `text/html`, body size, fallback 정황을 함께 봄

권장 요청 수:
- 2 ~ 3건

### C-2. 검색 파라미터 기반 XSS 시험

의도:
- 검색/조회 파라미터에 간단한 script/event payload 삽입
- 반사형/저장형 여부가 아니라, 입력 기반 공격성 흔적이 어떻게 보이는지 관찰

표준 예시:
- `<script>alert(1)</script>`
- `"><svg/onload=alert(1)>`
- `'><img src=x onerror=alert(1)>`

운영 원칙:
- 검색 파라미터 1개에만 넣음
- 실제 브라우저 실행 여부보다 로그/보고서 상의 흔적 비교가 목적
- 필요 이상으로 변형을 늘리지 않음

권장 요청 수:
- 2 ~ 3건

### C-3. 검색 파라미터 기반 SQLi 시험

의도:
- 검색/조회 파라미터에 전형적인 SQLi 흔적 문자열 입력
- HPP 유무, 입력 처리 차이, 500/200/검색 결과 변화 등을 관찰

표준 예시:
- `' OR '1'='1`
- `1' OR 1=1--`
- `test' UNION SELECT 1--`

운영 원칙:
- 검색 또는 조회 파라미터에만 사용
- 결과가 정상이어도 흔적이 남는지 확인
- 실제 DB 손상성 쿼리는 포함하지 않음

권장 요청 수:
- 2 ~ 3건

### C-4. HPP 결합 시험

의도:
- 동일 파라미터를 2번 넣었을 때 앱 처리 차이와 보고서 반영 여부 비교

표준 예시:
- `?q=apple&q=' OR '1'='1`
- `?search=phone&search=<script>alert(1)</script>`

운영 원칙:
- search 계열 파라미터에서만 수행
- HPP 문맥을 SQLi/XSS 설명에 함께 보이게 하는지 비교

권장 요청 수:
- 1 ~ 2건

## 8. D세트: 운영/보조 확인 요청

목적:
- 보고서 해석 시 보조 근거 확보
- 정상/비정상 요청 전후 흐름 확인

권장 항목:
- 메인 페이지 재접속
- 검색 1건 재실행
- 관리자 페이지 재접속
- 동일 IP / 동일 User-Agent 유지

권장 요청 수:
- 총 3 ~ 5건

## 9. 기본 실행 순서

권장 실행 순서는 아래와 같다.

### 앱별 1회 세션 실행 예시

1. A세트 실행 (정상 요청)

- Juice Shop:
  - 14:00:00 ~ 14:10:00
- OpenCart:
  - 14:20:00 ~ 14:30:00

중간에 5 ~ 10분 정도 띄우는 편이 좋다.

## 11. 비교 실험 시 꼭 볼 항목

두 앱 결과를 비교할 때는 아래를 우선 본다.

### 11.1 정상 요청 쪽
- `benign_normal_search` 비율
- 정상 조회가 low-signal 로 잘못 올라가는지 여부
- 검색 요청이 얼마나 안정적으로 benign 처리되는지

### 11.2 후보 밖 탐색성 요청 쪽
- `low_signal_fuzzing`
- `low_signal_dir_probe`
- 후보 밖 탐색성 요청 섹션의 해석 차이

### 11.3 사건 후보 쪽
- path traversal 시도의 severity 차이
- 200 + `text/html` 응답에 대한 보수 해석 차이
- HPP 문맥이 설명에 드러나는지
- XSS / SQLi 가 앱별로 어떻게 다르게 보이는지

### 11.4 운영 해석 쪽
- known asset IP 문맥이 자연스럽게 유지되는지
- 내부 테스트 / 운영 점검 / 자체 호출 가능성이 보고서에 반영되는지

## 13. 향후 운영 원칙

앞으로 실험은 아래 원칙을 기본으로 한다.

1. **표준 요청 세트는 고정한다.**
2. **앱별 경로 매핑만 조정한다.**
3. **동일 DB 적재 시 export 는 실험 시간 구간으로 분리한다.**
4. **실험 세션 기록 템플릿을 반드시 남긴다.**
5. **새로운 payload 를 추가할 때는 문서를 먼저 갱신한다.**

즉, 이 문서는 앞으로 실험 요청 세트의 기준 문서로 사용한다.

## 14. 최종 요약

이번 표준 요청 세트는 Juice Shop 과 OpenCart 의 웹 구조 차이를 인정하면서도,  
비교 실험에 필요한 **공통 실험 의도**를 유지하기 위해 만든 기준 문서다.

핵심은 다음과 같다.

- 요청은 앱마다 각각 실행한다.
- 정상 / 탐색 / 공격 / 보조 요청을 분리한다.
- 경로는 다를 수 있지만 실험 의도는 동일하게 유지한다.
- 같은 DB 를 쓰면 export 는 시간 구간으로 분리한다.
- 앞으로의 보고서 비교는 이 요청 세트를 기준으로 수행한다.

한 문장으로 요약하면:

> Juice Shop 과 OpenCart 의 구조는 다르지만, 실험 의도를 표준화한 요청 세트를 기준으로 각각 실행하고 시간 구간을 분리해 export 하면, 앞으로도 반복 가능한 비교 실험을 수행할 수 있다.
