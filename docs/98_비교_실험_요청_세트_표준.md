# 98_비교_실험_요청_세트_표준

- 작성 기준일: 2026-04-24
- 문서 역할: 모든 비교 실험의 공통 기준 문서
- 목적: 시간 구간 기반 비교 실험 절차, 명명 규칙, 산출물 보관 기준을 표준화한다.
- 적용 범위: provider 비교(OpenAI / Anthropic)와 웹서비스 비교(Juice Shop / OpenCart 등)에 공통 적용한다.
- 원칙: 모든 실험 요청과 분석 실행은 GUI가 아니라 CLI-only(`curl`, `python`, `scp`) 기준으로 수행한다.

---

## 1. 운영 원칙

앞으로 비교 실험은 아래 흐름을 기본으로 운영한다.

1. 특정 웹서비스에서 특정 요청/공격 세트를 짧은 시간 구간에 수행한다.
2. 그 시간 구간만 `export_db_logs_cli.py`로 export 한다.
3. 같은 raw log를 기준으로 provider 또는 웹서비스 차이를 비교한다.

핵심 의도는 다음과 같다.

- 한 실험 단위를 짧고 명확한 시간 구간으로 고정한다.
- 같은 raw log로 OpenAI / Anthropic 비교를 반복 가능하게 만든다.
- 같은 실험 틀을 Juice Shop, OpenCart 등 다른 웹서비스에도 재사용한다.
- 산출물 파일명과 회수 기준을 고정해 반복 실험 비용을 줄인다.

provider 운영 원칙:

- 비교 실험에서는 `config/llm.env` 하나만 적용한다.
- OpenAI / Anthropic 비교를 위해 env 파일을 여러 개 번갈아 `source`하지 않는다.
- provider는 매 실행마다 CLI에서 명시한다.
- 실행 후 산출물의 `meta.provider`, `meta.selected_model`을 확인해 기대한 provider로 실행됐는지 검증한다.

---

## 2. 실험 단위 정의

비교 실험의 기본 단위는 아래 필드를 가진다.

- `webservice`: 대상 웹서비스 이름
- `experiment_set`: 요청 세트 이름. 예: `A`, `B`, `C`
- `attack_category`: baseline, auth, sqli, xss, traversal, hpp, dir_probe 등
- `start_time`: 실험 시작 시각(KST)
- `end_time`: 실험 종료 시각(KST)
- `export_file`: 시간 구간 export 결과 JSON
- `provider`: `openai` 또는 `anthropic`
- `model`: provider별 실제 사용 모델명
- `known_asset_ips`: raw log 생성 환경 기준 자산 IP 목록

권장 기록 예시:

```text
webservice: juice-shop
experiment_set: A
attack_category: auth
start_time: 2026-04-24 13:20:00 KST
end_time: 2026-04-24 13:37:00 KST
export_file: data/raw/security_2026-04-24_13-20-00_to_2026-04-24_13-37-00_kst.json
provider: openai
model: gpt-5.4-mini
known_asset_ips: 127.0.0.1,192.168.35.145
```

## 3. 비교 유형

이 문서는 세 가지 비교 유형을 구분한다.

### 3.1 Full pipeline 비교

- 같은 raw log를 기준으로 `prepare -> stage1 -> stage2` 전체를 provider별로 각각 실행한다.
- provider/model 변경이 전체 결과에 미치는 영향을 본다.
- 가장 기본적인 비교 방식이다.

### 3.2 Stage1 고정 비교

- 같은 `*_llm_input.json`을 고정한다.
- provider별 stage1 verdict / severity / confidence 차이를 본다.
- prepare 차이를 배제하고 stage1 분류 차이만 보고 싶을 때 사용한다.

### 3.3 Stage2 고정 비교

- 같은 `*_stage1_results.json`을 고정한다.
- provider별 stage2 사건 우선순위, 서술 방식, 보고서 표현 차이를 본다.
- stage1 차이를 배제하고 stage2 보고 차이만 보고 싶을 때 사용한다.

---

## 4. 공통 비교 조건

같은 raw log 기반 비교에서는 아래 조건을 유지한다.

- 같은 `known_asset_ips`
- 같은 top-N 설정
- 같은 `mode`

여기서 top-N은 현재 파이프라인 기준으로 아래 항목을 뜻한다.

- stage1: `--stage1-candidate-limit`
- stage2: `--stage2-top-incidents`
- stage2: `--stage2-top-noise-groups`
- stage2: `--stage2-top-ips`

추가 원칙:

- 가능하면 같은 export JSON을 provider별로 공유한다.
- stage2 차이를 명확히 보려면 같은 `*_stage1_results.json` 고정 비교를 별도로 수행한다.
- `known_asset_ips`는 분석 서버가 아니라 raw log를 생성한 웹서비스/실험 환경 기준으로 맞춘다.

## 5. 시간 구간 기반 표준 절차

운영자는 아래 순서를 표준 절차로 사용한다.

### 5.1 실험 시작 시각 기록

- 실험 시작 직전에 KST 기준 `start_time`을 기록한다.
- 예: `2026-04-24 13:20:00`

### 5.2 CLI 요청 세트 수행

- `curl` 또는 필요 시 `python` CLI 스크립트로 요청 세트를 수행한다.
- 웹 GUI 클릭 기반 부가 트래픽은 비교 기준에서 제외한다.
- 요청별 `User-Agent`를 구분해 남기는 것을 권장한다.

### 5.3 실험 종료 시각 기록

- 마지막 요청 직후 KST 기준 `end_time`을 기록한다.
- 예: `2026-04-24 13:37:00`

### 5.4 시간 구간 export

- `src/export_db_logs_cli.py`로 `--start / --end` 시간 구간 export를 수행한다.
- 앞으로 비교 실험 예시는 날짜 단위 export보다 시간 구간 export를 기본 예시로 삼는다.

### 5.5 provider별 pipeline 실행

- 같은 export JSON을 기준으로 provider별로 서로 다른 `--base-name`을 사용해 실행한다.
- OpenAI / Anthropic 비교 시 산출물 파일명이 섞이지 않도록 base-name을 반드시 분리한다.
- provider는 env가 아니라 CLI에서 명시한다.

### 5.6 산출물 및 manifest 회수

- export JSON, prepare 산출물, stage1 결과, stage2 결과, `pipeline_manifest.json`을 회수한다.
- `pipeline_manifest.json`은 기본적으로 `work_dir/pipeline_manifest.json` 한 파일로 덮어써지므로 provider별 백업 복사를 권장한다.

### 5.7 결과 비교표 작성

- 실험 메타데이터, 기대 ground truth, prepare 결과, stage1 결과, stage2 요약, 비교 메모를 한 표로 정리한다.

## 6. base-name 규칙

`run_analysis_pipeline.py`는 `--base-name` 기준으로 아래 산출물을 만든다.

- `data/processed/<base>_llm_input.json`
- `data/processed/<base>_analysis_candidates.json`
- `data/processed/<base>_noise_summary.json`
- `data/processed/<base>_stage1_results.json`
- `data/processed/<base>_stage1_errors.json`
- `reports/<base>_stage2_report_input.json`
- `reports/<base>_stage2_report.json`
- `reports/<base>_stage2_report.md`

권장 base-name 규칙:

```text
<provider-prefix>-<webservice>-<attack-category>-<start>_to_<end>
```

권장 provider prefix:

- `op-`: OpenAI
- `cl-`: Anthropic / Claude

권장 예시:

- `op-juice-auth-2026-04-24_13-20-00_to_2026-04-24_13-37-00`
- `cl-juice-auth-2026-04-24_13-20-00_to_2026-04-24_13-37-00`
- `op-opencart-sqli-2026-04-25_09-10-00_to_2026-04-25_09-18-00`

base-name에 포함할 권장 요소:

- provider
- webservice
- attack_category
- 실험 시간 구간

필요하면 `experiment_set`도 추가할 수 있다.

예:

- `op-juice-a-auth-2026-04-24_13-20-00_to_2026-04-24_13-37-00`

---

## 7. 현재 파이프라인 산출물 규칙

현재 코드 기준 경로는 아래와 같다.

- export: `data/raw`
- prepare / stage1: `data/processed`
- stage2: `reports`
- manifest: `work_dir/pipeline_manifest.json`

현재 `src/export_db_logs_cli.py`의 자동 파일명 규칙:

- 날짜 단위 export: `data/raw/security_2026-04-24_kst.json`
- 시간 구간 export: `data/raw/security_2026-04-24_13-20-00_to_2026-04-24_13-37-00_kst.json`

현재 `src/run_analysis_pipeline.py`의 기본 동작:

- `--work-dir` 아래 `data/processed`, `reports`를 기본 경로로 사용한다.
- `pipeline_manifest.json`은 `work_dir` 루트에 저장된다.
- 같은 위치에서 다시 실행하면 manifest는 최신 실행 결과로 덮어써진다.

---

## 8. manifest 보관 원칙

`pipeline_manifest.json`은 provider 비교에서 특히 중요하다.

이유:

- 실제 실행 인자와 입력 파일을 복원할 수 있다.
- provider/model, mode, known_asset_ips, 산출물 경로를 한 번에 확인할 수 있다.
- 같은 이름으로 덮어써지므로 백업하지 않으면 이전 실행 정보를 잃는다.

권장 절차:

1. provider별 pipeline 실행 직후 manifest를 개별 파일명으로 복사한다.
2. 복사본은 `reports/` 또는 별도 비교 결과 디렉터리에 보관한다.

예:

```bash
cp /opt/web_log_analysis/pipeline_manifest.json /opt/web_log_analysis/reports/op-pipeline_manifest.json
cp /opt/web_log_analysis/pipeline_manifest.json /opt/web_log_analysis/reports/cl-pipeline_manifest.json
```

필요하면 시간 구간을 포함해 더 구체적으로 보관한다.

예:

```bash
cp /opt/web_log_analysis/pipeline_manifest.json /opt/web_log_analysis/reports/op-juice-auth-2026-04-24_13-20-00_to_2026-04-24_13-37-00_pipeline_manifest.json
```

## 9. 서비스별 요청 세트 구조

비교 실험 문서는 두 층으로 나눈다.

1. 공통 절차 문서: 이 문서(`docs/98_비교_실험_요청_세트_표준.md`)
2. 서비스별 요청 세트 문서: 예: `docs/98A_A세트_비교실험_실행요청.md`

분리 이유:

- 시간 구간 기록, export, pipeline, 보관 절차는 웹서비스가 바뀌어도 공통이다.
- 실제 요청 세트는 웹서비스별 엔드포인트와 ground truth가 다르다.
- 앞으로 Juice Shop 외 OpenCart 등으로 확장하기 쉽다.

권장 세트 분리 예:

- `A`: baseline / auth
- `B`: SQLi
- `C`: XSS
- `D`: Traversal / HPP / Dir Probe

## 10. CLI 명령 예시

아래 예시는 현재 운영 경로 `/opt/web_log_analysis` 기준이며, 그대로 복붙 가능한 수준으로 유지한다.

### 10.1 공통 환경 적용

```bash
cd /opt/web_log_analysis
source .venv/bin/activate
set -a
source ./config/llm.env
set +a
```

### 10.2 시간 구간 export 예시

```bash
python ./src/export_db_logs_cli.py \
  --host "$LOG_DB_HOST" \
  --port "$LOG_DB_PORT" \
  --user "$LOG_DB_USER" \
  --password "$LOG_DB_PASSWORD" \
  --start "2026-04-24 13:20:00" \
  --end   "2026-04-24 13:37:00" \
  --table security \
  --pretty
```

예상 export 파일:

```text
./data/raw/security_2026-04-24_13-20-00_to_2026-04-24_13-37-00_kst.json
```

### 10.3 OpenAI full pipeline 예시

```bash
python ./src/run_analysis_pipeline.py \
  --llm-provider openai \
  --export-input ./data/raw/security_2026-04-24_13-20-00_to_2026-04-24_13-37-00_kst.json \
  --work-dir /opt/web_log_analysis \
  --base-name op-security_2026-04-24_13-20-00_to_2026-04-24_13-37-00_kst \
  --mode routine \
  --pretty
```

### 10.4 Anthropic full pipeline 예시

```bash
python ./src/run_analysis_pipeline.py \
  --llm-provider anthropic \
  --stage1-model "$ANTHROPIC_MODEL" \
  --stage2-model "$ANTHROPIC_MODEL" \
  --export-input ./data/raw/security_2026-04-24_13-20-00_to_2026-04-24_13-37-00_kst.json \
  --work-dir /opt/web_log_analysis \
  --base-name cl-security_2026-04-24_13-20-00_to_2026-04-24_13-37-00_kst \
```

## 11. 결과 비교 기록 기준

실험 후에는 최소한 아래 항목을 기록한다.

- 실험 날짜
- 웹서비스
- 세트 이름
- 공격 카테고리
- 시간 구간
- raw export 파일명
- provider / model
- 실제 결과 `meta.provider`
- 실제 결과 `meta.selected_model`
- known_asset_ips
- 기대 ground truth
- prepare 결과
- stage1 결과
- stage2 요약
- 비교 메모
- 최종 평가

표준 결과 기록 템플릿이 필요하면 별도 문서로 분리할 수 있다.

## 12. 주의사항

- 모든 요청은 승인된 테스트 환경에서만 수행한다.
- 비교 실험의 기준은 GUI가 아니라 CLI 요청 세트다.
- 같은 raw log 비교에서는 `known_asset_ips`, top-N, `mode`를 바꾸지 않는다.
- `config/llm.env`를 한 번만 적용하고 provider는 실행 인자 기준으로 고정한다.
