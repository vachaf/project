## 1. 통합 결론

D세트는 Path Traversal, HTTP Parameter Pollution(HPP), Directory Probing을 분리해 검증했다.

최종 판단은 다음과 같다.

```text
D세트는 R1/R2 핵심 탐지 목표를 충족했다.
R1 Traversal은 D-01~D-03 기준으로 성공했고,
R2 HPP는 HPP+SQLi / HPP+XSS 결합 공격 탐지에 성공했다.
R3 Directory Probing은 실행과 보수적 판단은 성공했지만,
burst probing을 sequence-level context로 묶는 능력은 부분 성공이다.
```

D세트 전체에서 두 provider 모두 Apache 로그 기반 한계를 대체로 준수했다. 특히 실제 파일 노출, SQLi 성공, XSS 실행, HPP 처리 성공을 단정하지 않았다.

## 2. 라운드별 요약

| Round | 주제 | 핵심 결과 | 최종 판단 |
|---|---|---|---|

| R2 | HPP | HPP+SQLi, HPP+XSS가 candidate로 선별됨. benign HPP는 supporting context | 성공 |

## 3. R1 Traversal 통합 평가

R1은 `/etc/passwd`, `.env`, null byte, URL encoded traversal을 중심으로 수행했다.

| 항목 | 결과 |
|---|---|
| raw rows | 6 |
| candidate rows | 3 |
| filtered out rows | 3 |
| verdict | `suspicious_path_traversal` 3 |
| severity | `medium` 3 |
| 핵심 성공 | D-01~D-03 모두 path traversal candidate로 선별 |
| 한계 | D-04/D-05는 핵심 후보에 포함되지 않음 |

R1의 핵심 성공 지점은 `raw_request`와 normalized `uri`의 차이를 보존한 점이다. D-01에서는 `/public/images/../../../../etc/passwd`가 `uri=/etc/passwd`로 정규화되는 흐름이 확인되었고, D-02는 `%00.pdf`, D-03은 `%2e%2e%2f` 기반 `.env` 접근 의도를 보존했다.

다만 R1 전체를 D-01~D-05 완전 성공으로 표현하면 안 된다. 실제 핵심 평가는 D-01~D-03에 한정된다.

## 4. R2 HPP 통합 평가

R2는 benign duplicate parameter와 HPP+공격 payload 결합을 구분하는 실험이었다.

| 항목 | 결과 |
|---|---|
| raw rows | 4 |
| candidate rows | 2 |
| filtered out rows | 2 |
| supporting events | 1 |
| verdict | `suspicious_sqli` 1, `suspicious_xss` 1 |
| 핵심 성공 | HPP+SQLi / HPP+XSS 결합 문맥 탐지 |
| 한계 | POST body HPP는 현재 baseline에서 직접 확인 불가 |

R2의 핵심은 D-12와 D-13이다.

- D-12: `q=apple&q=x')) OR 1=1 --`
- D-13: `q=apple&q=<script>alert(1)</script>`

두 요청은 HPP 문맥과 SQLi/XSS payload가 함께 보존되었고, Stage1/Stage2에서 공격 결합 문맥으로 해석되었다. D-11 benign duplicate는 candidate로 과승격되지 않고 supporting context로 남았다.

OpenAI는 두 사건을 모두 `medium`으로 보수적으로 분류했고, Claude는 SQLi를 `high`, XSS를 `medium`으로 더 강하게 보았다.

## 5. R3 Directory Probing 통합 평가

R3는 민감 경로 탐색과 burst probing을 확인하는 실험이었다.

| 항목 | 결과 |
|---|---|
| raw rows | 10 |
| candidate rows | 1 |
| filtered out rows | 9 |
| verdict | `suspicious_scan` 1 |
| severity | `low` 1 |
| filtered out breakdown | `low_signal_dir_probe` 7, `low_signal_fuzzing` 2 |
| 핵심 성공 | 보수적 판단과 성공 단정 억제 |
| 한계 | burst probing sequence grouping 부족 |

R3에서는 `/.git/config`, `/.env`, `/admin`, `/administrator`, `/backup`, `/backups`, `/config`, `/server-status`, `/phpmyadmin`, `/manager/html` 요청이 수집되었다. 이 중 `/server-status` 403 요청만 Stage1 candidate로 올라갔다.

나머지 9건은 low-signal로 내려갔다. 이 판단은 보수적이고 안전하지만, D세트 R3의 목표였던 “short-burst probing을 하나의 sequence/context로 묶는 능력”은 충분히 만족하지 못했다.

## 7. Apache 로그 기반 한계

D세트 전체에서 확정할 수 없는 항목은 다음과 같다.

- 실제 파일 내용 반환 여부
- `/etc/passwd`, `.env` 내용 노출 여부
- `php://filter` wrapper 실제 처리 여부
- 서버가 HPP 파라미터 중 어떤 값을 사용했는지
- SQLi 성공 또는 DB 결과 변경 여부
- XSS 브라우저 실행 여부
- Directory probing 대상 리소스의 실제 존재 여부
- 200 `text/html` 대용량 응답이 실제 민감 리소스인지 SPA fallback인지 여부

따라서 D세트 보고 표현은 다음 원칙을 따른다.

```text
Apache 로그상 공격 의도와 요청 패턴은 관찰할 수 있다.
그러나 response body 원문, 애플리케이션 내부 처리 결과, DB 결과가 없으면
성공 공격이나 실제 정보 노출로 확정하지 않는다.
```

## 8. 후속 개선 과제

D세트 이후 우선순위가 높은 개선은 R3에서 나온다.

1. `low_signal_dir_probe` burst grouping 강화
2. Stage2의 후보 밖 탐색성 요청 섹션 강화
3. filtered-out 경로 목록과 시간 집중도 요약 추가
4. fallback HTML 반복 응답을 보조 지표로 정리
5. `/server-status` 단일 candidate와 주변 burst 요청 연결 개선

R1/R2는 현재 구조로도 핵심 검증은 충분히 수행되었다. 단, R1의 D-04/D-05를 정식 평가하려면 target과 실행 명령을 먼저 재정리해야 한다.

---

## 9. 최종 평가

D세트는 전체적으로 성공으로 본다. 다만 라운드별 성공 수준은 다르다.

- R1: Traversal 핵심 3건 탐지 성공
- R2: HPP 결합 공격 탐지 성공
- R3: 보수적 판단 성공, sequence grouping은 부분 성공

최종적으로 D세트는 Apache 로그 기반 분석에서 “시도 관찰”과 “성공 단정 금지” 원칙을 유지하면서, traversal/HPP/probing 계열을 구분하는 데 충분한 근거를 제공했다.

---

## 10. 발표용 한 줄 정리

D세트에서는 R1 Traversal과 R2 HPP 결합 공격 탐지는 성공했고, R3 Directory Probing은 저신호 요청을 보수적으로 낮추는 데는 성공했지만 burst probing을 하나의 sequence-level context로 묶는 부분은 개선이 필요했다. 두 provider 모두 Apache 로그만으로 실제 파일 노출, SQLi 성공, XSS 실행, HPP 처리 성공을 단정하지 않았다.
