## 1. 실험 개요

이번 기록은 2026-04-25에 수행한 B세트 Round 1의 실제 산출물 정리다. 실행 기준은 `B-01 ~ B-11`이었고, 운영 메모대로 `B-04`는 생략했으며 `B-07`은 마지막 배치에 두었다.

## 2. 실험 목적

이번 라운드의 목적은 B세트 Round 1에서 auth bypass, union-based, error-based 요청이 raw export, prepare, stage1, stage2를 거치며 어떻게 남는지 비교하는 것이다. 특히 실제 최종 비교가 어느 축에 의해 형성되었는지와, 그 해석이 현재 baseline visibility 한계와 어떻게 연결되는지를 확인한다.

## 3. 실행 범위

- Round 1 범위: `B-01 ~ B-11`
- 생략: `B-04`
- 마지막 배치: `B-07`

  - `B-01`, `B-02`, `B-03`
  - `B-05`, `B-06`, `B-07`

## 5. raw export 요약

raw export의 `security` 행은 총 10건이다. 실행 문서상 Round 1 범위와 대체로 일치하며, `B-04`를 제외한 10개 요청이 실제로 관찰되었다.

- `POST /rest/user/login` 3건
  - `B-01` `200 application/json 799 bytes`
  - `B-02` `200 application/json 799 bytes`
  - `B-03` `500 text/html 1449 bytes`
- `GET /rest/products/search` 6건
  - `B-05`, `B-06`, `B-07`, `B-08`, `B-09`, `B-11`
- `GET /rest/products/1'` 1건
  - `B-10` `500 text/html 3045 bytes`

즉 raw export 기준으로는 `B-01`과 `B-02`의 auth bypass 성공형 요청이 실제로 존재했다.

## 6. prepare 결과 요약

OpenAI와 Anthropic의 `llm_input`은 같은 raw export를 바탕으로 동일하게 구성되었다.

- 전체 raw row: 10건
- candidate: 7건
- filtered out: 3건
- filtered_out_breakdown: `low_signal_fuzzing` 3건

candidate로 남은 것은 다음 7건이다.

- `B-03` `POST /rest/user/login` `500`
- `B-05`, `B-06`, `B-07`, `B-08`, `B-09`, `B-11`

filtered out 된 3건은 다음과 같이 정리된다.

- `B-01` `POST /rest/user/login` `200 application/json 799 bytes`
- `B-02` `POST /rest/user/login` `200 application/json 799 bytes`
- `B-10` `GET /rest/products/1'` `500`

따라서 prepare 이후의 실제 비교 입력은 auth bypass 성공형 2건과 path 기반 오류 1건이 빠진 상태였고, 결과적으로 `/rest/products/search` 중심 SQLi 후보 6건과 `POST /rest/user/login` 500 1건으로 압축되었다.

## 7. OpenAI 결과 요약

OpenAI stage1 분포는 `suspicious_sqli` 6건, `server_error_probe` 1건이다. severity 분포는 `high` 4건, `medium` 2건, `low` 1건이다.

핵심 특징은 다음과 같다.

- `/rest/products/search` 6건은 모두 SQLi 축으로 유지했다.
- `sqlite_master` 접근 시도(`B-06`), `Users` 대상 UNION(`B-07`), NULL padding(`B-05`), CAST 변형(`B-11`)을 모두 `suspicious_sqli`로 분류했다.
- `POST /rest/user/login` 500 1건(`B-03`)은 `server_error_probe` / `low`로 두어, 인증 오용보다는 오류 탐색 쪽으로 더 보수적으로 읽었다.

OpenAI stage2 보고서도 최종적으로는 `/rest/products/search` 중심 SQLi 시도와 소규모 서버 오류 탐색으로 요약했다.

## 8. Anthropic 결과 요약

Anthropic stage1 분포는 `suspicious_sqli` 6건, `suspicious_auth_abuse` 1건이다. severity 분포는 `high` 4건, `medium` 3건이다.

핵심 특징은 다음과 같다.

- `/rest/products/search` 6건은 OpenAI와 마찬가지로 SQLi 축으로 유지했다.
- `POST /rest/user/login` 500 1건(`B-03`)은 `suspicious_auth_abuse` / `medium`으로 승격했다.
- stage2 보고서에서는 SQLi 6건 외에 인증 엔드포인트 의심 접근 1건을 별도 축으로 더 강하게 서술했다.

즉 Anthropic의 차이는 search API SQLi 해석보다 `B-03`의 성격을 인증 기능 오용으로 읽은 점에 있다.

## 9. 비교 메모

이번 라운드의 실질 비교는 `/rest/products/search` 중심 SQLi 비교로 읽는 것이 정확하다. 이유는 `B-01`과 `B-02`가 raw export에는 존재했지만 prepare candidate에서 빠졌기 때문이다.

공통점:

- 두 provider 모두 search API의 UNION, `sqlite_master`, `Users`, quote breakout, CAST 변형을 핵심 incident로 읽었다.
- 두 provider 모두 known asset 문맥을 함께 언급했다.

차이점:

- OpenAI는 `B-03`를 `server_error_probe`로 낮게 두었다.
- Anthropic은 같은 요청을 `suspicious_auth_abuse`로 유지했다.

정리하면 이번 R1의 provider 차이는 auth bypass 성공형 200 요청의 직접 비교가 아니라, prepare 이후 남은 `search API 중심 SQLi 6건 + login 500 1건`에 대한 해석 차이로 보는 편이 정확하다.

## 10. blind spot 해석

이번 raw export에는 `B-01`, `B-02`가 모두 존재했고 둘 다 `POST /rest/user/login`, `200`, `application/json`, `799 bytes`였다. 그러나 이 두 건은 candidate 구성 단계에서 누락되었다.

이 현상은 현재 baseline이 Apache 공통/security 로그 표면에 직접 남는 신호를 우선 사용하기 때문에 생긴 결과로 해석하는 것이 타당하다. 즉 공격 신호가 POST JSON body 내부에만 있고, 로그 표면에는 `POST /rest/user/login`, `200`, 일반적인 JSON 응답 정도만 남는 경우 prepare 단계에서 후보화가 누락될 수 있다.

따라서 이번 결과는 "모델이 auth bypass를 못 알아봤다"기보다 다음처럼 읽는 것이 맞다.

- candidate 구성 단계에서 이미 누락되었다.
- 로그 표면에 직접 드러나지 않는 body-only 성공 신호는 현재 baseline의 직접 입력이 아니다.
- 이번 최종 비교가 search API 중심 SQLi 비교로 수렴한 이유는 모델 성능 차이보다 데이터 가시성 한계에 더 가깝다.

이 해석은 [docs/05_Export_LLM_분석_전략.md](/mnt/c/Users/nkiak/Documents/3_1/project_repo/docs/05_Export_LLM_분석_전략.md)와 [docs/99_POST_body_visibility_한계와_해석_기준.md](/mnt/c/Users/nkiak/Documents/3_1/project_repo/docs/99_POST_body_visibility_한계와_해석_기준.md)의 원칙과 충돌하지 않는다.

## 11. 최종 평가

이번 B세트 Round 1은 실행 문서상 `B-01 ~ B-11` 비교였지만, 실제 산출물 기준 최종 비교 축은 `/rest/products/search` 중심 SQLi 해석이었다. raw export에는 auth bypass 성공형 `B-01`, `B-02`가 분명히 존재했지만, baseline visibility 한계 때문에 prepare candidate에 포함되지 않았다.

따라서 이번 라운드의 결론은 다음과 같다.

- raw 기준으로는 auth bypass 성공형 요청이 실제로 있었다.
- prepare 이후 비교 입력은 search API 중심 SQLi 6건과 login 500 1건으로 재구성되었다.
- provider 차이는 주로 `B-03` 해석 수준에서 나타났고, 핵심 incident 축은 양쪽 모두 search API SQLi였다.
- 이번 결과는 모델 우열보다는 현재 baseline의 POST body visibility 한계를 드러낸 사례로 기록하는 것이 가장 보수적이고 정확하다.
