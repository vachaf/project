## 1. 개선 목적

기존 D세트 R3에서는 directory probing 성격의 요청 10건이 수집되었지만, Stage1 candidate로는 `/server-status` 403 요청 1건만 올라갔다. 나머지 요청은 `low_signal_dir_probe` 또는 `low_signal_fuzzing`으로 filtered out 처리되었다.

이 판단은 보수적이라는 점에서 적절했다. 그러나 Stage2가 다음 정보를 충분히 활용하지 못하는 한계가 있었다.

- 같은 출발지 IP에서 짧은 시간에 여러 민감/관리 경로를 요청했다는 점
- `/.git/config`, `/.env`, `/admin`, `/backup`, `/phpmyadmin` 등이 하나의 probing 흐름이라는 점
- 다수의 `200 text/html` 응답이 동일한 크기로 반복되어 fallback HTML 가능성이 있다는 점
- `/server-status` 403 candidate가 주변 probing burst 중 하나라는 점

따라서 이번 개선은 low-signal 요청을 candidate로 과승격하지 않고, Stage2에 sequence-level context를 전달하는 것을 목표로 했다.

## 2. 개선 내용

추가된 핵심 구조는 다음과 같다.

```json
{
  "probing_sequence_summaries": [
    {
      "category": "low_signal_dir_probe_burst",
      "policy": "context_only",
      "src_ip": "...",
      "request_count": 10,
      "distinct_path_count": 10,
      "sample_paths": ["/.git/config", "/.env", "/admin", "/backup"],
      "status_counts": {"200": 9, "403": 1},
      "reason_hints": ["dir_probe:burst", "dir_probe:sensitive_path"]
    }
  ]
}
```

중요한 설계 원칙은 다음이다.

- 개별 low-signal directory probe를 incident candidate로 올리지 않는다.
- `probing_sequence_summaries`는 `context_only`다.
- 같은 `src_ip`, 짧은 시간 window, 민감/관리/백업 경로의 다수 접근을 sequence summary로 보존한다.
- 반복되는 `200 text/html` 동일/유사 응답 크기는 fallback-like HTML 보조 지표로만 사용한다.
- response body 원문이 없으므로 민감 파일 노출 성공은 확정하지 않는다.
- `lab-d-set-*`, 특정 IP, 특정 응답 크기 값은 탐지 조건으로 사용하지 않는다.

핵심 변화는 탐지 후보 수가 아니라 Stage2 입력의 문맥 정보다.

이로 인해 탐지 민감도를 무리하게 높이지 않고도, Stage2가 후보 밖 탐색성 요청을 더 정확히 설명할 수 있게 되었다.

## 4. 개선 후 raw/prepare 관찰

개선 후에도 raw export는 총 10건이다.

| URI | status | response_body_bytes | 해석 |
|---|---:|---:|---|
| `/.git/config` | 200 | 75002 | sensitive repo/config probe, fallback-like 가능성 |
| `/.env` | 200 | 75002 | sensitive config probe, fallback-like 가능성 |
| `/admin` | 200 | 75002 | admin path guessing, fallback-like 가능성 |
| `/administrator` | 200 | 75002 | admin path guessing, fallback-like 가능성 |
| `/backup` | 200 | 75002 | backup path probe, fallback-like 가능성 |
| `/backups` | 200 | 75002 | backup path probe, fallback-like 가능성 |
| `/config` | 200 | 75002 | config path probe, fallback-like 가능성 |
| `/server-status` | 403 | 279 | access control block, Stage1 candidate |
| `/phpmyadmin` | 200 | 75002 | admin/tool path probe, fallback-like 가능성 |
| `/manager/html` | 200 | 75002 | manager path probe, fallback-like 가능성 |

개선 후 `probing_sequence_summaries`에는 이 10개 경로가 하나의 burst context로 묶였다.

요약:

```text
same src_ip + short time window + 10 distinct probing paths
status distribution: 200 x 9, 403 x 1
repeated text/html response size: 75002 x 9
policy: context_only
```

주의: `75002`는 산출물에서 관찰된 값일 뿐, 코드 탐지 조건으로 hard-code된 값이 아니다.

## 8. 남은 한계

### 8.1 Stage1은 여전히 1건만 직접 판단

`/.git/config`, `/.env`, `/admin`, `/phpmyadmin` 등은 Stage1 candidate로 직접 올라가지 않는다. 이번 개선은 Stage1 candidate 확장이 아니라 Stage2 context 보강이다.

이는 현재 정책상 적절하다. 단발 또는 fallback-like directory probe를 무조건 incident로 올리면 오탐이 커질 수 있다.

### 8.2 User-Agent 해석 보정 필요

보고서에서 `lab-d-set-probe-burst-1`을 프로브성 User-Agent로 언급한다. 이는 산출물 설명으로는 가능하지만, 일반 운영 환경에서는 다음 근거가 더 중요하다.

- 같은 src_ip
- 짧은 시간 window
- 여러 민감/관리 경로
- status 분포
- 반복 text/html 응답 크기

### 8.3 response body 원문 부재

여전히 실제 `.git/config`, `.env`, admin page, backup file이 노출됐는지는 확인할 수 없다. 반복 200 응답은 fallback-like 가능성으로만 해석해야 한다.

## 9. 최종 판단

D세트 R3 개선은 성공으로 판단한다.

이유:

1. 기존 보수적 candidate 정책을 유지했다.
2. candidate 수와 severity를 증가시키지 않았다.
3. low-signal directory probe를 incident로 과승격하지 않았다.
4. Stage2에 sequence-level context를 전달했다.
5. OpenAI와 Claude 모두 개선된 context를 실제 보고서에 반영했다.
6. 반복 200 `text/html` 응답을 fallback-like 가능성으로 설명하고, 민감 리소스 노출 성공으로 단정하지 않았다.

최종 결론:

```text
D세트 R3 개선 후 재처리 결과, 탐지 후보 수와 severity는 기존과 동일하게 유지되었지만,
새로운 probing_sequence_summaries가 추가되어 Stage2가 주변 저신호 요청을 하나의 directory probing sequence로 해석할 수 있게 되었다.
따라서 탐지 민감도를 과도하게 높이지 않으면서 후보 밖 탐색성 요청의 문맥 전달력이 개선되었다.
```
