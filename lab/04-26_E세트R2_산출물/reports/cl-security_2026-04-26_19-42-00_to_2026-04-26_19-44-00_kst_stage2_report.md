  - 근거: 내부 IP라도 User-Agent 변조, 공격 기법 사용, 다중 경로 rapid probing 등의 비정상 패턴은 감지해야 합니다. 내부 테스트와 미승인 활동을 구분하기 위한 운영 절차 수립이 필요합니다.

## 9. 신뢰도와 한계
- Apache 웹 로그만으로 PHP 파일 래퍼 공격의 실제 성공 여부(설정 파일 내용 노출)를 확정할 수 없습니다. 200 상태코드와 32KB 응답은 서버 처리를 시사하나, 응답 본문 원문 검증 없이는 설정 정보 노출 성공으로 단정할 수 없습니다.
- probing_sequence_summary에서 /config.php와 /admin/config.php 경로가 200 text/html으로 기록되었으나, 이는 애플리케이션의 fallback HTML 템플릿 또는 error 페이지일 수 있으므로 실제 설정 파일 노출로 단정하지 않습니다.
- 출발지 IP 192.168.56.110은 known asset으로 등록되어 있어, 공격자 단정을 피하고 내부 테스트/자체 호출/운영 점검 가능성을 함께 고려했습니다. 조직 내 승인 여부 확인이 필수적입니다.
- 두 번째 요청(index.php 대상)의 404 응답은 서버가 해당 파라미터/경로 조합을 거부했음을 나타내나, 첫 번째 요청이 200으로 응답한 것과 비교하면 보호 메커니즘이 일관되지 않을 가능성이 있습니다.
- 필터링된 저신호 directory probe 2건은 후보 밖이므로 자세한 분석 대상이 아니나, 동일 IP 맥락에서 탐색 활동의 범위가 더 넓을 수 있음을 의미합니다.
- supporting_events가 없으므로 같은 IP의 시간적 연관성을 보충적으로 분석할 추가 정보가 부족합니다. WAF, 프록시, 또는 애플리케이션 로그 상관을 통한 보충 검증이 권장됩니다.

## 10. 발표용 한 줄 정리
이번 분석 구간(19:42~19:44, 2분)에서 PHP 스트림 래퍼 공격 기법을 사용한 파일 소스 추출 시도 2건과 민감 설정 경로에 대한 directory probing 흐름이 탐지되었습니다. 출발지 IP는 known asset이므로 즉시 담당자 연락을 통해 승인 여부를 확인해야 하며, 첫 번째 요청의 200 상태 응답은 config.php 파일 설정 정보 노출 가능성을 시사하므로 응답 본문 검증이 긴급합니다. 동시에 PHP 래퍼 차단 설정, WAF 규칙 완성도, 민감 파일의 웹 접근성 등 근본적 보안 통제를 점검하여 향후 유사 공격에 대한 방어를 강화해야 합니다.
