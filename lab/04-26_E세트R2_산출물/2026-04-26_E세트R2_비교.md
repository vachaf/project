## 1. 결론

E세트 R2는 코드 수정 후 의도한 개선 목표를 충족했다.

가장 중요한 변화는 **E-12 (`path=php://filter/convert.base64-encode/resource=config.php`)가 candidate로 승격된 것**이다. 기존 R2에서는 이 요청이 `low_signal_fuzzing` 또는 후보 밖 저신호 요청으로 빠져 LLM이 분석하지 못했다. 수정 후에는 `suspicious_file_disclosure` 후보로 보존되었고, OpenAI와 Anthropic 모두 high severity로 판단했다.

동시에 `/config.php`, `/admin/config.php` 직접 접근 2건은 candidate로 과승격되지 않고 `low_signal_dir_probe` 및 `probing_sequence_summaries`의 context-only 문맥으로 유지되었다. 따라서 이번 수정은 “탐지 보강”과 “과탐 억제”를 모두 만족한 것으로 판단한다.

## 2. 입력 로그 요약

이번 R2의 원시 로그 4건은 수정 전후 동일하다. 달라진 것은 실험 트래픽이 아니라 `prepare_llm_input.py`의 후보 선정 및 hint 부여 방식이다.

| ID | 요청 성격 | URI / Query | 상태 | 응답 크기 | 해석 |
|---|---|---|---:|---:|---|
| E-11 | PHP wrapper via `route=` | `/index.php?route=php://filter/convert.base64-encode/resource=index.php` | 404 | 19,874B | wrapper 기반 source disclosure 시도, 실패 또는 route 미인식 |
| E-12 | PHP wrapper via `path=` | `/index.php?path=php://filter/convert.base64-encode/resource=config.php` | 200 | 32,777B | wrapper 기반 config.php 읽기 시도, 성공 단정 금지 |
| E-13a | direct config probe | `/config.php` | 200 | 0B | 직접 설정 파일 경로 탐색, 본문 노출 증거 없음 |
| E-13b | direct admin config probe | `/admin/config.php` | 200 | 0B | 직접 admin 설정 파일 경로 탐색, 본문 노출 증거 없음 |

중요한 점은 `status_code=200`만으로 성공을 확정하지 않는 것이다. 특히 `/config.php`, `/admin/config.php`는 `response_body_bytes=0`이므로 파일 내용 노출로 볼 근거가 없다.

## 3. 전처리 개선 효과

### 3.1 candidate 분포

수정 후 `llm_input` 기준 count는 다음과 같다.

```text
candidate_rows=2
filtered_out_rows=2
probing_sequence_summaries=1
supporting_events=0
false_positive_review_candidates=0
```

두 candidate는 모두 `suspicious_file_disclosure`로 들어갔다.

### 3.2 수정 전후 후보 선정 비교

| 요청 | 수정 전 score / 처리 | 수정 후 score / 처리 | 핵심 변화 |
|---|---|---|---|
| E-11 `route=php://filter...index.php` | score 4, candidate | score 14, candidate | wrapper 전용 hint 추가로 confidence 상승 |
| E-12 `path=php://filter...config.php` | candidate 미진입 | score 12, candidate | 기존 누락 해소 |
| E-13a `/config.php` | low_signal_dir_probe | low_signal_dir_probe | 과승격 방지 유지, config probe hint 보강 |
| E-13b `/admin/config.php` | low_signal_dir_probe | low_signal_dir_probe | 과승격 방지 유지, admin config probe hint 보강 |

핵심 변화는 `php://filter`, `convert.base64-encode`, `resource=` 조합을 일반적인 PHP/file disclosure intent로 점수화한 것이다. 이로 인해 `route=`와 `path=` 양쪽 wrapper 변형이 모두 LLM 입력에 들어가게 되었다.

### 3.3 E-11: route wrapper 유지 및 score 상승

`lab-e-set-php-wrapper-route-1`은 기존에도 candidate였고, 수정 후에도 유지되었다.

```text
query_string=?route=php://filter/convert.base64-encode/resource=index.php
status_code=404
score=14
verdict_hint=suspicious_file_disclosure
```

핵심 hint:

```text
file_disclosure:php_filter_wrapper
file_disclosure:base64_source_intent
file_disclosure:resource_parameter
file_disclosure:sensitive_resource:index_php
```

평가:

- PHP wrapper 의도는 명확히 탐지되었다.
- 404이므로 실제 파일 노출 성공은 아니다.
- “시도 탐지”로 보는 것이 맞다.
- 수정 전에는 긴 query와 404 error 중심으로 겨우 candidate가 되었지만, 수정 후에는 공격 기법 자체가 근거로 반영되었다.

### 3.4 E-12: path wrapper 신규 candidate 승격

이번 수정의 핵심 개선이다.

`lab-e-set-php-wrapper-path-1`이 candidate로 올라왔다.

```text
query_string=?path=php://filter/convert.base64-encode/resource=config.php
status_code=200
score=12
verdict_hint=suspicious_file_disclosure
```

핵심 hint:

```text
file_disclosure:php_filter_wrapper
file_disclosure:base64_source_intent
file_disclosure:resource_parameter
file_disclosure:sensitive_resource:config_php
```

평가:

- 기존에 놓쳤던 `path=` 기반 PHP wrapper file disclosure intent가 보존되었다.
- `config.php` 대상이므로 E-11보다 위험도는 높게 볼 수 있다.
- `status_code=200`, `response_body_bytes=32777`은 별도 확인 필요성을 높이는 지표다.
- 다만 Apache 로그에 response body 원문이 없으므로 실제 config 내용 노출 성공은 확정할 수 없다.

### 3.5 E-13: direct config path 과승격 방지

`/config.php`, `/admin/config.php`는 candidate가 아니라 filtered/context로 유지되었다.

raw 기준:

```text
/config.php        status=200 response_body_bytes=0
/admin/config.php  status=200 response_body_bytes=0
```

전처리 결과:

```text
filtered_out_breakdown={"low_signal_dir_probe": 2}
probing_sequence_summaries=1
```

probing sequence에는 다음 sample path가 포함되었다.

```text
/index.php
/config.php
/admin/config.php
```

reason hint:

```text
dir_probe:burst
dir_probe:sensitive_path
dir_probe:sensitive_config_path
file_probe:config_php
file_probe:admin_config_php
dir_probe:admin_path
```

평가:

- direct config path 접근을 완전히 버리지 않고 context-only로 보존했다.
- 하지만 단발 직접 접근을 incident candidate로 과승격하지 않았다.
- `response_body_bytes=0`이므로 “본문 노출 증거 없음”으로 보는 것이 정확하다.
- 다만 OpenCart/PHP 환경에서는 `config.php`가 실행만 되고 출력이 없을 수 있으므로, 보고서에서는 “0B라서 안전”이 아니라 “Apache 로그만으로 노출 여부 확인 불가”라고 적어야 한다.

### 4.3 남은 한계: verdict label은 아직 `suspicious_path_traversal`

두 provider 모두 Stage1 verdict는 여전히 `suspicious_path_traversal`로 수렴했다.

```text
verdict 분포: {"suspicious_path_traversal": 2}
```

이는 현재 결과를 무효화할 정도의 문제는 아니다. reason_hints와 reasoning_summary에서는 PHP wrapper / file disclosure intent가 명확히 드러난다. 다만 기법 분류 관점에서는 정확도가 낮다.

PHP `php://filter/convert.base64-encode`는 단순 `../` 경로 순회와 다르다. PHP stream wrapper를 이용해 파일 소스 또는 설정 파일 내용을 읽으려는 LFI/source disclosure 계열 시도다. 따라서 향후 Stage1 verdict enum 또는 prompt에 `suspicious_file_disclosure`를 공식 verdict로 허용하는 보강을 검토할 수 있다.

권장 후속 개선:

```text
현재: suspicious_path_traversal  ← traversal과 php://filter가 함께 흡수됨
개선: suspicious_path_traversal  ← ../, %2e%2e 등 경로 순회
      suspicious_file_disclosure ← php://filter, wrapper, source/config disclosure intent
```

단, 지금 R2 재실험을 막을 정도의 문제는 아니다.

## 5. Stage2 판단 비교

### 5.1 OpenAI

OpenAI의 Stage2는 가장 보수적으로 정리되었다.

핵심 표현:

- “PHP wrapper를 이용한 파일 읽기/소스 노출 시도”
- “Apache 로그만으로 실제 파일 내용 노출 성공은 확인되지 않음”
- `/config.php`, `/admin/config.php`는 “짧은 시간대 추가 탐색”으로 해석
- known asset IP이므로 내부 테스트/운영 점검 가능성을 병기

평가:

- 가장 안정적이다.
- 200 text/html을 성공으로 단정하지 않았다.
- direct config path를 candidate가 아닌 probing context로 분리했다.
- 제목도 “PHP wrapper 기반 파일 노출 시도 정황”으로 핵심을 잘 드러냈다.

### 5.2 Anthropic

Anthropic도 PHP wrapper 2건을 모두 식별했다. 다만 OpenAI보다 더 공격적으로 위험도를 해석했다.

강점:

- `config.php` 대상 wrapper 요청의 위험도를 잘 강조했다.
- `base64 인코딩을 통한 소스코드 추출 의도`를 구체적으로 설명했다.
- known asset IP와 실험 User-Agent를 근거로 내부 테스트 가능성도 병기했다.
- E-12에 대해 P1 조치가 제안될 정도로 운영 대응 우선순위를 강하게 잡았다.

주의점:

- `status=200`, `response_body_bytes=32777`에 대해 “실제 파일 내용 반환 가능성”을 더 강하게 언급했다.
- “fallback HTML 템플릿일 가능성이 높다”처럼 Apache 로그 표면만으로는 확정하기 어려운 표현이 일부 있다.

따라서 Anthropic 결과를 사용할 때는 다음 문구를 추가해 보정하는 것이 좋다.

```text
200 text/html 및 응답 크기만으로 실제 PHP source/config 노출 성공을 확정할 수 없다. response body 원문 또는 애플리케이션 로그 확인이 필요하다.
```

## 8. Juice Shop과의 차이

E세트 R2는 Juice Shop 기반 실험과 성격이 다르다.

| 항목 | Juice Shop | OpenCart E R2 |
|---|---|---|
| 앱 구조 | Node/SPA/API 중심 | PHP/OpenCart route 중심 |
| 주요 공격 위치 | URL path 또는 REST query | `index.php` query parameter |
| 핵심 payload | traversal, SQLi, XSS | `php://filter`, `resource=config.php` |
| 성공 판단 근거 | JSON/API 응답, fallback HTML 차이 | PHP wrapper 처리 여부, text/html/0B 응답 해석 |
| admin/config path | SPA fallback 가능성 큼 | 실제 PHP 파일/관리자 경로일 수 있음 |
| 보수적 해석 필요 지점 | SPA fallback 200 | 200 text/html, 0B body, PHP empty output |

이번 수정은 OpenCart 전용 hard-code가 아니라 일반 PHP/file disclosure 신호를 추가한 것이므로, PHP 기반 다른 웹앱에도 적용 가능한 개선이다.

## 9. 남은 개선 과제

### 9.1 `suspicious_file_disclosure` verdict 신설 검토

현재 Stage1 최종 verdict는 여전히 `suspicious_path_traversal`로 나온다. 그러나 `php://filter/convert.base64-encode/resource=`는 경로 순회보다 PHP wrapper 기반 source/config disclosure에 가깝다.

후속 개선안:

```text
suspicious_file_disclosure
suspicious_source_disclosure
suspicious_lfi
```

중 하나를 Stage1 verdict 후보로 명시하는 방안을 검토한다.

### 9.2 direct config path 200/0B의 해석 보강

`/config.php`, `/admin/config.php`는 200 OK이지만 `response_body_bytes=0`이다. 이를 candidate로 즉시 승격하지 않는 현재 정책은 유지하는 것이 안전하다. 다만 보고서에서는 다음을 더 명시할 필요가 있다.

```text
200/0B는 파일 없음 또는 성공을 의미하지 않는다. PHP 설정 파일이 실행되었지만 출력이 없었을 가능성, 접근은 되었으나 본문이 없었던 가능성, 애플리케이션/Apache 설정상 빈 응답을 반환한 가능성을 모두 열어둔다.
```

후속 개선안:

- `probing_sequence_summaries`에 `file_probe:config_php_200_empty_body` 같은 context-only hint 추가 검토
- Stage2 prompt에 “200/0B direct config path는 성공 단정 금지, 단 확인 필요” 문구 보강

### 9.3 `convert.base64-encode` 의미 설명 강화

두 모델 모두 `base64-encode`를 언급했지만, 왜 이 기법이 위험한지에 대한 설명은 부족했다.

보강할 설명:

```text
php://filter/convert.base64-encode/resource=... 는 PHP가 파일을 실행하지 않고 파일 내용을 필터링된 형태로 읽어 반환하도록 유도하는 데 사용된다. 공격자는 base64 결과를 디코딩해 PHP 소스 또는 설정값을 확인하려는 의도를 가진다.
```

이 설명은 발표 자료나 최종 비교 보고서에 넣는 것이 좋다.

### 9.4 wrapper 변형 추가 테스트

추가로 검토할 수 있는 payload:

```text
/index.php?path=php://filter/convert.base64-encode/resource=admin/config.php
/index.php?route=php://filter/resource=config.php
/index.php?file=php://filter/convert.base64-encode/resource=config.php
```

다만 이 추가 실험은 E세트 R2 결론을 바꿔야 할 정도의 필수 작업은 아니다.

## 10. 최종 평가

E세트 R2는 코드 수정 후 성공적으로 개선되었다.

성공한 점:

1. 기존에 누락됐던 `path=php://filter...config.php`가 candidate로 보존되었다.
2. 두 PHP wrapper 요청이 모두 `suspicious_file_disclosure` hint를 받았다.
3. `/config.php`, `/admin/config.php`는 candidate로 과승격되지 않았다.
4. direct config path는 `probing_sequence_summaries`에서 context-only로 보존되었다.
5. OpenAI와 Anthropic 모두 PHP wrapper 기반 source/config disclosure intent를 해석했다.
6. 두 provider 모두 실제 파일 노출 성공을 대체로 단정하지 않았다.
7. Stage2 권고 조치가 기존보다 구체화되었다.

남은 개선점:

1. Stage1 verdict가 아직 `suspicious_path_traversal`로 나온다.
2. `suspicious_file_disclosure`를 Stage1 최종 verdict로 허용하거나 prompt에 더 명시할지 검토할 수 있다.
3. direct `/config.php` 200/0B 응답의 의미를 Stage2에서 더 분명히 제한해야 한다.
4. `convert.base64-encode`가 PHP 소스 노출에 사용되는 이유를 보고서/발표에서 보강하면 좋다.
5. Anthropic 보고서는 200/32KB 응답을 다소 강하게 해석하므로, 발표/문서화 시 “Apache 로그만으로 성공 미확정” 문구를 보강하는 것이 좋다.
