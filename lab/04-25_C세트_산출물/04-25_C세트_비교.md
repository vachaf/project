## 1. 실험 개요

C세트는 XSS 계열 요청에 대해 다음 세 가지를 확인하기 위한 실험이다.

1. URL encoding, double encoding, HTML entity encoding 등 인코딩/난독화된 XSS payload를 복원해 의미를 읽는가.
2. `<script>` 탐지를 넘어 `document.cookie`, `document.location`, `javascript:` protocol, event handler 등 구체적 위협 의도를 구분하는가.
3. 교육용 검색 또는 튜토리얼 검색을 실제 공격으로 단정하지 않고 false positive 계열로 낮출 수 있는가.

이번 산출물은 C세트 코드 보강 이후 재처리된 결과다. 특히 HTML entity decode view, XSS 세부 hint, `false_positive_review_candidates` 보존 구조가 적용되었다.

중요한 해석 원칙은 다음과 같다.

- Apache 로그상 payload 전송은 확인할 수 있다.
- 그러나 브라우저 실행 성공, 쿠키 탈취 성공, 외부 전송 성공은 Apache 로그만으로 확정할 수 없다.
- `500 text/html`은 서버 처리 오류 정황일 수 있으나 XSS 실행 성공 근거는 아니다.
- `200 application/json` 역시 브라우저 실행 성공 근거는 아니다.

## 4. 핵심 성공 지표 검토

팀원분이 제시한 네 가지 방향성은 대부분 반영할 수 있다. 다만 “완벽하게 충족”이라고 단정하기보다는 Apache 로그 기반 한계를 고려해 “실험 목적을 충족”으로 표현하는 것이 적절하다.

| 항목 | 검증 결과 | 평가 | 주요 근거 |
|---|---|---|---|
| C-08 오탐 제거 | HTML entity payload가 XSS 계열로 해석됨 | 성공 | `encoding:html_entity_payload`, `encoding:html_entity_decoded_xss`, `xss:html_entity_decoded_script` 계열 hint 반영 |
| XSS hint 강화 | event handler, JS protocol, browser data access, external exfil intent 등 세부 hint 추가 | 성공 | C-03 `onerror`, C-05 `document.cookie` + 외부 URL, C-06 `javascript:`가 구분됨 |
| C-10 보존 | candidate 과승격 없이 false positive review context로 보존 | 성공 | `false_positive_review_candidates=1` |
| Stage2 판단 유보 | 실행 성공/쿠키 탈취/외부 전송 성공 단정 억제 | 성공 | OpenAI와 Claude 모두 Apache 로그만으로 실행 성공을 확정하지 않는 방향으로 서술 |

### 4.1 C-08 HTML Entity 개선

C-08은 이번 수정의 핵심 검증 항목이다. 기존에는 `&#x3C;` 내부의 `#`가 SQL comment marker처럼 오탐될 가능성이 있었다. 수정 후에는 HTML entity를 별도 decode view로 보고, entity decode 결과가 XSS 구조임을 명시하는 방향으로 정리되었다.

정리하면 다음과 같다.

- `&#x3C;script&#x3E;...` 계열을 HTML entity encoded XSS로 인식
- entity 내부 `#`를 SQL comment로 과잉 해석하지 않음
- Stage2에서 다층 인코딩/HTML entity를 우회 및 복원 관점으로 설명

### 4.2 XSS hint 강화

기존에는 XSS 판단 근거가 `xss:script_tag`, `xss:alert_call`, `xss:document_cookie` 중심이었다. 수정 후에는 다음과 같은 세부 hint가 추가되었다.

- `xss:event_handler:onerror`
- `xss:javascript_protocol`
- `xss:browser_data_access`
- `xss:external_navigation`
- `xss:external_exfil_intent`
- `xss:mixed_case_script_tag`
- `xss:html_entity_decoded_script`

이 변화는 C세트의 목적과 맞다. 단순 탐지보다 어떤 XSS 구조인지, 어떤 피해 의도를 암시하는지 더 명확하게 표현할 수 있게 되었다.

### 4.3 C-10 false positive review 보존

C-10 `tutorial for onerror event handler`는 실제 공격 구조가 약하다. 따라서 candidate로 과승격하지 않는 것이 적절하다.

다만 C세트는 FP bait 판단력도 평가해야 하므로, 완전히 사라지는 것보다 `false_positive_review_candidates`에 남기는 편이 좋다. 이번 수정은 이 절충안을 반영했다.

따라서 C-10은 다음처럼 해석한다.

- Stage1 공격 후보는 아님
- filtered out / low signal 계열로 보는 것이 타당
- 그러나 FP bait 평가를 위해 review context로 보존됨

### 5.1 Encoding depth mismatch

C-02와 C-04는 실행 방식 때문에 문서상 의도보다 raw log의 encoding depth가 깊게 남은 것으로 보인다. 이는 모델 판단 실패가 아니라 `curl --data-urlencode`에 이미 인코딩된 문자열을 넣었을 때 발생하는 실행상 차이다.

따라서 비교 문서에서는 다음처럼 기록하는 것이 정확하다.

- C-02/C-04는 XSS 의미 복원에는 성공했다.
- 다만 “URL encoded”와 “double encoded”의 실제 raw depth는 실행 옵션 때문에 한 단계 깊어질 수 있다.
- 향후 재실험 시 `docs/98B_C세트_비교실험.md`의 수정된 실행 명령을 사용해야 한다.

## 7. Stage2 결과 비교

### 7.1 OpenAI

OpenAI는 전반적으로 보수적이다.

특징:

- Apache 로그만으로 브라우저 실행 성공, 쿠키 탈취, 외부 전송 성공을 확인할 수 없다고 명시
- known asset IP를 고려해 내부 테스트 가능성을 유지
- `document.cookie`와 외부 URL 조합은 위험 의도로 보되 성공으로 단정하지 않음
- 운영 조치는 `review_raw_log`, `review_error_log`, `correlate_request_id`, `correlate_src_ip`, `watch` 중심

평가:

- Apache 로그 표면 기준 한계를 가장 안정적으로 지킨다.
- XSS payload의 위험도는 충분히 설명하되, 실행 성공 단정은 피한다.

### 7.2 Anthropic

Anthropic은 XSS campaign/test traffic 구조를 더 적극적으로 묶어 설명한다.

특징:

- 동일 IP, 동일 endpoint, 짧은 시간 내 반복 요청을 자동화된 XSS 테스트 흐름으로 강하게 해석
- HTML entity, URL encoding, double encoding, mixed case 등을 다층 인코딩/우회 기법으로 묶어 설명
- user-agent의 `lab-c-set-xss-*` 라벨을 내부 테스트 정황으로 적극 반영
- `investigate_immediately` 계열 권고가 OpenAI보다 더 강하게 나타남

평가:

- TTP 묶음과 campaign 서술은 좋다.
- 다만 일반 Apache 로그 분석에서는 User-Agent 라벨에 과도하게 의존하지 않도록 주의가 필요하다.
- 이번 보고서에서는 실제 실행 성공은 단정하지 않았으므로 수용 가능한 수준이다.

## 10. Apache 로그 기반 한계

이번 C세트는 XSS payload 식별과 의미 복원에는 성공했다. 그러나 다음은 확정할 수 없다.

- 브라우저에서 payload가 실제 실행됐는지 여부
- response body에 payload가 반영됐는지 여부
- DOM sink에 도달했는지 여부
- 쿠키가 실제 탈취됐는지 여부
- 외부 URL로 실제 전송이 발생했는지 여부

따라서 최종 표현은 다음처럼 제한한다.

```text
XSS payload 전송 및 의도는 Apache 로그상 관찰된다.
그러나 브라우저 실행 성공, 쿠키 탈취 성공, 외부 전송 성공은 Apache 로그만으로 확정할 수 없다.
```

## 11. 최종 평가

C세트 수정 후 산출물은 성공으로 판단한다.

핵심 근거는 다음과 같다.

1. raw 10건이 정상적으로 같은 시간대에서 재처리되었다.
2. candidate 9건, filtered out 1건, false positive review candidate 1건으로 정리되었다.
3. HTML entity decode가 반영되어 C-08을 XSS 우회 payload로 더 정확히 해석했다.
4. HTML entity 내부 `#`를 SQL comment marker로 보는 오탐은 제거된 것으로 판단된다.
5. C-03/C-05/C-06 등에서 XSS 세부 hint가 강화되었다.
6. C-09는 `likely_false_positive`, C-10은 review context로 보존되어 FP bait 처리도 개선되었다.
7. OpenAI와 Anthropic 모두 Stage1 error 없이 `suspicious_xss 8 / likely_false_positive 1`로 수렴했다.
8. Stage2는 XSS 실행 성공, 쿠키 탈취 성공, 외부 전송 성공을 단정하지 않았다.

다만 아래 한계는 남는다.

- C-02/C-04는 실행 당시 `curl --data-urlencode` 방식 때문에 실제 raw log의 encoding depth가 문서상 의도보다 깊게 남았다.
- Anthropic은 User-Agent 라벨을 내부 테스트 정황으로 적극 활용하므로, 일반 운영 환경에서는 query string, decoded payload, response metric 중심으로 재해석해야 한다.
- Apache 로그만으로는 XSS 실행 성공 여부를 확인할 수 없으므로, 실제 취약성 검증에는 브라우저/애플리케이션 응답 검증이 별도로 필요하다.

## 13. 발표용 한 줄 정리

C세트에서는 코드 수정 후 HTML entity decode와 XSS 세부 hint가 보강되었고, 교육용 event handler 검색은 false positive review context로 보존되었다. 두 provider 모두 9개 후보 중 8건을 XSS 의심, 1건을 likely false positive로 분류했으며, OpenAI는 더 보수적으로 실행 성공을 제한했고 Anthropic은 XSS campaign/test 흐름을 더 적극적으로 서술했다. Apache 로그 기준으로는 XSS payload 전송과 의도는 관찰되지만, 브라우저 실행·쿠키 탈취·외부 전송 성공은 확정할 수 없다.
