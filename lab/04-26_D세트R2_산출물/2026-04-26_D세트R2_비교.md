## 1. 실험 개요

D세트 R2는 HTTP Parameter Pollution(HPP) 요청에 대해 다음을 확인하기 위한 실험이다.

1. 동일 이름의 파라미터가 반복되는 요청을 `hpp_detected`, `hpp_param_names`로 보존하는가.
2. benign duplicate parameter와 공격 payload가 결합된 HPP를 구분하는가.
3. HPP + SQLi, HPP + XSS를 각각 SQLi/XSS 단독이 아니라 “HPP 문맥이 결합된 공격 시도”로 해석하는가.
4. `200 application/json` 응답을 근거로 실제 SQLi 성공이나 XSS 실행 성공을 단정하지 않는가.
5. POST body HPP는 raw POST body visibility 한계 때문에 Apache 로그만으로 직접 판단할 수 없음을 유지하는가.

이번 R2는 HPP 결합 공격 탐지 관점에서는 성공으로 판단한다. 다만 모든 HPP 요청이 Stage1 candidate로 올라간 것은 아니며, benign duplicate와 POST body HPP 계열은 candidate가 아니라 context 또는 filtered out으로 처리되었다.

## 4. R2 항목별 평가

| ID | 실험 항목 | 관찰 결과 | 처리 결과 | 평가 |
|---|---|---|---|---|
| D-11 | Benign duplicate parameter | `q=apple&q=banana`, status 200, JSON 30B | supporting context | 성공 |
| D-12 | HPP + SQLi | `q=apple&q=x')) OR 1=1 --`, response 18662B | candidate | 성공 |
| D-13 | HPP + XSS | `q=apple&q=<script>alert(1)</script>`, response 30B | candidate | 성공 |
| D-14 | POST body HPP optional | body 내부 값은 현재 LLM 입력에서 직접 보이지 않음 | filtered out/미평가 | visibility 한계 확인 |

### 4.1 D-11 Benign Duplicate Parameter

D-11은 `?q=apple&q=banana` 형태의 중복 파라미터 요청이다.

평가:

- 동일 이름 `q`가 반복되므로 HPP 관찰 대상으로는 유효하다.
- 공격 토큰이 없기 때문에 Stage1 candidate로 과승격하지 않은 것은 적절하다.
- supporting event로 보존되어 D-12/D-13의 HPP 문맥을 설명하는 데 사용할 수 있다.

판정:

```text
Benign HPP observed.
Not promoted to incident candidate.
Preserved as contextual evidence.
```

### 4.2 D-12 HPP + SQLi

D-12는 동일 파라미터 `q`를 반복하면서 뒤쪽 파라미터에 SQLi payload를 넣은 요청이다.

관찰 구조:

```text
/rest/products/search?q=apple&q=x')) OR 1=1 --
hpp_detected=true
hpp_param_names=[q]
embedded_attack_hint=sqli
response_body_bytes=18662
resp_content_type=application/json
status_code=200
```

평가:

- `OR 1=1`과 SQL comment marker가 확인되어 SQLi 시도로 보는 것이 타당하다.
- 동시에 동일 파라미터 `q`가 반복되므로 HPP 문맥이 보존되었다.
- response_body_bytes가 18662로 커진 점은 R2A Boolean 실험과 유사하게 응답 차이 정황을 제공한다.
- 다만 Apache 로그만으로 DB 쿼리 결과나 데이터 유출 성공은 확정할 수 없다.

판정:

```text
HPP-coupled SQLi attempt observed.
Response-size anomaly observed.
SQLi success not confirmed from Apache log alone.
```

### 4.3 D-13 HPP + XSS

D-13은 동일 파라미터 `q`를 반복하면서 뒤쪽 파라미터에 XSS payload를 넣은 요청이다.

관찰 구조:

```text
/rest/products/search?q=apple&q=<script>alert(1)</script>
hpp_detected=true
hpp_param_names=[q]
embedded_attack_hint=xss
response_body_bytes=30
resp_content_type=application/json
status_code=200
```

평가:

- `<script>alert(1)</script>`가 명확히 관찰되므로 XSS 시도 정황은 충분하다.
- HPP가 함께 있으므로 “HPP를 이용한 XSS 시도”로 표현하는 것이 적절하다.
- JSON 응답과 status 200은 브라우저 실행 성공을 의미하지 않는다.
- Apache 로그만으로 DOM 반영, 브라우저 실행, 사용자 영향은 확정할 수 없다.

판정:

```text
HPP-coupled XSS attempt observed.
Browser execution not confirmed.
```

### 4.4 D-14 POST Body HPP Optional

D-14는 POST body 내부의 중복 파라미터를 확인하는 optional 실험이다.

평가:

- 현재 baseline은 raw POST body 원문을 LLM 입력으로 사용하지 않는다.
- 따라서 Apache 로그만으로 body 내부의 `email=a&admin=true&email=b` 구조를 직접 확인할 수 없다.
- 이 항목은 “탐지 실패”가 아니라 “visibility 한계 확인”으로 기록하는 것이 맞다.

판정:

```text
POST body HPP is not directly visible in current Apache-log-only baseline.
Do not infer privilege escalation or body-level HPP success.
```

## 9. Apache 로그 기반 한계

이번 R2에서 확인할 수 있는 것은 HPP 구조와 payload 결합 정황이다.

확인 가능:

- 동일 파라미터 `q` 중복 전달
- HPP + SQLi payload 결합
- HPP + XSS payload 결합
- response status / content-type / response_body_bytes
- benign duplicate HPP의 context 보존

확인 불가:

- 서버가 첫 번째/마지막 `q` 중 무엇을 실제로 사용했는지
- SQL 쿼리가 실제로 조작됐는지
- DB 결과가 실제로 변경되거나 유출됐는지
- XSS payload가 브라우저에서 실행됐는지
- POST body 내부 중복 파라미터가 어떻게 처리됐는지

따라서 보고서 표현은 다음처럼 제한한다.

```text
Apache 로그상 동일 파라미터 중복 전달과 SQLi/XSS payload 결합 정황은 관찰되었다.
그러나 서버 측 파라미터 선택 방식, SQLi 성공, XSS 실행, POST body HPP 처리 결과는 Apache 로그만으로 확정할 수 없다.
```

## 11. 후속 권장 사항

1. R2 비교 문서는 D-12/D-13 중심으로 확정한다.
2. D-11 benign duplicate는 supporting context로 남긴 현재 방식이 적절하다.
3. D-14 POST body HPP는 현재 구조에서는 visibility 한계로 분리해 기록한다.
4. POST body HPP를 본격 평가하려면 raw POST body logging/보존 정책을 별도 설계해야 한다.
5. D세트 R3 Directory Probing은 별도 export window로 진행한다.
6. R3 이후 D세트 통합 비교 문서에서 R1/R2/R3를 한 번에 정리한다.

---

## 12. 발표용 한 줄 정리

D세트 R2에서는 HPP + SQLi와 HPP + XSS가 각각 candidate로 선별되었고, 두 provider 모두 중복 파라미터 `q`와 공격 payload의 결합을 HPP 문맥으로 해석했다. OpenAI는 두 사건을 모두 medium으로 보수적으로 분류했고, Claude는 SQLi를 high로 더 강하게 보았지만 두 모델 모두 실제 SQLi 성공이나 XSS 실행 성공은 단정하지 않았다. Benign HPP는 candidate로 과승격되지 않고 supporting context로 보존되었으며, POST body HPP는 Apache 로그 visibility 한계로 핵심 평가에서 제외한다.
