## 2. 실험 목적

이번 실험은 `docs/98_비교_실험_요청_세트_표준.md`와 `docs/98A_A세트_비교실험_실행요청.md` 기준의 A세트 비교 실험이다. A세트는 baseline/auth 전용 세트이며, 정상 조회/검색 baseline 과 인증 실패, 반복 로그인 실패, 인증 우회 성공 요청 같은 auth 중심 행위를 같은 틀에서 비교하기 위한 목적을 가진다.

## 6. 비교 요약

공통점:

- 두 provider 모두 같은 raw log 를 바탕으로 동일한 공통 수치를 냈다.
- 두 결과 모두 incident 0건, candidate_rows 0건으로 결론이 완전히 일치한다.
- 배경 트래픽의 핵심을 `socketio_polling` 중심의 정상 세션 유지 트래픽으로 해석했다.
- 일반 검색 2건도 모두 사건화하지 않고 benign 범주로 유지했다.

차이점:

- OpenAI는 "현재 시점에서 incident 아님"과 "후속 고신호 행위가 붙는지 관찰"에 조금 더 무게를 둔 보수적 표현을 사용했다.
- Anthropic은 동일 사실관계를 더 운영 보고서형 문장으로 풀었고, 자산 목록 갱신, 기준선 관리, 인접 구간 확인 같은 후속 조치를 더 구체적으로 적었다.
- 결론 차이는 아니라 서술 톤과 운영 메모의 구조 차이에 가깝다.

이번 비교의 한계:

- A세트 자체는 baseline/auth 비교 실험이지만, 이번 시간 구간 결과는 실제로 zero-candidate 구간이어서 auth 관련 판별력을 드러내지 못했다.
- 두 보고서 모두 전처리 및 1차 분류 요약 데이터를 바탕으로 작성되어 원본 DB 로그 전체 맥락을 직접 대조한 결과는 아니다.
- 같은 raw log 를 썼기 때문에 동일 조건 비교라는 의미는 있으나, 실제 공격 후보가 없는 구간에서는 provider 간 성능 차이를 강하게 논증하기 어렵다.

다음 실험에서 보강할 점:

- 인증 실패 반복, 인증 우회 성공 같은 A세트 핵심 요청이 실제 후보로 남는 구간을 다시 확보해 비교해야 한다.
- 가능하면 동일 raw log 기준으로 stage1 또는 stage2 고정 비교를 추가해 서술 차이와 판정 차이를 분리해 보는 편이 좋다.
- known_asset_ips 와 실제 내부 클라이언트 IP 목록의 정합성을 먼저 맞춰 배경 트래픽 해석의 불확실성을 줄일 필요가 있다.

## 7. 최종 판단

이번 2026-04-24 13:20:00~13:37:00 KST 구간은 실질적으로 공격성 구간이라기보다 정상적인 Socket.IO 세션 유지 트래픽과 소량의 일반 검색 요청이 대부분을 차지한 조용한 구간으로 보는 것이 타당하다. 즉, A세트가 baseline/auth 비교 실험이라는 목적을 갖고 있었더라도, 이번에 회수된 실제 raw log 구간 자체는 auth 관련 사건성을 충분히 보여주지 못했다.

따라서 두 provider 결과는 "둘 다 같은 raw log 를 보고 incident 없음으로 수렴했다"는 의미로 해석하는 것이 맞다. OpenAI와 Anthropic 모두 실제 `meta.provider` 값이 기대 provider 와 일치했고, `selected_model`도 각각 `gpt-5.4-mini`, `claude-sonnet-4-6`으로 확인되어 provider 오염 문제는 이번 결과에서 보이지 않았다.

다만 이번 결과를 provider 성능 비교로 강하게 말할 수는 없다. 동일 raw log 기반 비교라는 실험 통제 자체는 적절했지만, 후보 incident 가 0건인 구간에서는 오탐/미탐, 우선순위화, 서술 품질 차이를 실질적으로 검증할 기회가 거의 없기 때문이다. 이번 결과는 "동일한 저신호 구간에서 두 provider가 같은 결론으로 수렴했다"는 수준까지는 말할 수 있지만, 일반화된 성능 우열 비교로 확대 해석하기는 어렵다.
