## 1. 실험 개요

D세트 R3는 Directory Probing 요청에 대해 다음을 확인하기 위한 실험이다.

1. `.git/config`, `.env` 같은 민감 경로 접근 시도를 식별하는가.
2. `/admin`, `/administrator` 같은 관리자 경로 추측을 단발 high severity로 과승격하지 않는가.
3. `/backup`, `/backups`, `/config`, `/server-status`, `/phpmyadmin`, `/manager/html` 같은 짧은 burst 요청을 directory probing 흐름으로 묶어낼 수 있는가.
4. 200 `text/html` 대용량 응답을 실제 민감 파일 노출로 단정하지 않는가.
5. 403 `/server-status` 응답을 접근 제어 정상 동작으로 해석하면서도 정찰 시도 가능성을 남기는가.

이번 R3는 실행 데이터 자체는 정상적으로 수집되었다. 그러나 Stage1 candidate로는 `/server-status` 1건만 선별되었고, 나머지 9건은 `low_signal_dir_probe` 또는 `low_signal_fuzzing`으로 filtered out되었다. 따라서 **탐지 판단은 보수적으로 잘 됐지만, burst probing을 하나의 sequence/context로 묶는 전처리 능력은 아직 약하다**고 보는 것이 정확하다.

## 10. 후속 개선 검토

R3 결과를 기준으로 다음 개선을 검토할 수 있다.

1. `low_signal_dir_probe` burst grouping 강화
   - 같은 `src_ip`, 짧은 시간대, 민감 경로 사전 기반 요청이 3건 이상이면 `probing_sequence_summary` 또는 `supporting_events`로 보존한다.

2. fallback HTML fingerprint 활용 검토
   - `status=200`, `response_body_bytes=75002`, `resp_content_type=text/html`이 반복되면 민감 리소스 노출이 아니라 SPA fallback 가능성으로 명시한다.
   - 단, `resp_html_*` fingerprint는 현재 핵심 근거로 쓰지 않는 정책이므로 보조 지표로만 검토한다.

3. Stage2 후보 밖 탐색성 요청 섹션 강화
   - `low_signal_dir_probe` 7건을 단순 분포가 아니라 경로 목록, 시간 집중도, 출발지 동일성 기준으로 요약한다.

4. 단발 `/server-status`와 burst probe 구분
   - `/server-status` 403은 incident candidate로 남기되, 주변 burst 요청과 연결해 “directory probing sequence 중 하나”로 설명할 수 있게 한다.

## 11. 후속 권장 사항

1. R3 비교 문서는 “보수적 탐지는 성공, sequence grouping은 부분 성공”으로 확정한다.
2. R1/R2/R3를 묶는 D세트 통합 비교 문서를 작성한다.
3. 통합 문서에서는 R1은 traversal 탐지 성공, R2는 HPP 결합 공격 탐지 성공, R3는 directory probing 저신호 보존/묶음 한계로 정리한다.
4. 코드 개선을 한다면 R3의 `low_signal_dir_probe` grouping과 Stage2 후보 밖 탐색성 요청 서술 강화를 우선 검토한다.

## 12. 발표용 한 줄 정리

D세트 R3에서는 총 10건의 directory probing 성격 요청이 수집되었고, `/server-status` 403 요청 1건만 `suspicious_scan` low severity candidate로 선별되었다. 두 provider 모두 실제 정보 노출이나 침해 성공을 단정하지 않았고, 나머지 9건은 `low_signal_dir_probe` 7건과 `low_signal_fuzzing` 2건으로 낮춰 처리했다. 따라서 R3는 보수적 판단은 성공했지만, burst probing을 하나의 sequence-level context로 묶는 부분은 후속 개선이 필요하다.
