## 1. 실험 개요

Round 2B의 목적은 Evasion / Temporal Chain / False Positive Bait가 수정 코드 반영 후 의도대로 보존되고 분리되는지 확인하는 것이다. 이번 라운드는 Round 2A처럼 `response_body_bytes`나 `duration_us`를 정량 비교하는 구간이 아니라, provider 변별력과 보존 로직, 오탐 억제 결과를 확인하는 보조 실험이다.

핵심 확인 항목은 세 가지다.

1. double encoding, mixed case, inline comment, whitespace 변형이 SQLi 의미를 잃지 않고 candidate로 남는지 본다.
2. `lab-b-set-chain-step-*` 중 저신호 중간 step이 완전히 사라지지 않고 temporal chain context로 보존되는지 본다.
3. 교육용 SQL 검색이 `suspicious_sqli`로 과잉 승격되지 않고 false-positive 계열로 분리되는지 본다.

중요하게도 이번 문서는 Apache 로그 표면 기준 비교다. response body 원문, application 내부 result count, DB 내부 실행 결과, POST body 원문은 현재 근거로 사용하지 않는다.

## 9. A세트 / B세트 R1 / R2A와의 연결

- A세트에서는 provider별 보수성 차이가 이미 보였고, 이번 R2B에서도 같은 패턴이 반복된다. OpenAI는 보수적 운영 확인형, Anthropic은 더 적극적 해석형이다.
- B세트 R1에서는 POST body visibility 한계 때문에 raw에 있던 auth 흐름 일부가 prepare 이후 비교축에서 약해졌다. 이번 R2B는 GET query string 중심이라 Apache 로그 표면에서 payload 구조가 더 직접 보인다.
- R2A에서는 Boolean `xclose`는 성공적으로 비교됐지만 Time-based는 실패했다. 반면 R2B는 byte/duration 정량 비교가 아니라 evasion / chain / false positive 처리 개선 여부를 보는 라운드다.
- 이번 R2B의 의미는 탐지 숫자를 늘린 것보다, evasion/chain/FP 처리 개선이 실제 산출물에서 확인됐다는 점에 있다.

## 10. 1차 → 2차 개선 종합

| 지표 | 개선 목표 | 1차 상태 | 2차 상태 | 개선 결과 |
|---|---|---|---|---|
| Double Encoding 보존 | filtered_out → candidates 보존 | filtered_out | score=11, candidate 포함, decoded_depth_2 hint | 완전 해소 |
| Temporal Chain | 저신호 chain step 보존 + chain 서술 개선 | 일부 저신호 step 누락, 단발 서술 중심 | chain-step-02는 supporting_events로 보존, CL은 단계적/캠페인 서술 개선 | 부분 개선 |
| FP Bait | suspicious_sqli → likely_false_positive | SELECT tutorial이 suspicious_sqli로 분류 | likely_false_positive로 분리 | 완전 해소 |
| 물리/구조 지표 활용 | decoded depth, supporting_events, response bytes 등 근거 활용 | encoding/supporting 지표 활용 약함 | CL은 decoded/supporting 지표를 적극 인용, OP는 부분 인용 | CL 유의미 개선 / OP 부분 개선 |

이 표를 기준으로 보면 2차의 핵심 성과는 탐지 건수 증가보다 보존성과 해석 보조 정보가 강화된 점에 있다. Double Encoding과 `fp-edu-select-1` 분리는 명확한 개선이고, Temporal Chain은 `chain-step-02`가 여전히 고신호 candidate로 승격된 것은 아니지만 supporting_events로 남아 Stage2 context에 들어간다. 따라서 이 구간은 완전한 chain candidate 보존이 아니라 chain context 보존으로 보는 것이 정확하며, 과승격을 피하면서 temporal context를 살린 절충안으로 해석하는 편이 맞다.

FP Bait도 같은 방식으로 읽어야 한다. `fp-edu-select-1`은 `likely_false_positive`로 분리되어 개선 성공이지만, `fp-edu-union-1`은 candidate가 아니라 supporting_events에 남은 상태다. 따라서 LLM Stage1이 모든 FP bait를 직접 평가한 것은 아니며, 후속으로 `false_positive_review_candidates` 또는 Stage2의 `false_positive_dismissed` 섹션을 검토하는 보강이 가능하다.

## 11. 최종 평가

Round 2B는 수정 후 의도한 개선 목표를 충족했다.

다만 Temporal Chain은 완전한 candidate 보존이 아니라 supporting context 보존으로 개선된 것이며, FP Bait 역시 `fp-edu-select-1`은 Stage1에서 직접 likely_false_positive로 분리됐지만 `fp-edu-union-1`은 supporting_events에 남는 형태로 처리됐다. 따라서 R2B의 개선은 탐지 성능 자체보다 보존성, 오탐 억제, Stage2 문맥 제공이 강화된 것으로 해석하는 것이 정확하다.

- Double URL Encoding은 candidate로 보존되었다.
- Temporal chain 저신호 step은 supporting_events로 보존되었다.
- Educational SQL search는 `likely_false_positive`로 분리되었다.
- OpenAI와 Anthropic 모두 SQLi 탐지는 유지했고, false-positive bait도 1건 분리했다.

핵심 개선 결과를 항목별로 다시 정리하면 다음과 같다.

1. `lab-b-set-evade-double-enc-1`은 candidate로 포함되었고, `encoding:double_decoded_payload`, `encoding:decoded_depth_2`, `encoding:double_decoded_sqli`가 남았다.
2. 이는 실험 UA 때문이 아니라 double decode 후 SQLi 구조가 드러났기 때문이라고 정리하는 것이 맞다.
3. `lab-b-set-chain-step-02`는 고신호 incident로 승격하지 않고 supporting_events로 보존되어 temporal chain context로 적절하다.
4. 다만 이것은 완전한 chain candidate 보존이 아니라 chain context 보존이며, 과승격을 피하면서 temporal context를 살린 절충안으로 보는 편이 정확하다.
5. `lab-b-set-fp-edu-union-1`은 supporting_events로 남아 교육용 SQL 검색 문맥을 유지했다.
6. `lab-b-set-fp-edu-select-1`은 `possible_false_positive_sql_keyword_search`로 낮아졌고 Stage1 verdict에서 `likely_false_positive`로 분리되었다.
7. 따라서 FP bait도 전부를 Stage1이 직접 평가한 것은 아니며, 후속으로 `false_positive_review_candidates` 또는 Stage2의 `false_positive_dismissed` 섹션을 통해 재검토하는 방식이 적절하다.

다만 해석 한계도 분명하다. 현재 기준은 Apache 로그 표면 비교이므로 response body 원문, application 내부 result count, DB 내부 실행 결과, POST body 원문이 없다. 따라서 “실제 데이터 유출 확정”, “DB schema 유출 확정”, “성공 공격” 같은 표현은 쓰지 않는다. 가능한 표현은 “SQLi 시도 관찰”, “오류 유발 정황”, “응답 status/bytes 차이”, “Apache 로그 표면 기준” 정도로 제한하는 것이 정확하다.

수정 코드의 일반성도 유지됐다. 보존 기준은 `lab-b-set-*` User-Agent 탐지가 아니라 double decode, `src_ip` / `uri` / time window, `query_string` 구조, 자연어 교육 문맥 같은 일반 Apache 로그 특징이므로 실험환경 특화 규칙으로만 보기는 어렵다. 후속 개선은 실험 UA prefix가 아니라 같은 `src_ip` / `uri` / time window 내 고신호 SQLi 후보 주변의 저신호 query row를 supporting_events 또는 `false_positive_review_candidates`로 보존하는 방식이 적절하다.

## 12. 발표용 한 줄 정리

B세트 Round 2B에서는 코드 수정 후 double encoding payload가 candidate로 보존되고, chain 중간 저신호 요청은 supporting context로 남았으며, educational SQL 검색은 likely_false_positive로 분리되었다. 두 AI 모두 14건의 SQLi 의심 요청을 탐지했지만, OpenAI는 보수적 운영 확인형, Anthropic은 더 강한 위협 헌팅형 해석을 보였다. 다만 결론은 끝까지 Apache 로그 표면 기준으로만 제한한다.
