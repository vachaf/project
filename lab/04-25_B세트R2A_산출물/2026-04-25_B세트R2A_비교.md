## 1. 실험 개요

이번 기록은 2026-04-25에 수행한 B세트 Round 2A의 실제 산출물 정리다. 실행 범위는 `docs/98B_B세트_비교실험_라운드2.md`의 Round 2A이며, Boolean Blind와 Time-based SQLi의 물리 지표 비교를 목적으로 한다.

이번 라운드의 핵심은 다음 두 가지다.

1. Boolean Blind에서 `xclose` pair가 Apache 로그 표면 지표로 구분 가능한지 확인한다.
2. Time-based에서 `randomblob` delay ladder가 `duration_us` / `ttfb_us` 기준으로 실제 지연을 만드는지 확인한다.

중요하게도 이번 문서는 Apache security 로그 표면에 남은 지표만 사용한다. response body 원문, application 내부 result count, DB query 결과, POST body 원문은 현재 baseline 근거로 쓰지 않는다.

## 3. raw export 요약

raw export의 `security` 행은 총 17건이다.

- `access`: 0건
- `security`: 17건
- `error`: 0건

실행된 요청은 모두 `GET /rest/products/search`였고, SQLi payload는 모두 `query_string`에 남았다. 이는 B세트 Round 1의 `POST /rest/user/login` body-only 요청과 다르게, Apache 로그 표면에서 payload 자체를 직접 볼 수 있는 구조다.

### 3.1 Boolean Blind raw 지표

| 구분 | User-Agent | status | bytes | content-type | duration_us | ttfb_us | 해석 |
| --- | --- | ---: | ---: | --- | ---: | ---: | --- |
| Baseline | `lab-b-set-blind-base-1` | 200 | 631 | application/json | 63,703 | 62,878 | 정상 기준값 |
| TRUE xclose | `lab-b-set-blind-bool-xclose-true-1` | 200 | 18,662 | application/json | 50,209 | 29,863 | 응답 크기 대폭 증가 |
| FALSE xclose | `lab-b-set-blind-bool-xclose-false-1` | 200 | 30 | application/json | 24,158 | 22,539 | 응답 크기 대폭 감소 |
| Subquery auxiliary | `lab-b-set-blind-subquery-1` | 200 | 30 | application/json | 25,350 | 23,712 | FALSE와 같은 크기 |
| Quote TRUE legacy | `lab-b-set-blind-bool-quote-true-1` | 500 | 942 | text/html | 19,193 | 18,664 | 오류 경로 |
| Quote FALSE legacy | `lab-b-set-blind-bool-quote-false-1` | 500 | 942 | text/html | 8,228 | 7,209 | 오류 경로 |

### 3.2 Boolean Blind 판정

`xclose` primary pair는 성공적으로 물리 지표 차이를 남겼다.

- TRUE: `18,662 bytes`
- FALSE: `30 bytes`
- byte delta: `18,632 bytes`
- 두 요청 모두 `200 application/json`

따라서 이번 Round 2A에서 Boolean Blind는 **SQLi 시도 관찰**을 넘어, Apache 로그 표면 기준 **Boolean physical delta observed**로 기록한다.

단, 응답 body 원문이나 application result count는 확인하지 않았으므로 “전체 상품 출력”, “빈 결과셋”, “실제 데이터 유출 확정”이라고 쓰지는 않는다. 보수적으로는 “TRUE/FALSE 조건에 따라 response_body_bytes가 크게 갈렸다”가 정확하다.

### 3.3 quote-only legacy pair 판정

quote-only pair는 TRUE/FALSE가 모두 같은 오류 경로로 수렴했다.

- TRUE quote: `500 text/html 942 bytes`
- FALSE quote: `500 text/html 942 bytes`

따라서 quote-only pair는 SQLi 오류 유도 정황은 있으나, Boolean Blind 물리 지표 검증에는 부적합하다. 문서상 secondary/legacy로 둔 판단은 타당하다.

## 4. Time-based Track 결과 — 실패

Time-based Track은 실패로 기록한다.

### 4.1 baseline median

| 구분 | count | duration_us median | ttfb_us median |
| --- | ---: | ---: | ---: |
| normal baseline (`q=apple`) | 3 | 20,380 | 17,202 |
| SQLi-shape no-delay baseline (`x')) AND 1=1 --`) | 3 | 18,949 | 16,128 |

판정 기준은 SQLi-shape no-delay baseline을 우선 사용한다.

### 4.2 randomblob delay ladder 결과

| 구분 | count | duration_us | ttfb_us | duration delta | ratio | 판정 |
| --- | ---: | ---: | ---: | ---: | ---: | --- |
| 500k | 1 | 23,195 | 18,046 | +4,246 | 1.22x | not observed |
| 1m | 1 | 16,053 | 15,577 | -2,896 | 0.85x | not observed |
| 5m | 1 | 36,649 | 31,610 | +17,700 | 1.93x | not observed |
| 10m | 1 | 29,779 | 18,287 | +10,830 | 1.57x | not observed |
| highload 30m | 1 | 18,570 | 18,109 | -379 | 0.98x | not observed |

### 4.3 Time-based 최종 판정

문서 기준상 Time-based weak positive는 다음 중 하나를 만족해야 한다.

- `duration_us` delta >= 500,000us
- 또는 delay median이 baseline median의 3배 이상

이번 결과에서 가장 큰 차이는 5m 단계의 `+17,700us`, `1.93x`다. 기준에 미달한다.

따라서 이번 Round 2A의 Time-based 결과는 다음처럼 기록한다.

> `randomblob` 기반 지연 유도형 SQLi payload는 실행되었지만, Apache security log 표면 지표 기준으로 유의미한 time-based physical delay는 관찰되지 않았다. Time-based Track은 실패 또는 미검증으로 판정한다.

## 9. A세트 및 B세트 R1과의 연결

### 9.1 A세트와의 차이

A세트 비교에서는 인증 흐름과 200 성공 응답 해석이 provider 차이를 만들었다. OpenAI는 보수적, Anthropic은 성공 incident를 더 적극적으로 강조하는 경향이 있었다.

R2A에서도 유사한 차이가 반복된다.

- OpenAI: 내부 테스트 가능성과 성공 미확정을 강조
- Anthropic: SQLi payload와 200 응답을 더 강하게 위험 정황으로 해석

다만 R2A는 A세트와 달리 payload가 GET query string에 직접 남는다. 따라서 POST body visibility 문제가 적고, 두 모델 모두 SQLi 자체는 동일하게 포착했다.

### 9.2 B세트 R1과의 차이

B세트 R1에서는 raw export에 auth bypass 성공형 POST 200 요청이 있었지만 prepare candidate에서 빠졌고, 최종 비교는 `/rest/products/search` 중심 SQLi로 수렴했다.

R2A도 비슷하게, metric 계산에 필요한 baseline row가 candidate에서 빠졌다. 그러나 R2A는 raw export에서 baseline과 pair를 직접 비교할 수 있으므로, 수동 metric 검토를 통해 Boolean 결과를 보완할 수 있었다.

정리하면 다음과 같다.

- B세트 R1: POST body visibility 한계 때문에 auth bypass 성공형 비교가 약화됨
- B세트 R2A: GET query string 기반이라 SQLi payload 가시성은 좋음
- 하지만 Round 2A 정량 비교는 아직 Stage2 자동 분석보다 raw metric 수동 보완이 더 정확함

## 10. 최종 평가

이번 B세트 Round 2A의 결론은 다음과 같다.

1. `xclose` Boolean Blind pair는 성공적이었다.
   - TRUE `18,662 bytes`
   - FALSE `30 bytes`
   - delta `18,632 bytes`
   - 둘 다 `200 application/json`
   - Apache 로그 표면 기준 Boolean physical delta observed

2. quote-only pair는 legacy로 두는 것이 타당하다.
   - TRUE/FALSE 모두 `500 text/html 942 bytes`
   - Boolean 물리 지표 차이 없음
   - 오류 유도 SQLi로는 볼 수 있으나 Boolean Blind 검증에는 부적합

3. Time-based Track은 실패했다.
   - SQLi-shape baseline median: `18,949us`
   - 가장 큰 delay delta: `+17,700us`
   - 최대 ratio: `1.93x`
   - 기준인 `+500,000us` 또는 `3x`에 미달
   - 따라서 Time-based physical delay는 관찰되지 않음

4. 두 provider 모두 SQLi 탐지는 성공했다.
   - OpenAI와 Anthropic 모두 13건을 `suspicious_sqli`로 분류

5. provider 차이는 탐지 여부보다 해석 강도에서 나타났다.
   - OpenAI: 보수적, 성공/침해 미확정 강조
   - Anthropic: 민감함, high severity와 즉시 조사 권고가 많음

6. 현재 Stage2는 Round 2A 정량 pair metric을 직접 계산하지 못한다.
   - baseline/control row 일부가 Stage1 후보에서 빠짐
   - Boolean/Time-based 성공 여부는 raw export의 `response_body_bytes`, `duration_us`, `ttfb_us`를 별도로 비교해야 함
   - 후속으로 `round2_pair_metrics` 구조 도입을 검토할 필요가 있음

## 11. 발표용 한 줄 정리

B세트 Round 2A에서는 `xclose` Boolean Blind pair가 `18,662 bytes` 대 `30 bytes`로 명확한 물리 지표 차이를 만들었지만, `randomblob` 기반 Time-based Track은 최대 `+17,700us`, `1.93x`에 그쳐 실패했다. 두 AI 모두 SQLi 탐지는 성공했으나, OpenAI는 보수적으로 성공 미확정을 강조했고 Anthropic은 더 공격적으로 위험도를 높게 해석했다.
