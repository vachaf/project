## 0. 결론 먼저

두 모델 모두 **Juice Shop(D세트 R2)에서 더 선명하고 자신 있는 판정**을 내렸다. OpenCart(E세트 R1)에서는 경로 탐색(Path Traversal)을 올바르게 탐지했지만, OpenCart 고유의 `route=` 파라미터 구조를 일반 쿼리 문자열 traversal과 구분하지 못했고, **`/admin/index.php`에 200 OK가 반환된 두 건을 저신호 잡음으로만 처리한 점**이 명확한 탐지 공백이다.

## 1. 실험 범위 확인 — E세트 R1에서 실제로 실행된 내용

E세트 실험 계획서의 전체 범위는 Round 1~5이지만, 이번 번들은 **Round 1(E-01~E-04)만 실행된 결과**다. 무엇이 실행되지 않았는지 파악이 중요하다.

| 실험 ID | 공격 유형 | 실행 여부 | 로그 잔적 |
|---|---|---|---|
| E-01 | OpenCart 기본 홈 baseline | ✅ 실행 | `/index.php` GET 200 |
| E-02 | Route parameter baseline | ✅ 실행 | `/index.php?route=product%2Fcategory` 404 |
| E-03 | Route 파라미터 Path Traversal | ✅ 실행 | `/index.php?route=../../../../etc/passwd` 404 |
| E-04 | Admin path probe (2건) | ✅ 실행 | `/admin/index.php` GET **200 × 2** |
| E-11 | PHP wrapper (`php://filter`) | ❌ 미실행 | — |
| E-12 | Path param PHP wrapper | ❌ 미실행 | — |
| E-13 | `/config.php`, `/admin/config.php` | ❌ 미실행 | — |
| E-21~23 | Search SQLi / XSS / Entity XSS | ❌ 미실행 | — |
| E-31~32 | POST body visibility | ❌ 미실행 | — |
| E-05 | Directory probing burst (9개 경로) | ❌ 미실행 | — |

> **즉, 이번 비교는 "OpenCart에서 route parameter traversal + admin path probe"와 "Juice Shop에서 HPP 기반 SQLi/XSS"의 대결이다.** PHP wrapper, config 노출, POST body 등 E세트의 핵심 실험은 후속 라운드에서 별도 분석이 필요하다.

## 3. 공격 분리 및 탐지 정확성 비교

### 3.1 OpenCart — route parameter Path Traversal 탐지

**판정 결과 (두 모델 동일):**
```
verdict   : suspicious_path_traversal
severity  : medium
confidence: Claude=medium / OpenAI=high
```

#### ✅ 탐지 성공 부분

두 모델 모두 `route=../../../../etc/passwd`에서 `dotdot_slash`, `etc_passwd` 힌트를 올바르게 포착했다. 404 응답 + text/html을 근거로 "파일 노출 성공 가능성 낮음"이라는 보수적 해석도 일관되게 유지됐다.

#### ❌ 탐지 공백 — OpenCart route 파라미터 특성 미인식

두 모델 모두 이 공격을 **"쿼리 문자열에 traversal 패턴이 존재한다"**는 일반론으로 설명했고, **OpenCart의 `route=` 파라미터가 PHP 파일 include 경로에 직접 매핑된다**는 애플리케이션 레이어 특성을 보고서에 명시하지 않았다. 실제 E세트 실험 계획서가 명시한 "route parameter abuse"라는 별도 분류가 보고서에 반영되지 않은 것이다.

| 항목 | 기대 표현 | 실제 표현 (Claude) | 실제 표현 (OpenAI) |
|---|---|---|---|
| 공격 경로 설명 | "OpenCart route 파라미터를 통한 PHP include 경로 조작 시도" | "쿼리 문자열에 경로 탐색 패턴이 존재" | "요청 파라미터에 ../../../../etc/passwd 형태의 전형적인 디렉터리 순회 문자열" |
| 앱 특이성 표현 | `route=` 파라미터의 PHP include 직접 연결 언급 | ❌ 미언급 | ❌ 미언급 |

**시사점:** route parameter traversal은 Juice Shop의 `/api/product?id=`와는 다른 맥락이다. PHP 기반 CMS/쇼핑몰에서 `route=` 파라미터는 종종 `include("controllers/".route.".php")`와 같이 동작하므로, Apache 로그 분석만으로도 "쿼리 문자열 traversal"이 아닌 "PHP controller 로딩 경로 조작 시도"로 서술될 수 있어야 한다.

### 3.2 OpenCart — /admin/index.php 200 OK 처리의 문제

이번 분석에서 가장 중요한 탐지 공백이다.

**로그 상황:**
```
lab-e-set-admin-path-1   /admin/index.php   200   3061 bytes   text/html
lab-e-set-admin-index-1  /admin/index.php   200   3061 bytes   text/html
```

**두 모델의 처리 방식:**
두 건 모두 `low_signal_fuzzing`으로 필터링되어 incident로 승격되지 않았다. Stage2 보고서에서는 probing sequence context로만 언급됐다.

**문제점:**

D세트 R3에서는 `/server-status`에 대한 403 응답 **1건**을 `suspicious_scan (low severity)`로 incident 승격했다. 그런데 이번 OpenCart에서는 `/admin/index.php`에 **200 OK가 두 번 반환됐음**에도 저신호로 처리됐다.

이를 정리하면:

| 상황 | D세트 R3 | E세트 R1 |
|---|---|---|
| 경로 | `/server-status` | `/admin/index.php` |
| 응답 | 403 Forbidden | **200 OK** |
| 탐지 결과 | `suspicious_scan` — incident 승격 | `low_signal_fuzzing` — 잡음 처리 |
| 위험 평가 | low severity | (승격 없음) |

403보다 200이 더 위험한 신호임에도 불구하고, admin 경로의 200 응답이 잡음으로 처리된 이유는 두 가지로 추정된다. 첫째, 해당 요청의 Score가 낮아 candidate 선정 기준을 통과하지 못했다. 둘째, `/admin/index.php`가 OpenCart의 정상 진입점으로 해석됐을 가능성이 있다. 그러나 **알려지지 않은 UA(`lab-e-set-admin-path-1`)에서 admin 경로가 200 OK로 응답**되는 상황은 적어도 "context-only"보다는 한 단계 높은 경보가 필요한 상황이다.

두 모델 모두 이 문제를 Stage2 보고서에서 적극적으로 지적하지 않았고, Claude만이 "저신호 디렉토리 탐색 흐름" 섹션에서 간접 언급하는 수준에 그쳤다.

### 3.3 Juice Shop — HPP + SQLi/XSS 탐지

**판정 결과:**

| 사건 | Claude | OpenAI |
|---|---|---|
| HPP + SQLi | `suspicious_sqli` / **high** | `suspicious_sqli` / **medium** |
| HPP + XSS | `suspicious_xss` / medium | `suspicious_xss` / medium |

**탐지 품질 평가:**

Juice Shop 분석은 두 환경 중 훨씬 명확했다. 이유는 세 가지다.

1. **페이로드가 query string에 그대로 기록됨:** `q=x')) OR 1=1 --`, `q=<script>alert(1)</script>` — 디코딩 없이도 육안으로 확인 가능
2. **응답 크기 차이가 명확한 이상 신호를 제공:** SQLi 요청 18,662B vs XSS 요청 30B → SQLi 요청이 실제로 더 많은 데이터를 반환했을 가능성
3. **HPP 기법이 구조적으로 명확:** `q=apple&q=<payload>` 형태로 중복 파라미터가 URI에 직접 노출

Claude는 SQLi를 `high` severity로 올렸고 그 근거로 "응답 JSON 18662바이트 — 서버가 요청을 처리했음을 시사"라는 증거 기반 상향을 제시했다. OpenAI는 동일 근거에도 `medium`을 유지하며 "200+JSON만으로 성공 단정 불가" 원칙을 우선 적용했다. 두 접근 모두 보수적 해석 원칙을 지켰지만 **severity 우선순위 판단에서 Claude가 더 세분화된 판정**을 내렸다.

## 5. 환경별 탐지 특성 비교 요약

### 5.1 Juice Shop이 더 선명하게 탐지된 이유

| 요인 | Juice Shop | OpenCart |
|---|---|---|
| 페이로드 가시성 | query string에 평문 노출 | URL 인코딩 + route= 파라미터 내 위치 |
| 응답 코드 신호 | 모두 200 OK (처리된 것 확실) | 404 (traversal 차단), 200 (admin — 잡음 처리) |
| 이상 신호 정량화 | 18662B vs 30B 응답 차이 명확 | 응답 크기 3061B × 2 (동일 → fallback 추정) |
| API 구조 명확성 | REST `/rest/products/search` 단일 엔드포인트 | PHP `index.php?route=` 범용 라우터 |
| 공격 기법 구조화 | HPP(q 파라미터 중복) + payload 패턴 | route parameter traversal (일반 패턴) |

### 5.2 OpenCart에서 탐지가 더 어려운 이유

1. **route= 파라미터의 이중성:** 정상 사용(`route=product/category`)과 공격 사용(`route=../../../../etc/passwd`)의 구조가 동일한 형식 → 탐지 엔진이 "route abuse"를 일반 traversal과 구분하기 어려움
2. **admin 경로 200 OK의 모호성:** `/admin/index.php`는 PHP 기반 CMS에서 정상 접근 경로이기도 함 → 200 응답이 "정상"인지 "침해"인지 로그 수준에서 판별 어려움
3. **응답 크기 homogeneity:** admin 접근 2건 모두 3061B로 동일 → fallback HTML로 처리됐지만, 실제 OpenCart admin 로그인 페이지가 반환된 것일 수도 있음

## 7. 탐지 누락 및 개선 과제

### 7.1 OpenCart route parameter abuse 분류 미흡

**현상:** `route=../../../../etc/passwd`를 일반 path traversal로 처리했지만, OpenCart/PHP 환경에서 이는 **PHP controller include 경로 조작 시도**라는 더 구체적인 분류가 가능하다.

**개선 방향:** reason_hints에 `opencart:route_parameter_abuse` 또는 `php:controller_path_injection` 계열 hint를 추가하면 Stage1에서부터 더 정밀한 분류가 가능하다. 단, E세트 계획서가 명시했듯 "OpenCart 전용 조건을 일반 탐지 score에 과하게 넣지 않음" 원칙과의 균형이 필요하다.

### 7.2 Admin 경로 200 OK의 저신호 처리 재검토 필요

**현상:** `/admin/index.php` 200 OK 2건이 `low_signal_fuzzing`으로 처리됐다. D세트 R3에서 `/server-status` 403 1건이 incident로 올라간 것과 대조적이다.

**개선 방향:** 비표준 UA(`lab-e-set-admin-path-1`)에서 `/admin`, `/admin/index.php`, `/admin/config.php` 등 admin 경로에 200 OK가 반환될 경우, `low_signal_fuzzing`보다 높은 score를 부여하거나 probing sequence context에서 "admin 경로 200 응답 포함" 플래그를 Stage2 입력에 명시하는 방안을 검토해야 한다.

### 7.3 E세트 미실행 Round 2~5의 필요성

이번 R1에서 다루지 못한 핵심 실험들이 남아 있다.

| 미실행 실험 | 예상 탐지 과제 |
|---|---|
